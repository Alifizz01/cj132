6   The Monte-Carlo failure sweep: which failures, and how bad

The previous sections built the machinery for analysing one damaged array: Section 1 gave us a Circuit we can reach into and break by name, Section 3 gave us the electro-thermal loop that finds the self-consistent temperatures and operating point, and Section 5 gave us the breakdown criteria that decide whether any cell has been pushed past its safe limit. Each of those answers the question "given this fault, what happens?". This section answers the question one level up: which faults should we even ask about, and how do we turn thousands of individual fault answers into trustworthy statements about the whole design? That is what the Monte-Carlo failure sweep is for. It is the outermost driver — the thing that decides which failures to simulate, calls the §1+§3+§5 stack once per failure, and gathers the results into rankings and distributions.

6.1   Recap: why we sample instead of enumerate (Chapter 1 §8)

Chapter 1 §8 introduced the Monte-Carlo idea and the wall it is built to get around. Both halves matter here, so let us restate them.

Monte-Carlo method. A way to estimate a quantity you cannot compute exactly by running many random trials and averaging their outcomes. Instead of evaluating every possibility, you sample a manageable number of them at random and let the average of the samples stand in for the true answer. The more trials you run, the closer the estimate.
The 1/√N error law. The statistical error of a Monte-Carlo estimate shrinks in proportion to 1/√N, where N is the number of trials. Concretely: to halve the error you must quadruple the number of runs (because √(4N) = 2√N). This is slow but utterly reliable — it does not care how complicated each trial is. We make this precise in §6.7.
The one idea. An array of N cells has 2N possible failure combinations (each cell is either alive or dead) — for even a few hundred cells that number is larger than anything we could ever simulate. So we do not enumerate failures; we sample them. The whole section is one driver that picks a failure pattern, runs the coupled solve, records what happened, resets, and repeats — then summarises the pile of results. Choosing which patterns to sample, and how many, is the real skill.
Why 2N is hopeless, in one number. Suppose N = 600 cells (a realistic array from §1). Then 2600 is roughly 10180 — vastly more than the number of atoms in the observable universe (~1080). Enumerating every failure combination is not "slow"; it is physically impossible. Sampling is not a shortcut we take for convenience — it is the only road that exists.

6.2   What one RUN is

The atom of the whole sweep is a single run: one failure pattern, simulated end to end, with its outcomes written down. Everything else is just "do many runs and look at the pile". Here is exactly what one run does.

  1. Pick a failure pattern. Choose a set of cell ids to break — one cell, or k cells, or each cell with some probability, depending on the sampling mode (§6.4).
  2. Fail those cells. For each chosen id, call circuit.fail(id,"dead"), which marks that cell as a dead element in the netlist (Section 1 gave us per-cell addressability precisely so we could do this).
  3. Solve the coupled model. Run the electro-thermal loop (§3): build cells at their temperatures, solve the circuit, form per-cell power, solve the thermal field, iterate to the self-consistent state.
  4. Record the outcomes. From the converged state, write down four numbers (and one location):
    • total array power — how much the whole array still delivers, in W;
    • maximum cell temperature — the hottest cell, in °C;
    • number of hot-spots — how many cells exceed a hot-spot threshold;
    • breakdown tripped? — whether any cell tripped a §5 breakdown criterion, and where (which cell/line), so we can map vulnerability, not just count it.
  5. Reset. Un-fail the cells (restore the circuit to pristine) so the next run starts from a clean array, not an accumulating wreck.
  6. Repeat. Go back to step 1 for the next pattern.
Pitfall — forgetting to reset. Step 5 is easy to drop and fails silently. If you do not restore the circuit between runs, the dead cells from run 1 are still dead in run 2, so run 2 is really testing 3 failures, run 3 tests 6, and your "single-cell failure sweep" is quietly a "cumulative catastrophe sweep". Every run must start from the same pristine array. Either reset explicitly, or rebuild a fresh Circuit per run.

6.3   The three sampling modes — what each one is for

The pattern in step 1 is the only thing that changes between the three modes the project wants. They are not interchangeable: each one is shaped to answer a different question. Define each, then read the contrast table.

Mode 1 — POSITION sweep. Fail one cell at a time, each location in turn (or each line in turn), and run the solve once per location. It is deterministic — there is no randomness, you simply march through every position — and costs O(N) runs (one per cell, or one per line). It answers: "where is the array most vulnerable?" — which single position, if it fails, hurts most, and which hurts least.
Mode 2 — COUNT sweep. Fail k cells at once, for k = 1, 2, 3, …, choosing the k cells at random and repeating several random draws at each k. By plotting a damage metric against k you see how damage grows with the number of simultaneous failures. It answers: "how many failures can we tolerate?" — at what failure count does power loss or peak temperature cross an unacceptable line.
Mode 3 — RANDOM sampling. Draw many random failure patterns — each cell fails independently with some probability p, or each pattern is k randomly chosen cells — and record the metric for every one. With enough patterns you build the full statistical distribution of power loss / max temperature / hot-spot count. It answers: "what is the spread of outcomes?" — not just the average damage but its variability, the tail, the worst-case-you-might-realistically-see.

Deterministic vs random, in one sentence. The position sweep is exhaustive over single faults — it asks every single-cell question and misses none of them, but only single-cell questions. The count and random sweeps are statistical — they cannot ask every question (there are too many), so they sample, and in exchange they can speak about combinations and spreads the position sweep can never reach. You pick the mode by matching it to the question you actually have.

6.4   Outputs and analysis: ranking, histograms, robustness

A sweep produces a pile of recorded outcomes; the analysis turns that pile into decisions. Three things come out of it.

  • Ranking — most vs least damaging. Sort the positions (mode 1) or patterns (modes 2/3) by a damage metric — say power loss, or peak temperature. The top of the list is the array's weak spot (the cell or line you most want to protect or add redundancy around); the bottom is where a failure barely matters. A position sweep naturally produces a map of vulnerability by location.
  • Histograms. For the random sweep, draw a histogram of each recorded metric — power loss, max temperature, hot-spot count. A histogram is a bar chart of how often each outcome range occurs; it shows the whole shape of the result, including the average, the spread, and the dangerous tail, rather than collapsing everything to one number.
  • Robustness comparison between layouts. Run the same sweep on two different Circuit configurations — say a 1-line-per-module layout versus a 2-lines-per-module (redundant) layout — and compare their rankings and histograms. The layout whose distribution sits at lower power loss / lower peak temperature for the same failures is the more robust design. This is how the sweep stops being a description of one array and becomes a tool for choosing between designs.
Comparing layouts is the whole point. A single histogram tells you how one design behaves under failure. Two histograms, from the same sweep run on two designs, tell you which design to build. Keep this in mind through §6.5 and §6.8 — almost every reproducibility rule below exists so that a layout-vs-layout comparison is honest.

6.5   Reproducibility: fix the seed

The count and random modes use a pseudo-random generator to choose which cells fail. A "pseudo-random" generator is not truly random — it is a deterministic recipe that produces a stream of numbers that look random. That recipe has a starting value, and controlling it is how we make a "random" experiment repeat exactly.

Seed. The starting value of a pseudo-random generator. The generator turns the seed into a long sequence of pseudo-random numbers; the same seed always produces the same sequence. So fixing the seed makes a Monte-Carlo run reproducible: re-run it and you get bit-for-bit the same failure patterns, the same results, the same histogram.
Why fixing the seed is essential — two reasons. (1) Fair comparison. To compare two layouts honestly you must hit them with the identical set of failure patterns. Same seed → same patterns → any difference in the results is a difference in the designs, not in the dice. (2) Debugging. When a run does something strange, a fixed seed lets you reproduce the exact failing pattern and inspect it, instead of chasing a bug that vanishes on the next run.

6.6   How many runs? Convergence and the standard error

For the random sweep we must decide when to stop. Run too few patterns and the estimated mean (say, mean power loss) is noisy and might mislead; run too many and we waste compute. The §8 1/√N law, made precise, tells us exactly how many we need.

Sample standard deviation s. A measure of how spread out the individual run outcomes are around their average. If every run gives nearly the same power loss, s is small; if outcomes scatter widely, s is large. It describes the spread of one run's result.
Standard error (of the mean), SE. How uncertain the estimated average is — i.e. how far the mean of your N runs is likely to be from the true mean. It is not the spread of individual runs; it is the spread of the average, and it shrinks as you add runs: SE = s/√N. This is the 1/√N law of §8 written out — the error of the estimated mean falls like 1/√N.
SE = s / √N
where
• SE = standard error of the estimated mean (same units as the metric, e.g. W)
• s = sample standard deviation of the per-run outcomes (e.g. W)
• N = number of runs.
Rearranged to solve for the runs needed to reach a target standard error SEtarget:
N = (s / SEtarget)2.  Equivalently, scaling from a known run: Nnew = Nold · (SEold / SEtarget)2.
Worked example A — runs needed for a target standard error. EXAM-HOT We have run 100 random patterns and the estimated mean power loss has a standard error of 8 W. We want to tighten that to 2 W. How many runs do we need in total?
Step 1 — identify what we know, with units. Nold = 100 runs; SEold = 8 W; SEtarget = 2 W.
Step 2 — use the scaling form. Because SE = s/√N and s does not change as we add runs, the runs scale as the inverse square of the error:
N = Nold · (SEold / SEtarget)2 = 100 · (8 W / 2 W)2
Step 3 — arithmetic. 8 / 2 = 4; 42 = 16; 100 · 16 = 1600 runs.
Sanity check. We cut the error by a factor of 4 (from 8 W to 2 W). The 1/√N law says error falls like 1/√N, so cutting error 4× needs 42 = 16× the runs: 16 × 100 = 1600. ✓ The units cancel inside the ratio (W/W), leaving a pure count, as a number of runs must be. ✓

In practice you do not even need s up front: watch the running estimate. Plot the estimated mean (and a band of ±SE around it) as runs accumulate. Early on it lurches; as N grows it settles into a narrowing band. When the band is tighter than the precision you need — and the estimate has stopped drifting — you have run enough. That stabilising curve is the convergence you are looking for.

6.7   Cost and parallelism: why a big sweep is affordable

A sweep of thousands of runs sounds expensive. Two facts make it cheap. The first is about structure; the second is about where the time goes.

Embarrassingly parallel. A workload made of pieces that are completely independent — no piece needs any result from another, and they share no state. Such a workload splits across processors with almost no coordination overhead, giving a near-linear speed-up (8 cores ≈ 8× faster). The name is not an insult; it means the parallelism is so obvious it is "embarrassing" how little cleverness it takes.
Each run is independent → the sweep is embarrassingly parallel. One run fails some cells, solves, records, and resets — it touches nothing any other run needs. So you can hand different runs to different CPU cores and they never have to talk. A library such as joblib farms the runs across cores for you, and the wall-clock time drops by roughly the number of cores.
The cost driver, and why §2 matters. Each run is one (or a few) circuit solves plus the thermal solves of the §3 loop. The thermal solves are the vectorised Newton solves from Section 2 — they update every cell at once and are cheap. So the per-run cost is dominated by the circuit solve, and the thermal side stays light no matter how many runs you do. That vectorisation from §2 is precisely what makes a thousand-run sweep affordable rather than ruinous: a slow per-cell thermal solve multiplied by thousands of runs would sink the whole exercise.
Worked example B — sizing a position sweep and its wall-clock. EXAM-HOT An array has 600 cells. We want a full position sweep: fail every single cell, one at a time, and solve. (a) How many runs? (b) If one run takes 0.5 s, what is the wall-clock on 1 core versus 8 cores?
Step 1 — number of runs. A position sweep fails one cell at a time, every location in turn → one run per cell → 600 runs. (This is the O(N) cost from §6.3 with N = 600.)
Step 2 — wall-clock on 1 core. Runs are done one after another: 600 runs × 0.5 s/run = 300 s (= 5 minutes).
Step 3 — wall-clock on 8 cores. The runs are independent (embarrassingly parallel), so 8 cores share the work: 300 s / 8 = 37.5 s ≈ 38 s.
Sanity check. 600 × 0.5 = 300 ✓. 300 / 8 = 37.5 ✓. Note 37.5 s is the ideal linear speed-up; real parallelism carries a little overhead (spawning workers, gathering results), so expect about 38 s or a touch more — near-linear, not perfectly linear. That gap between "37.5" and "~38" is exactly the coordination cost the word embarrassingly promises will be small. ✓

6.8   Variance reduction: the same accuracy with fewer runs

The 1/√N law is harsh — quadrupling runs to halve the error. Two standard techniques get the same accuracy from fewer runs by sampling more cleverly than "uniformly at random".

Stratified sampling. Divide the space of failure patterns into regions (strata) — e.g. by failure count, or by array region — and deliberately spend a planned share of runs in each stratum, instead of leaving coverage to chance. This guarantees every region is represented and removes the luck of random under-sampling, shrinking the variance of the estimate.
Importance sampling. Spend more runs where the outcome matters most — here, where damage is likely or severe — and fewer where little happens, then re-weight the results to stay unbiased. By concentrating effort on the patterns that drive the answer, you reach the same accuracy with fewer total runs than uniform sampling.

6.9   Socratic ladder — choosing which failures to trust

Q·a (foundation). We have three sampling modes. In one line each, what question does each mode answer?
Position sweep → "where is the array most vulnerable?" (which single location hurts most/least). Count sweep → "how many simultaneous failures can we tolerate?" (how damage grows with the number of dead cells). Random sampling → "what is the spread of outcomes?" (the full distribution of power loss / max-T / hot-spot count, not just the average).
Q·b (build up). When is a deterministic position sweep on its own enough — and when is it not?
It is enough when your real risk is a single cell or line failing in isolation, and you want a complete, no-luck-involved map of which single failure is worst. It is exhaustive over single faults and deterministic, so it can never miss one and never needs a seed. It is not enough when failures come in combinations (two, three, many at once) or when you need to know the spread of outcomes — a position sweep asks only single-fault questions and is silent about everything else.
Q·c (why). Once we admit multiple simultaneous failures, why must we turn to random sampling rather than just enumerating them?
Because the number of multi-cell failure combinations explodes: there are 2N patterns in all (≈10180 for 600 cells), far beyond any enumeration. We cannot ask every combination, so we sample random patterns and let their distribution estimate the truth. Random sampling is the only way to say anything about the spread of multi-failure outcomes within a finite compute budget.
Q·d (build up). Having chosen random sampling, how do we know we have run enough patterns?
Track the standard error of the estimated mean, SE = s/√N, and watch the running estimate. Keep adding runs until SE is below the precision you need and the running mean has stopped drifting (it settles into a narrowing band). If you need error SEtarget and a pilot of Nold runs gave SEold, you need N = Nold(SEold/SEtarget)2 runs — e.g. 100 runs at 8 W → 1600 runs for 2 W.
Q·e (deep — the target). Given we can't simulate all 2N failures, how do we choose which failures to simulate so the answer is trustworthy?
We match the sampling to the question, then run enough and pin the randomness. (1) Pick the mode by the question: a position sweep when single-fault vulnerability is the concern (deterministic, O(N), exhaustive over single faults); a count sweep to chart damage against the number of simultaneous failures; random sampling to get the distribution of outcomes. (2) For the random modes, run until the standard error is below your needed precision and the running estimate has stabilised — not a fixed guess, but a convergence you watch. (3) Fix the seed so the chosen patterns are reproducible: the experiment repeats exactly, two layouts are hit with identical patterns (so any difference is the design, not the dice), and any oddity is debuggable. (4) If runs are scarce, use stratified or importance sampling to cover regions deliberately or concentrate effort where damage lives, buying the same accuracy from fewer runs. Trustworthiness comes from choosing the right question-shaped mode, running until the error is provably small, and making the randomness controlled and repeatable — not from blindly throwing random faults and hoping.

6.10   Two traps that fake a result

Pitfall — comparing two layouts with different seeds. If layout A is swept with seed 1 and layout B with seed 2, the two designs face different random failure patterns. Now any difference in their histograms mixes two causes — the design difference you care about and the random-pattern difference you don't — and you cannot tell them apart. The conclusion "B is more robust" might just be "B got an easier set of failures". Always use the same seed for both layouts so they meet the identical patterns and the only variable left is the design.
Pitfall — reporting a single worst case from too few random runs. Run only a handful of random patterns and the worst one you happen to see is dominated by luck — run again with a different seed and the "worst case" changes. A worst case from too few runs is an anecdote, not a result. Either run enough patterns that the worst-case region is well sampled (watch the tail of the histogram settle), or — better for true worst-single-faults — use the deterministic position sweep, which finds the actual worst single location with no luck involved.
