8   Putting it together: the full pipeline, end to end

Seven sections in, we have built seven separate machines: a parametric, fault-addressable Circuit (§1); a vectorised thermal solver (§2); the electro-thermal coupling loop that makes electricity and temperature agree (§3); an orbit-driven environment from hapsira (§4); a pair of breakdown criteria (§5); a Monte-Carlo failure sweep (§6); and a long-format results store with Excel export (§7). Each was taught on its own. This final section does the one thing none of the earlier ones could: it assembles them into a single workflow and shows how one concrete engineering question flows through every module from end to end. The question we will trace is deliberately sharp — "which single-cell failure is most likely to melt the substrate?" — and by the end you will be able to name, for each module, exactly what it contributes to the answer.

The one idea. The seven modules are not seven tools you pick from; they are seven stages of one assembly line. The array shape (§1), the substrate (a small new loader), and the orbit/environment (§4) are built once; then the Monte-Carlo driver (§6) turns a crank thousands of times, and on every turn a single failure pattern is pushed through the same fixed sequence — circuit.fail (§1) → electro-thermal loop (§3, which calls the vectorised thermal solve §2) → breakdown criteria (§5) → a row in the results store (§7) — and at the end the store is aggregated into heat-maps, histograms, rankings and an Excel summary (§7). Understanding the pipeline is understanding where each section lives in that sequence.

8.1   The new-module map: what we are actually building

The chapter has been a design for a set of new Python modules, each one responsible for a single stage and each one standing on the proven Chapter-1 primitives (the cell physics, the SPICE model writer cellBuilder/cell.buildModel, and the ngspice runner ng_sim). Nothing below re-derives the cell; every module is new assembly on old bricks. Here is the complete map, smallest responsibility to largest.

substrate.pySubstrate loader / dataclass: absorptivity α, emissivity ε, conductivity and thickness. PowerPy already exposes getSubstrateData with exactly this schema, so this module wraps a known data source.(feeds §2)

Why these seven, and not one big script. Each module has exactly one reason to change. If the orbit model improves, only environment_orbit.py moves; if the breakdown limits are re-tuned, only the criteria inside montecarlo.py (or the small criteria helper) move; if the array is re-wired, only the Circuit declaration changes. The thermal solver never knows there is a Monte-Carlo loop above it, and the Monte-Carlo loop never knows how the thermal solve works inside — it just calls the coupling loop and reads back numbers. That separation is what lets a five-stage pipeline stay debuggable.

8.2   The end-to-end flow, stage by stage

Now the assembly line itself. Read it once as five numbered stages; each stage names the module and section it belongs to. Stages 1–2 happen once; stages 3–4 are the body of the Monte-Carlo loop and repeat once per run; stage 5 happens once at the end.

  1. Build the array (once) — §1. Declare the topology and clone the prototype cell into the whole tree: 
    circuit = Circuit.fromPrototype(cell,
              cells_per_line, lines_parallel,
              modules_per_block, n_blocks)
    This fixes the shape, assigns every cell its hierarchical id (B3.M2.L1.C7), and fills the flat circuit.cells registry that fault injection will reach into.
  2. Load the substrate and define the orbit (once) — §4 (+ substrate.py). Pull the substrate's α, ε, conductivity and thickness from getSubstrateData; define the orbit with hapsira and generate the Environment time series — P_Sun, P_Albedo, P_IR and tilt at every orbit point, including the eclipse where direct sun drops to zero. From this series we can pick the worst orbit point (the harshest combination of fluxes) to stress the array.
  3. For each Monte-Carlo run: plant a failure and solve — §6 → §1 → §3 → §2 → §5. The driver (§6) chooses a failure pattern; then, for that run:
    • plant it — circuit.fail(id, "dead") degrades the chosen cell(s) by id (§1);
    • solve it — at the chosen orbit point(s), run the electro-thermal coupling loop (§3), which repeatedly calls the vectorised thermal solver (§2) until temperature and the operating point agree (the self-consistent fixed point);
    • judge it — apply the breakdown criteria (§5) to every cell's converged state: destroyed if Tfront ≥ Tlimit or (reverse-biased and Prev = |V·I| ≥ Plimit).
  4. Record per cell, then reset and repeat — §7 → §6. Write the per-cell record — Tfront, Trear, signed Pelec, reverse flag, Prev, which criterion tripped — as rows in the long-format store (§7), tagged with the run id and the failure pattern. Un-fail the cell so the next run starts from a pristine array (§6's reset rule), and loop back to stage 3.
  5. Aggregate (once) — §7. When every run is done, aggregate the store: heat-maps of vulnerability by location, histograms of power-loss / max-temperature / hot-spot count, and a ranking of failure positions by damage. Export an Excel summary the engineer can read and act on.
The shape of the whole thing, in one breath. Build once (array §1, substrate, environment §4) → loop many times (fail §1, couple §3 calling thermal §2, judge §5, record §7) → aggregate once (rank, histogram, Excel §7). The "loop many times" body is identical every iteration; only the failure pattern the driver hands in (§6) changes. That is the entire system.

8.4   A worked end-to-end scenario, with the chapter's numbers EXAM-HOT

Let us run the question on a concrete array and read the result off the established numbers. We reuse the chapter's typical shape and the §2/§5 thermal results exactly, so the scenario is fully traceable.

Worked scenario — the most-damaging single-cell failure on a 600-cell array.
Step 1 — build the array (§1). Take the chapter's typical topology: cells_per_line=10, lines_parallel=1, modules_per_block=3, n_blocks=20. Total cells = 10×1×3×20 = 600. Series depth (what sets the bus voltage) = cells_per_line × modules_per_block = 10×3 = 30 cells ⇒ Vbus = 30 × 2.35 V ≈ 70.5 V — the project's ~70 V bus. Note lines_parallel = 1: each line is the only current path in its module — there is no line-level redundancy. Hold that fact; it is the whole story.
Step 2 — conditions (§4). Load the substrate (α, ε, conductivity, thickness via getSubstrateData) and build the orbit's Environment with hapsira. From the time series, pick the worst orbit point — the harshest sun/tilt combination — and run the sweep there, where failures bite hardest.
Step 3 — choose mode and sweep (§6). The question is single-fault vulnerability, so use the position sweep: fail one cell at a time across all 600 locations — 600 deterministic runs, no seed needed.
Step 4 — the decisive run (§1→§3→§2→§5). Consider the run that kills the cell at a particular series position. circuit.fail(id,"dead") sets that cell's photocurrent to zero (§1). But a dead cell in a series line does not simply vanish: the other cells in its line still try to push the string current through it, forcing it into reverse bias. With no effective bypass, it is driven to its reverse-breakdown voltage V = −20 V while still carrying the string current I = 0.48 A. The electro-thermal loop (§3), calling the vectorised solver (§2), converges to:
Prev = |V·I| = |(−20 V)(0.48 A)| = 9.6 W  →  Tfront ≈ 187 °C
Now apply the §5 criteria: Prev = 9.6 W ≥ Plimit = 2 W ⇒ FAIL on reverse-power; Tfront = 187 °C ≥ Tlimit = 150 °C ⇒ FAIL on temperature. Both criteria trip — this position is flagged destroyed and, being among the hottest/most-stressed, ranks most-damaging.
Step 5 — the rest of the array, and the power (§5→§7). The healthy cells, meanwhile, sit near ~39 °C (the §2 healthy-cell figure, 38.89 °C) — cool, generating, well clear of both limits. But the failed cell's whole line has stopped generating (its current path is broken), so the array's total power drops by that lost string's share — the contribution of one line out of the array. Every per-cell number goes into the long-format store (§7).
What the engineer learns. The ranking (§7) names the series position whose single-cell death drives 9.6 W / 187 °C and trips both criteria as the array's weak spot. Two fixes follow directly: (1) add a bypass diode at that position — from §5, clamping −20 V to −0.7 V collapses 9.6 W to 0.34 W and the cell settles cool, a destroyed-to-fine flip; or (2) change the layout — raise lines_parallel from 1 to 2, the redundant module §6 compared, so a dead cell's line is no longer the only path and the others carry on. The pipeline did not just say "it gets hot"; it said where, how hot, why, and what to do. ✓
Why lines_parallel = 1 was the whole story. With a single line per module, one dead cell breaks the only path and forces itself into −20 V reverse breakdown — the 9.6 W / 187 °C catastrophe. The §1 topology knob, the §5 criteria, and the §6 layout comparison all meet at this one cell: the array is fragile because of a wiring choice, and the fix is a different wiring choice. That is the chapter's argument in a single cell.

8.5   Honest status: what is proven, what is pending

A study-notes chapter should be candid about which parts already stand and which are still a plan. Here is the honest accounting.

Proven now. The vectorised thermal solver (§2, thermal_vectorized.py) is pure numpy — it depends on no external simulator — and it is verified against a hand-worked single-cell energy balance: the §2 worked example (65.26 °C idle, 38.89 °C extracting, 187.48 °C dissipating) was reproduced by hand, step by Newton step, with the residual collapsing to ~10−11. Where the solver overlaps a closed-form balance, it agrees. That stage is trustworthy today.
Pending validation. The circuit / ngspice path (§1) and the Monte-Carlo machinery (§6) depend on the existing Chapter-1 cell model and on ngspice actually running and reporting correctly. Those are validated in two steps, smallest first:
  • Tiny 2×2 circuit first. Build a 4-cell Circuit (cells_per_line=2, lines_parallel=2 → one tiny block), run it through ngspice, and read back each cell's V and I to confirm the per-cell operating point is extracted correctly and matches a hand calculation. This is the smallest test that exercises the netlist build, the run, and the read-back together.
  • Then end-to-end. Only once the 2×2 read-back is trusted do we wire the full pipeline (§1–§7) and run a real sweep, checking that healthy cells land near ~39 °C and a forced reverse cell reproduces the ~187 °C / 9.6 W signature from §2/§5.
The main implementation risk — per-cell V,I extraction from ngspice. Everything downstream rides on one fragile step: pulling each cell's converged voltage and current back out of the ngspice result, correctly, by id. The §3 loop needs that per-cell P_elec vector to compute heat; §5 needs each cell's V (and its sign) to apply the reverse-power criterion. If the read-back maps a current to the wrong cell, or drops a sign, then one circuit solve quietly corrupts the whole per-cell power vector — and every temperature, every breakdown flag, every ranking built on it is wrong, silently. This is exactly why the 2×2 read-back test exists: prove the extraction on four cells you can check by hand before trusting it on six hundred.

8.7   Socratic ladder — trace the question through every module

Ladder — trace one question, "which single-cell failure most endangers the panel?", through every module and name what each contributes.
Rung 1 (foundation). Which module builds the array, and what does it give us that makes "a single-cell failure" even meaningful?
circuit.py (§1). Circuit.fromPrototype(...) clones the prototype cell into the full Group/CellRef tree, assigns every cell a stable hierarchical id (B3.M2.L1.C7), and fills the flat circuit.cells registry. That per-cell addressability is what lets us speak of "this single cell" at all — without it you can count cells but never point at one.
Rung 2 (build up). Which module plants the failure, and how does it choose which failures to try for this question?
Planting is circuit.fail(id,"dead") from §1 — it sets the chosen cell's photocurrent to zero and leaves every other cell untouched. Choosing which failures is the §6 driver in montecarlo.py: because the question is single-fault vulnerability, it uses the position sweep — one dead cell marched across every location, deterministic, O(N) (600 runs for 600 cells), no randomness or seed required.
Rung 3 (the engine). Which module(s) find the temperatures the failure causes, and how do the two cooperate?
electrothermal.py (§3) drives the loop; thermal_vectorized.py (§2) does the thermal arithmetic. The coupling loop solves the circuit at the current temperatures to get each cell's P_elec, hands that power vector to the vectorised Newton thermal solve to get new temperatures, and iterates until the two stop changing — the self-consistent fixed point. §3 is the "keep going until they agree" wrapper; §2 is the fast inner solve it calls each pass.
Rung 4 (the verdict). Which module decides a cell is "destroyed", and what exactly is its rule?
The §5 breakdown criteria (living in montecarlo.py / a small criteria helper). A cell is destroyed if either test trips: temperature, Tfront ≥ Tlimit (the structural/bond limit, ~150 °C, not the ~660 °C melt); or reverse-power, Prev = |V·I| ≥ Plimit evaluated only when V<0. The logical OR is deliberate — each test catches a real failure the other misses.
Rung 5 (the record). Which module stores and ranks the outcomes, and where in its output is the answer to our question?
results.py (§7). Every run writes a long-format row per cell — Tfront, Trear, signed Pelec, reverse flag, Prev, tripped criterion — into Parquet/HDF5. Aggregation then sorts the 600 positions by damage (peak temperature or power loss). The top of that ranking is the answer: the single-cell position whose failure most endangers the panel. An Excel summary makes it actionable.
Rung 6 (deep — the target). Put it together: trace the whole question through every module in order, naming each contribution; and say why, for this question, Monte-Carlo chooses the failures the way it does.
The full trace. circuit.py (§1) builds 600 addressable cells → the §6 driver in montecarlo.py chooses the failures, here a deterministic position sweep (one dead cell per location, 600 runs) → circuit.fail(id,"dead") (§1) plants each one → electrothermal.py (§3), calling thermal_vectorized.py (§2) at the worst orbit point from environment_orbit.py (§4) on the substrate from substrate.py, converges to the temperatures and operating point → the §5 criteria turn each cell's state into a destroyed Boolean → results.py (§7) records every run and ranks the positions by damage; the top-ranked position is the answer. Why position mode chooses failures this way: the question asks about a single cell's failure and wants the worst location. There are only N single-cell failures (one per cell), so we can afford to test all of them — a deterministic, exhaustive march, no sampling and no seed, which is guaranteed to find the true worst single position with no luck involved. (Random or count sampling would be the right choice only if we cared about combinations of failures or the spread of outcomes — the 2N-combination questions §6 samples because they cannot be enumerated.)

8.8   Two pipeline-level pitfalls

Pitfall 1 — sweeping at a benign orbit point and declaring victory. The §4 environment is a time series: the fluxes and tilt rise and fall around the orbit, so the same failure is far more dangerous at some orbit points than others. If you run the whole sweep at a benign point — full sun, easy tilt, nothing stressed — even the worst single-cell failure may stay under the limits, and you will conclude "the array is safe." It is not; you simply never tested it when it was hot. Always identify the worst-case orbit point (or sweep across the harsh part of the orbit) before drawing any safety conclusion. A clean sweep at the easiest moment of the orbit proves nothing about the hardest.
Pitfall 2 — getting the per-cell read-back wrong poisons everything downstream. One circuit solve produces the entire per-cell P_elec vector that the §3 loop feeds to the §2 thermal solve, and the per-cell V that §5's reverse-power criterion gates on. Get that extraction wrong — a current attributed to the wrong id, a dropped sign on V — and the error does not stay local: every temperature, every breakdown flag, and every ranking computed from that vector is wrong, and wrong silently (the numbers still look plausible). This is the same risk flagged in §8.5; it is why the 2×2 read-back test must pass before the pipeline is trusted. The pipeline is only as correct as its weakest hand-off, and this is the weakest.
