Words You Need First

Chapter 2 adds some mathematical and software vocabulary on top of Chapter 1. Meet these once here; each is fully explained where it is used, and all are in the Glossary.

Recursion A structure (or function) defined in terms of smaller copies of itself. Our circuit is a tree: a group made of groups, made of … cells.
Series / parallel composition The one electrical rule from Chapter 1, now made a parameter at every level: voltages add in series, currents add in parallel.
Residual The left-over of an equation written as "something = 0". If f(T)=0 at the answer, then f at a guess is the residual — how wrong the guess is.
Newton–Raphson A method to drive a residual to zero by repeatedly following the function's slope (its derivative) to its zero-crossing. It converges very fast (the correct digits roughly double each step).
Jacobian For a system of equations in several unknowns, the table of all partial derivatives (how each equation responds to each unknown). It is the multi-variable "slope" Newton's method needs.
Vectorised Doing the same arithmetic on a whole array at once with a single compiled operation (numpy), instead of a slow Python loop over the elements.
Fixed point / self-consistent A state that maps to itself: feed it in, run the model, get the same state out. The electro-thermal loop hunts for one.
hapsira The maintained astrodynamics (orbit-mathematics) library we use to propagate the orbit and compute sun, eclipse, albedo and infrared fluxes over time.
Breakdown criteria The explicit rules — a temperature limit AND a reverse-power limit — that decide when a cell counts as destroyed.
Monte-Carlo modes Three ways to choose which failures to simulate: position sweep, count sweep, and random sampling.
Long format & Parquet Storing results as one row per (run, cell) in a compact columnar file — the source of truth, with Excel as a generated summary.

1The Customizable Circuit Class

1 The customizable Circuit class: a parametric, fault-addressable array

Chapter 1 gave us a solar array built as a fixed ladder — cell → string → section → panel — with the wiring rules baked in: a string is always cells in series, a section is always strings in parallel. That hierarchy is perfect for describing one specific, well-behaved array. It is the wrong tool the moment you want to experiment with the wiring, or reach into the array and break one chosen cell on purpose. This chapter adds a new object, the Circuit class, that fixes exactly those limitations. You will learn to declare an array's shape with a handful of numbers, watch Circuit assemble the electrical netlist for you, and — the part everything later depends on — keep every single cell individually addressable so a fault study can fail any one of them by name. We define every term the moment it appears, as always.

1.1 Why the fixed hierarchy is not enough

The Chapter-1 hierarchy has three hard edges. The failure study we are about to build runs straight into all three:

  • You cannot nest custom levels. The ladder has exactly four rungs, in one fixed order. If your real hardware groups cells into four levels that do not map onto cell/string/section/panel, you have no clean way to express it.
  • You cannot flip a junction between series and parallel. "String = series" and "section = parallel" are wired into the class. If a wiring diagram is ambiguous — is this joint adding voltages or adding currents? — testing the other reading means rewriting code, not changing a setting.
  • You cannot reach in and fail ONE specific cell. The hierarchy knows about groups of cells, not about cell number 7 of a particular line. To study what one dead cell does to the whole array, you must be able to point at that one cell.
Term topology — the shape of a circuit: which components connect to which, and how the groups nest, irrespective of the exact numbers. When we say the Circuit class lets you "declare the topology," we mean you describe that shape with parameters, and the class works out the wiring.
Key idea The new Circuit class turns the array's shape into data. Instead of a fixed four-level ladder written in code, you pass numbers — how many cells per line, how many lines in parallel, and so on — and a series/parallel choice at each level. The class then builds the netlist and hands every cell a permanent address. One declaration buys you all three things the fixed hierarchy could not give: custom nesting, switchable series/parallel, and per-cell access.
Note The word netlist appears throughout this section, so let us pin it down now. A netlist is the plain-text list of every electrical component in a circuit together with the named connection points (nodes) each one joins. It is what a circuit simulator actually reads. "Building the netlist" just means writing out that list of parts and connections — which is the bookkeeping the Circuit class does for you.

1.2 The four-level topology, level by level

The Circuit class is general, but the failure study uses one concrete four-level shape, derived from the project's design sketch. We will teach it from the bottom up. Each level is named, has one governing parameter (a number you choose), and has a default series-or-parallel rule.

Term line — cells_per_line cells wired in series (end-to-end), e.g. 10. Series wiring makes voltages add, so a line's job is to build voltage. Ten cells of 2.35 V in a line give about 23.5 V.
Term module — lines_parallel lines wired in parallel (side by side), e.g. 1, 2, or more. This is the project's "two parallel circuits" idea. Parallel wiring makes currents add while the voltage stays that of a single line, so a module's job is to build current and redundancy — if one line fails open, the others still carry current.
Term block — modules_per_block modules combined, by default in series, e.g. about 3. Stacking modules in series adds their voltages, stepping the array up toward the spacecraft's roughly 70 V power bus. (10 cells × 3 modules × 2.35 V ≈ 70.5 V — almost exactly the target bus.)
Term circuit — n_blocks blocks combined, by default in parallel, e.g. about 20 (a realistic range is 15–25). Each block sits behind its own blocking diode. Putting many blocks in parallel adds their currents to reach the full array current, and the per-block diode keeps a failed block from dragging the healthy ones down.
Term blocking diode — a diode is a one-way valve for current. A blocking diode in series with each block lets that block push current out to the bus but refuses to let current flow backwards into it. So if one block goes dark or shorts, current from the other parallel blocks cannot pour back through the bad one — the diode blocks it. That is why every block needs its own.

Key idea — series/parallel is a PARAMETER, not a law Look at the "default combine" column: series, parallel, series, parallel. Those are defaults, not fixed facts. In Circuit the choice combine = "series" or "parallel" is a value you set at every level. So if a colleague reads the wiring diagram differently — say they think the modules in a block are actually paralleled — testing their reading is a one-value change (flip one combine), not a rewrite of the class. This is the single biggest advantage over the fixed Chapter-1 hierarchy.
Note Be careful with the two roles of "series" and "parallel" here. Within a line, series means cells nose-to-tail. Within a block, series means whole modules stacked. The rule is the same at both levels — series adds voltage, parallel adds current — but what is being combined (single cells vs. whole modules) differs. The recursive design in §1.3 is what lets one rule apply cleanly at every scale.

1.3 How Circuit stores the array: a recursive tree

Internally, Circuit does not store four nested Python loops. It stores a tree. To see why that is the natural choice, we first need the idea of recursion.

Term recursion — a structure (or a procedure) defined in terms of smaller copies of itself. A folder on your computer is recursive: a folder contains files and other folders, each of which contains files and yet more folders, down to the bottom. Our array is the same: a block is made of modules, which are made of lines, which are made of cells. "Made of smaller copies of the same kind of thing" is the recursive pattern.

Term tree — a branching structure with one top node (the root) that splits into child nodes, each of which may split again, ending in leaves that do not split. A family tree and a folder hierarchy are trees. Our array is a tree: the whole circuit is the root, blocks and modules and lines are internal nodes, and the individual cells are the leaves.

The clever part is that the tree needs only one node type for every internal level, because every internal level does the same job: take some children and combine them either in series or in parallel.

Group(combine="series"|"parallel", children=[Group | CellRef], id="B3.M2.L1")

Read that as: a Group knows one thing about wiring — whether to combine its children in series or parallel — and it holds a list of children. Each child is itself either another Group (a sub-level) or a leaf. A block is a Group of module Groups; a module is a Group of line Groups; a line is a Group whose children are leaves. Same node type, nested.

Term CellRef — the leaf node: CellRef(cell, id). It wraps one actual cell object (the Chapter-1 cell) together with a permanent id. The leaves are where the real photovoltaic devices live; the Groups above them only describe wiring.

Every node carries a stable, human-readable hierarchical id that names its exact place in the tree. A leaf id looks like:

B3.M2.L1.C7
read left to right as Block 3, Module 2, Line 1, Cell 7. Each segment names one level of the tree, so the id is literally the path from the root down to that one cell. Internal groups get the shorter prefix — the line itself is B3.M2.L1.

Walking the tree to find one cell would be slow if you had to search it every time. So Circuit also keeps a flat lookup table:

Term flat registry — circuit.cells — a dictionary dict[id -> cell] mapping every leaf id straight to its cell, e.g. circuit.cells["B3.M2.L1.C7"] hands you that exact cell. It is "flat" because it ignores the nesting — one giant table, every cell reachable in one hop.
Term O(1) ("order one", constant time) — a description of how long a lookup takes as the array grows. O(1) means the time does not depend on how many cells there are: finding one cell in a 10-cell array and in a 10 000-cell array takes the same constant effort. A dictionary gives O(1) access; searching the whole tree would be slower (it grows with the number of cells).
Key idea Two views of the same array, each good at a different job. The tree (Group/CellRef) captures the wiring — who is in series with whom — which is what you need to build the netlist. The flat registry (circuit.cells) captures addressability — instant access to any one cell by id — which is what you need to inject a fault. The hierarchical id is the bridge: it is both a path through the tree and a key into the dictionary.

1.4 Building the netlist: one recursive walk

To simulate the array we must turn the tree into a netlist the circuit simulator can read. PowerPy's simulator is ngspice, an open-source version of SPICE (Simulation Program with Integrated Circuit Emphasis — the classic analog-circuit simulator). The netlist is generated by a single recursive walk of the tree that assigns nodes as it goes.

Term node — a named electrical connection point; every wire junction is a node. Components that share a node are electrically joined there. A rail is just a node that many things connect to in common — for a parallel group, the two shared nodes its children all hang between.

The walk treats the two combine types differently — and this is the electrical heart of the whole class:

The recursive node-assignment rule
A SERIES group chains its children head-to-tail. The output node of one child is the input node of the next — neighbours share one node. Because the children sit one after another, their voltages add from the bottom of the chain to the top. (This is how a line stacks cell voltages, and how a block stacks module voltages.)
A PARALLEL group ties all its children between the same two rail nodes. Every child hangs between the one top rail and the one bottom rail, so they all see the same voltage, and their currents add into the rails. (This is how a module sums line currents, and how the circuit sums block currents.)
Each leaf emits its cell. At a CellRef the walk calls cell.buildModel(name=id) — the very Chapter-1 routine that writes one cell's SPICE subcircuit — passing the leaf's id as the instance name so the cell appears in the netlist under its hierarchical address.
A blocking diode is appended per block. As the walk finishes each block it adds that block's series blocking diode before the block ties onto the parallel bus.
The walk bottoms out at the leaves (there is always a finite number of cells), so the recursion terminates. Series groups merge nodes (voltages add); parallel groups fan out to shared rails (currents add). The top of the tree exposes two terminals, conventionally out and gnd. ✓
Term subcircuit — in SPICE, a reusable named block of components (here, one solar cell's equivalent circuit) that the netlist can instantiate many times, like calling a function. cell.buildModel writes one such subcircuit instance per leaf.

With the netlist built and two terminals exposed, you drive it to get an answer: either apply a test source at the bus voltage to read the operating point, or sweep the voltage across a range to trace the array's full current–voltage (IV) curve — exactly the curve idea from Chapter 1, now for the whole array.

Key idea — only the assembly is new Notice what the Circuit class does not reinvent. The single-cell physics (cell), the SPICE model writer (cellBuilder / cell.buildModel), and the simulation runner (ng_sim) are all the proven Chapter-1 primitives, untouched. Circuit only adds the assembly: how those cells are wired together and addressed. New scaffolding, same bricks — which is why the results stay trustworthy.

1.5 Failing one cell on purpose: fault injection

Here is the payoff of giving every cell an address. To study failures you must be able to plant a fault at a chosen location and re-run the simulation. Circuit exposes one method for this:

circuit.fail(cell_id, mode)
looks up cell_id in the flat registry (O(1)) and degrades that one cell according to mode. Everything else in the array is untouched.

The failure modes reuse Chapter-1 cell controls — no new physics, just a chosen way to break a cell:

  • open / dead  →  cell.setSeason(0). With the season (illumination) set to zero, the cell generates no photocurrent — it is producing nothing. This is the default "failed cell," the most common thing to model.
  • short  →  the cell is given a near-zero resistance, so it behaves like a piece of wire — no voltage develops across it.
  • crack(frac)  →  a partial failure: scale the cell's area (and hence its short-circuit current Isc) by the fraction frac, modelling a cracked cell that still works but delivers less current.
Term Monte-Carlo (sweep) — a method that runs the simulation many times, each time planting a failure at a randomly chosen cell (or set of cells), to build up statistics on how the array behaves when faults strike. The name comes from the casino: it leans on random sampling. Each run of the sweep simply calls circuit.fail(...) at a random id, simulates, and records the result.
Key idea The stable per-cell id is what makes the Monte-Carlo study possible. Because every cell has a unique address and circuit.fail can reach any of them in O(1), the sweep can fail "a random cell," re-run, and repeat thousands of times — measuring, for example, how much bus voltage is lost when a random cell dies, or how many simultaneous failures the array tolerates. Without per-cell addressing this entire study would be impossible.

1.6 Constructing a whole circuit from one prototype cell

Typing out a 4000-cell tree by hand is unthinkable. Following PowerPy's style, Circuit offers a constructor that clones a single example cell into the entire tree:

Circuit.fromPrototype(cell, cells_per_line, lines_parallel, modules_per_block, n_blocks, combine_levels=...)
cell is one prototype (e.g. a 3G30 triple-junction cell from Chapter 1); the four counts set the shape at each level; the optional combine_levels overrides the default series/parallel rule per level (the one-value-change lever of §1.2). The constructor builds the full Group/CellRef tree, assigns every id, and fills the flat registry.
Note "Clones one prototype" means every cell starts identical — same four numbers, same area. That is exactly what you want as a baseline: a perfectly uniform array. Faults are then introduced deliberately, one circuit.fail at a time, so any change in the array's output is unambiguously due to the fault you planted, not to cells differing at the start.

1.7 Worked examples

Worked Q1-A — count the cells and find the bus voltage of a tiny circuit

Given: Circuit.fromPrototype(cell, cells_per_line=2, lines_parallel=2, modules_per_block=2, n_blocks=2) with default combines (line series, module parallel, block series, circuit parallel). The prototype cell has Vmp = 2.35 V and max-power current Imp. Find (i) the total number of cells and (ii) the nominal bus voltage and the parallel current multiple.

Step 1 — total cell count. Multiply the four counts together, because the tree nests them: 2 cells/line × 2 lines/module × 2 modules/block × 2 blocks = 2×2×2×2 = 16 cells.
Step 2 — find the series depth (what adds voltage). Voltage adds only along series links. Walking from a cell up to the bus, the series levels are: cells within a line (cells_per_line = 2 in series) and modules within a block (modules_per_block = 2 in series). The parallel levels in between do not add voltage. So the number of cells stacked in series from bottom to top is cells_per_line × modules_per_block = 2 × 2 = 4 cells in series.
Step 3 — nominal bus voltage. Four cells in series at the max-power point: Vbus = 4 × Vmp = 4 × 2.35 V = 9.4 V.
Step 4 — find the parallel width (what adds current). Current adds only across parallel links. The parallel levels are: lines within a module (lines_parallel = 2 in parallel) and blocks within the circuit (n_blocks = 2 in parallel). So the number of parallel current paths is lines_parallel × n_blocks = 2 × 2 = 4 paths, giving a bus current of 4 × Imp.
Cross-check (this proves the topology reading AND the count at once): series-depth × parallel-width = 4 × 4 = 16 = the total cell count from Step 1. The two independent routes to 16 agree, so the series/parallel bookkeeping is self-consistent. As a scale anchor, the same rule on the full array (cells_per_line = 10, modules_per_block = 3) gives 10 × 3 × 2.35 ≈ 70.5 V — right at the project's ~70 V bus, confirming the method scales. ✓
Worked Q1-B — address and fail the 7th cell of block 3, module 2, line 1

Goal: mark one specific cell as dead so the next simulation shows its effect. The cell is cell 7, in line 1, of module 2, of block 3.

Step 1 — build the id from the path. The hierarchical id is the path from the root to the cell, segment by segment: block 3 → B3, module 2 → M2, line 1 → L1, cell 7 → C7. Join with dots: B3.M2.L1.C7.
Step 2 — locate the cell (conceptually). That id is the key into the flat registry: circuit.cells["B3.M2.L1.C7"] is the one cell we mean, reached in O(1). We do not have to walk the tree.
Step 3 — inject the fault. Call circuit.fail("B3.M2.L1.C7", "dead"). Internally this finds that cell and applies cell.setSeason(0), so it stops producing photocurrent; every other cell is untouched.
The single line circuit.fail("B3.M2.L1.C7", "dead") does the whole job. Re-running the netlist build and simulation now reflects exactly one dead cell, at a known location — the atomic operation a Monte-Carlo sweep repeats with random ids. ✓

1.8 Why a recursive tree, not four nested loops? — a Socratic ladder

Ladder — Why represent the array as a recursive tree instead of just four nested Python loops?
Q1. First, the foundation: what is a tree here, in one sentence?
A branching structure whose top node (the whole circuit) splits into child nodes (blocks, then modules, then lines) and bottoms out at leaves (individual cells). Each internal node is a Group that combines its children in series or in parallel; each leaf is a CellRef wrapping one real cell. It is recursive because a Group's children can themselves be Groups — smaller copies of the same thing.
Q2. What does each node need to know so the walk can emit a netlist?
Astonishingly little: just its combine type (series or parallel) and its list of children. That single fact — series vs. parallel — tells the walk whether to chain children head-to-tail (share a node, voltages add) or tie them between two rails (currents add). A leaf needs only its cell and its id, so it can emit cell.buildModel(name=id). The same handful of rules works at every level, which is precisely what a uniform node type buys you.
Q3. Why do we insist on a stable, unique id for every cell?
Because a fault study has to name the cell it breaks. Four anonymous nested loops can count cells but give no permanent handle to "this exact one." A hierarchical id like B3.M2.L1.C7 is a permanent address: it survives across runs, it doubles as the key into the flat circuit.cells registry, and it reads back as a human-checkable path through the tree.
Q4. Now tie it together: how does the tree-plus-id design enable Monte-Carlo fault injection in a way four nested loops would not?
The tree gives correct wiring (so the netlist is right no matter how the levels are nested or flipped); the ids plus the flat registry give O(1) access to any cell by name. Together they let the sweep do its core loop — pick a random id, call circuit.fail(id, mode), simulate, record — thousands of times, cheaply and unambiguously. Four hard-coded loops would bake the topology in (no flipping a level, no custom nesting) and would offer no addressable handle to fail one chosen cell. The recursive tree is what makes the array both reconfigurable and fault-addressable — the two things Chapter 1 could not do.
Pitfall — count series and parallel levels correctly EXAM-HOT Voltage adds only along SERIES segments; current adds only across PARALLEL segments. The classic mistake is to multiply Vmp by the total cell count, or to mix a parallel level into the voltage product. In Worked Q1-A the bus voltage is Vmp × (cells_per_line × modules_per_block) = Vmp × 4 = 9.4 V — not Vmp × 16 = 37.6 V. If you ever find yourself multiplying voltage by every cell in the array, you have folded the parallel levels into the series product. Identify which levels are series (here: line, block) and which are parallel (here: module, circuit) first, then multiply only the series counts for voltage and only the parallel counts for current.

2Vectorising the Thermal Solve

In Chapter 1 we learned that every solar cell is two things at once — an electrical device and a small radiator that can only shed heat to cold space. We also met the little two-node energy balance that decides how hot a cell sits. This section takes that balance and asks a sharper, more computational question: when an array has hundreds of cells and we want to run thousands of Monte-Carlo trials, how do we find every cell's temperature fast? The answer is one of the most important ideas in scientific computing, and PowerPy leans on it heavily: stop looping in Python, and instead solve the same nonlinear equation for all cells at once with a single vectorised Newton–Raphson sweep. We will build that idea from the ground up — defining every term the moment it appears — and finish by watching it find a quantified hot-spot inside a four-cell array.

2.1 Recap — the two-node steady-state balance from Chapter 1

Recall the picture from Chapter 1. We model a single cell as two nodes: a front face (subscript F, the side that looks at the Sun) and a rear face (subscript R, the side bonded to the substrate that looks at the planet and deep space). Each node has its own temperature, and the two are linked by conduction through the thin material between them. At steady state — meaning nothing is changing in time any more — the heat flowing into each node must exactly balance the heat flowing out. "Into minus out = 0" for each node gives us two equations.

We write each balance as a residual: a single number that is zero when the books balance and non-zero when they do not. Call them f1 (front node) and f2 (rear node):

f1(T1, T2) = αF·A·PSun·tilt − εF·A·σ·(T14 − Tsp4) − Pelec + C·A·(T2 − T1)
f2(T1, T2) = (αR·PAlbedo + εR·PIR)·A·tilt − εR·A·σ·(T24 − Tsp4) − C·A·(T2 − T1)
where every symbol means:
T1, T2 — front- and rear-node temperatures, in kelvin (K) (this matters enormously — see the pitfall, because they are raised to the fourth power).
σ (sigma) — the Stefan–Boltzmann constant, σ = 5.670367×10−8 W/m²K⁴, the universal constant that converts a surface's temperature-to-the-fourth into radiated power per unit area.
Tsp — the temperature of deep space, Tsp = 2.7 K (the cosmic microwave background; the cold sink the cell radiates to).
A — the cell area (m²); it scales every flux because power = flux × area.
C — the through-thickness conductance coefficient (conductivity ÷ thickness, W/m²K); C·A is how readily heat flows from the rear node to the front node per kelvin of temperature difference.
αF, αR (alpha) — the absorptivity of the front and rear faces: the fraction of incident light each face soaks up (0–1), a property of the substrate/coating.
εF, εR (epsilon) — the emissivity of the front and rear faces: how efficiently each face radiates heat away (0–1), also from the substrate.
tilt — a geometric factor (0–1) for how squarely the face meets the incoming flux; tilt = 1 means face-on.
PSun, PAlbedo, PIR — the incident solar, reflected-sunlight (albedo) and planetary-infrared flux densities (W/m²).
Pelec — the electrical power extracted from the cell (W). Sign convention: positive when the cell delivers power (that energy leaves as electricity instead of heat); negative when the cell is reverse-biased and dissipating, i.e. acting as a heater.
Term Residual. A residual is the left-over imbalance of an equation you are trying to satisfy, written so that the target value is zero. Here each residual is (heat in) − (heat out) for one node. If you guess a temperature and the residual comes out at f1 = +3, the front node is currently absorbing 3 W more than it sheds, so it must warm up. The whole game is to nudge the temperatures until both residuals are zero.
Key idea Equilibrium is the pair (T1, T2) that makes f1 = f2 = 0 simultaneously. Two unknowns, two equations. Both equations are nonlinear in temperature — because of the radiated-power terms, which go as T4, not T. That fourth power is the whole reason we cannot just solve this with simple algebra and need an iterative root finder.

2.2 The old approach, and why it is the bottleneck

How did PowerPy's legacy code find equilibrium? Straightforwardly, and slowly. For each cell it handed the pair (f1, f2) to scipy.optimize.fsolve — a general-purpose black-box root finder — and let it grind out that one cell's temperatures. Then it moved to the next cell and did it again. In code, the shape was simply a Python for loop:

for cell in array.cells:          # N iterations, in slow Python
    T1, T2 = fsolve(residuals, x0=[300.0, 300.0], args=cell.params)
    cell.T_front, cell.T_rear = T1, T2

This is correct, and for a handful of cells it is perfectly fine. The problem is scale. Consider what a real PowerPy study asks for:

  • An array might have N = hundreds of cells.
  • A robustness study runs many Monte-Carlo trials — thousands of randomised what-if scenarios (degradation, fault patterns, tilt spreads).
  • And the thermal solve itself sits inside an outer loop (Section 3), so it is re-run again and again as temperatures and electrical loads settle.

Multiply those together and the inner fsolve call is executed an enormous number of times. Two costs then dominate:

Why it is slow (1) Python-level loop overhead. Every trip round a Python for loop has interpreter cost — far slower than the same arithmetic in compiled machine code. Do it N × (trials) times and it adds up. (2) A full solver spin-up per cell. fsolve is a general routine that does not know our equations; each call must set itself up, probe the function, and often estimate derivatives numerically — real work, repeated from scratch for every single cell even though all cells obey the same two equations. We are paying a heavyweight, general-purpose price for a problem with a tiny, known, identical structure.

The cure is to throw away the per-cell loop entirely. To do that we need to (a) understand the root-finding method well enough to write it ourselves, and (b) notice a structural fact about our equations that lets us apply it to every cell in one stroke. The next two subsections supply exactly those.

2.3 Newton–Raphson, from scratch

Newton–Raphson (often just "Newton's method") is the classic way to solve f(x) = 0 when f is nonlinear. Start with the one-variable case, because the idea is purely geometric and easy to see.

The scalar method. You have a function f(x) and you want the x where it crosses zero (its root). You make a guess x, and you ask a simpler question than "where is the root of this curve?": you ask "where is the root of the tangent line to the curve at my current guess?" The tangent is a straight line, so that is trivial to answer — and because a smooth curve hugs its tangent near the point of contact, the tangent's root is usually a much better guess than where you started. Repeat, and you home in fast.

Concretely: at the point x the curve has height f(x) and slope f′(x) (the derivative). The tangent line through that point hits zero after a horizontal step of f(x) / f′(x). So the update rule is:

x ← x − f(x) / f′(x)
Read "←" as "becomes": compute the right-hand side, then let that be your new x. Geometrically you are sliding down the tangent line until it crosses the x-axis, and using that crossing as the next guess.
Term Derivative f′(x). The instantaneous slope of the curve — how much f changes per unit change in x. A big slope means a small step reaches zero; a shallow slope means you must step far. That is exactly why the update divides by the slope.
Term Quadratic convergence. Near the root, the number of correct digits roughly doubles every iteration. If you have 2 good digits now, expect ~4 next step, ~8 the step after. This is why Newton needs only a handful of iterations: the error does not shrink by a fixed factor each time (that would be linear) — it shrinks roughly as the square of the previous error. We will literally watch this happen in the worked example: the residual falls 3.96 → 0.38 → 0.013 → 1.5×10−5 → 2×10−11.

The system version. Our problem is not one equation in one unknown; it is two equations (f1, f2) in two unknowns (T1, T2). Write the unknowns as a vector x = (T1, T2) and the residuals as a vector f(x) = (f1, f2). Newton's idea is unchanged — replace the curve by its linear approximation and solve that — but the single slope f′ is replaced by a whole matrix of slopes, because each residual now varies with each unknown.

Term Jacobian matrix J. For a vector function f(x), the Jacobian is the grid of all first-order partial derivatives: its entry in row i, column j is Jij = ∂fi/∂xj — "how much residual i changes when only unknown j is nudged." It is the multi-dimensional generalisation of the scalar slope f′. For our 2-equation system it is a 2×2 matrix.

With the slope generalised to the Jacobian, division by the slope becomes multiplication by the matrix inverse J−1, and the update reads:

x ← x − J−1 f(x)
Exactly the scalar rule with "divide by the slope" upgraded to "apply the inverse of the slope matrix." Solve the linear system J Δx = −f for the correction Δx, then add it to x.

2.4 The 2×2 Jacobian for our cell

Let us actually build J for our two residuals by differentiating them. We need four partial derivatives. The trick is that almost every term is linear in T1 or T2 (its derivative is just a constant), and the only nonlinear term is the quartic T4, whose derivative is 4T3. Differentiate f1 first.

Holding T2 fixed and differentiating f1 with respect to T1: the absorbed solar term is constant (→ 0); the emission term −εFAσT14 gives −4εFAσT13; the conduction term +CA(T2−T1) contributes −CA. Differentiating instead with respect to T2: only the conduction term depends on T2, giving +CA. Doing the same for f2 yields the four entries:

∂f1/∂T1 = −4·εF·A·σ·T13 − C·A      ∂f1/∂T2 = +C·A
∂f2/∂T1 = +C·A                ∂f2/∂T2 = −4·εR·A·σ·T23 − C·A

Naming the four entries a, b, c, d for compactness, the Jacobian is:

J = [ [ a, b ],  [ c, d ] ]   with   a = −4εFAσT13 − CA,   b = c = CA,   d = −4εRAσT23 − CA
Notice b = c = CA: the off-diagonal terms are equal and positive, because the single conduction link couples the two nodes symmetrically. The diagonal terms are negative: heating a node increases both its radiated loss and the heat it conducts away, so its own residual falls — physically a self-stabilising (restoring) response.

2.5 The key insight — block-diagonal, therefore vectorisable

Here is the observation that makes the whole array tractable in one stroke. In the Chapter 1 model there is no lateral conduction — no heat path from one cell sideways into its neighbour. Each cell radiates and conducts only between its own two nodes. Consequently cell k's two equations contain only cell k's two temperatures and nothing from any other cell.

Now imagine stacking all N cells into one giant system of 2N equations in 2N unknowns and writing its Jacobian. Because no equation reaches across to another cell's unknowns, every off-cell partial derivative is zero. What survives is N little 2×2 blocks marching down the diagonal, with zeros everywhere else. That shape has a name.

Term Block-diagonal matrix. A matrix that is zero everywhere except for square blocks placed along its main diagonal. Each block is an independent little matrix; there is no coupling between blocks. Our global thermal Jacobian is block-diagonal with N blocks, each a 2×2 of the [[a,b],[c,d]] form above — one block per cell.
Key idea A block-diagonal system does not need to be assembled and inverted as one big 2N×2N matrix. The blocks are independent, so we can solve each 2×2 on its own. And every 2×2 has the same closed-form inverse, so instead of looping over the blocks in Python we evaluate that one formula for all N cells simultaneously, with the cell index living in a numpy array. That is the leap from "N solver calls" to "one array operation."

We need the inverse of a 2×2 in closed form. For a matrix [[a,b],[c,d]] the determinant is det = ad − bc, and the inverse is the swap-and-negate pattern divided by the determinant:

[ [a,b], [c,d] ]−1 = (1/det)·[ [ d, −b ], [ −c, a ] ]   ,   det = a·d − b·c
Swap the two diagonal entries, negate the two off-diagonal entries, divide everything by the determinant. (The determinant measures how much the matrix scales area; if it is zero the matrix squashes everything onto a line and has no inverse — flagged in the pitfall.)

Apply J−1 to the residual vector (−f) and you get the per-cell Newton correction in two plain scalar formulas. Because b = c here we keep them general anyway:

det = a·d − b·c
ΔT1 = −( d·f1 − b·f2 ) / det
ΔT2 = −(−c·f1 + a·f2 ) / det
T1 += ΔT1  ;   T2 += ΔT2
These are the components of −J−1f. Crucially, in PowerPy every symbol here — T1, T2, a, b, c, d, f1, f2, det — is a length-N numpy array, one slot per cell. The arithmetic above runs elementwise across all cells at once, with no Python loop over cells.
Term Vectorised. An operation is "vectorised" when a single array-level instruction replaces an explicit element-by-element Python loop, letting the work run inside fast, pre-compiled library code (here, numpy's C routines) instead of the slow Python interpreter. T1 += dT1 where T1 is an array of 400 temperatures updates all 400 in one call — that is vectorisation.

So the entire thermal solve collapses to: build the residual and Jacobian arrays for all cells, apply the closed-form 2×2 update, and repeat about 5–8 times until the residuals are below tolerance. Five to eight sweeps — not N × (solver iterations) — and each sweep is a handful of numpy array operations. In rough code:

T1 = np.full(N, 300.0)              # all N front-node temps, kelvin
T2 = np.full(N, 300.0)             # all N rear-node temps
for _ in range(8):                # ~5-8 SWEEPS, not N cells
    f1 = aF*A*Psun*tilt - eF*A*sig*(T1**4 - Tsp**4) - Pelec + C*A*(T2 - T1)
    f2 = (aR*Palb + eR*Pir)*A*tilt - eR*A*sig*(T2**4 - Tsp**4) - C*A*(T2 - T1)
    a  = -4*eF*A*sig*T1**3 - C*A
    d  = -4*eR*A*sig*T2**3 - C*A
    b  = c = C*A
    det = a*d - b*c
    T1 += -( d*f1 - b*f2)/det      # all N cells, elementwise
    T2 += -(-c*f1 + a*f2)/det

Every line operates on whole arrays. The loop that remains is over iterations (a fixed handful), never over cells.

2.8 Where this fits — the outer electro-thermal loop

One honest loose end. We treated Pelec as a given input. In reality it is not known up front: a cell's electrical output depends on its temperature (Chapter 1's four numbers all drift with heat), and its temperature depends on Pelec — a chicken-and-egg coupling. So PowerPy wraps this thermal solve inside an outer electro-thermal loop: solve the circuit at the current temperatures to get Pelec, feed that into the thermal solve to get new temperatures, re-solve the circuit, and iterate until both settle. That outer loop is the subject of Section 3; for now, simply note that the fast vectorised thermal solver we just built is the inner engine it calls repeatedly — which is precisely why making it fast matters so much.

2.9 Socratic ladder — why can we vectorise, and what would break it?

Ladder — from "what is a Jacobian" to "what single assumption would break the vectorisation"
Rung 1. What is a Jacobian, in one sentence?
It is the matrix of all first partial derivatives of a vector function: entry (i,j) is ∂fi/∂xj, the rate at which residual i changes when only unknown j moves. It is the many-variable version of the scalar slope f′ that Newton's method divides by.
Rung 2. Why does Newton's method converge so fast (only a handful of iterations)?
Because it converges quadratically near the root: each step roughly doubles the number of correct digits, since the new error scales as the square of the old error rather than shrinking by a fixed fraction. We saw the residual drop 3.96 → 0.38 → 0.013 → 10−5 → 10−11 in five steps. It works because Newton replaces the curve with its tangent (its best linear approximation) and a smooth curve hugs that tangent ever more tightly as you near the root.
Rung 3. What makes the global Jacobian for the whole array block-diagonal?
The absence of lateral cell-to-cell conduction. Each cell's two residuals depend only on that cell's own two temperatures, so every partial derivative that would link one cell to another is zero. The Jacobian is therefore N independent 2×2 blocks on the diagonal with nothing off the blocks — which is exactly what lets us invert each 2×2 by the same closed-form formula, applied elementwise across all cells (vectorised) rather than cell-by-cell.
Rung 4. What single physical assumption, if dropped, would break the vectorisation — and what would the Jacobian become?
Adding lateral conduction between neighbouring cells. Then each cell's residual would depend on its neighbours' temperatures too, introducing non-zero entries off the diagonal blocks. The Jacobian would no longer be block-diagonal — it would become banded (coupled blocks). The simple elementwise 2×2 inverse would no longer be valid, and you would need a genuine banded/sparse linear solve at each Newton step instead. The whole speed trick rests on that one modelling assumption: no sideways heat flow.

2.10 Pitfalls

Pitfall — kelvin in the T⁴ term The radiated-power terms use T4, and that T must be in kelvin, never Celsius. The fourth power makes the error catastrophic, not cosmetic: at our equilibrium 65 °C = 338 K, and 3384 ≈ 1.3×1010 whereas 654 ≈ 1.8×107 — off by a factor of ~700. Always convert to kelvin before raising to the fourth power. (The space term Tsp = 2.7 K is likewise kelvin.)
Pitfall — dividing by det ≈ 0 The update divides by det = ad − bc. If the Jacobian is near-singular, det → 0 and the correction blows up (or returns NaN). In our problem det stays comfortably away from zero (≈ 0.1 in the worked example, kept off zero by the radiative terms even though the two rows are conduction-dominated and nearly equal), so it is safe here — but a robust implementation should still guard the division: check abs(det) > tiny (or clip det away from zero) before dividing, so a pathological cell cannot produce a NaN that silently poisons the whole array.

3The Electro-Thermal Coupling Loop

3   The electro-thermal coupling loop: making electricity and temperature agree

Up to now we have treated a cell's temperature as if someone handed it to us. In Section 2 we built a vectorised Newton thermal solve that, given how much power each cell dissipates, returns the temperature of every cell at once. But that hides a circularity we can no longer ignore. A cell's IV curve — the very current and voltage it produces — depends on its temperature (Chapter 1 §3: temperature shifts isc, imp, vmp, voc). And a cell's temperature depends on how much electrical power it extracts versus dissipates (Chapter 1 §7 and Chapter 2 §2: the power it does not turn into useful electricity becomes heat). Electricity needs temperature; temperature needs electricity. Neither is known first.

The one idea. Temperature and the electrical operating point are mutually dependent. We cannot compute either in isolation, so instead of solving them separately we hunt for a self-consistent pair — a temperature field and an operating point that, when you feed each into the other, reproduce themselves. That pair is the answer. The whole section is one loop that chases it: build, solve the circuit, compute heat, solve temperature, repeat until nothing moves.

3.1   The chicken-and-egg problem, stated precisely

Imagine you want to know the temperature of one cell in a string. To find it from the thermal solve you need its dissipated power. To know its dissipated power you need its operating point (its voltage and current). To know its operating point you must solve the circuit. But the circuit's behaviour depends on every cell's IV curve — which depends on every cell's temperature, which is the thing you were trying to find. The dog is chasing its own tail.

The way out is not to break the circle but to find the point where the circle closes on itself — a state that is unchanged by going once around the loop. Mathematicians call that a fixed point, and an engineer calls the result self-consistent.

Self-consistent solution / fixed point. A state that maps to itself. If you take the current temperature field, build the cells, solve the circuit, compute the heat, and run the thermal solve, you get a new temperature field. At a fixed point the new field equals the one you started with: solving changes nothing. Symbolically, if the whole "go once around the loop" operation is a function F, a fixed point is a temperature array T* with F(T*) = T*. That T* (and the operating point that goes with it) is the physically real, electrically-and-thermally-agreeing answer. Everything before it is just a guess on the way there.
Why "fixed point" and not "the equation's root"? We could in principle write one enormous coupled system — every circuit equation and every thermal equation stacked together — and solve it in a single shot. But that system is huge (thousands of cells), nonlinear on both sides, and couples a SPICE circuit solver to a thermal solver that do not share a common Jacobian. It is far simpler to alternate: solve the circuit with temperatures held fixed, then solve the temperatures with powers held fixed, and let the two converge to each other. Alternating between two cheap solves is the entire trick. The fixed point is what they converge to.

3.2   The legacy approach, and what we keep from it

The original PowerPy had this idea already, in a method called panel.thermalEquilibrium. Its logic was sound:

  • Guess a temperature for the panel.
  • Build the cells at that temperature and compute the electrical power P_elec.
  • Solve the thermal balance (heat in = heat out) for the temperature that power implies.
  • Repeat, driving the guess and the result together — wrapped in a SciPy minimize call that searched for the temperature making the thermal residual zero.

The idea is exactly the loop we are about to build. The problem was the implementation: it ran per panel, iteratively, and slowly — a general-purpose optimiser repeatedly evaluating a scalar residual, one panel at a time, with no use of the structure of the problem.

Keep the loop, replace the engine. We keep the outer "guess → power → temperature → repeat" loop, because the physics demands it. But we throw out the slow per-panel minimize and make each thermal step the vectorised Newton solve from §2, which updates the temperature of all cells simultaneously from a single power vector. One cheap, structured solve replaces a forest of scalar optimisations.

3.3   The new coupling loop

Here is the loop in full. Read it once as steps, then again as pseudocode — they say the same thing, but the steps explain why and the code pins down exactly what.

  1. Initialise T. Set the temperature array — one entry per cell — to a guess, e.g. 28 °C everywhere. The guess need not be good; the loop will correct it.
  2. Build every cell's model at its current T. For each cell, look up isc/imp/vmp/voc shifted to that cell's temperature via the regressors (Chapter 1 §3), and assemble its equivalent-circuit parameters.
  3. Solve the circuit once. Run ngspice on the whole array at the bus voltage, with savecurrents enabled. Read back, for every cell, its terminal voltage V_i and its branch current I_i.
  4. Form the per-cell power vector. Compute P_elec,i = V_i · I_i for every cell. Where a cell is reverse-biased (driven backwards by its neighbours — a shaded or failed cell), this product is negative: the cell is absorbing electrical power, i.e. it has become a heat source.
  5. Run the vectorised thermal solve. Hand that whole P_elec vector to the Newton solver from §2. It returns a new temperature array T_new — every cell's temperature at once.
  6. Relax. Do not jump straight to T_new. Take a partial step: T ← T + ω·(T_new − T), with a damping factor 0 < ω ≤ 1. (Why we hold back is the whole of §3.4.)
  7. Check convergence. If max|T_new − T| < tol, the temperature stopped moving — stop. Otherwise return to step 2 with the relaxed T.

And the same loop as pseudocode:

# Electro-thermal coupling loop
T = full(n_cells, 28.0)          # step 1: initial guess, °C
omega = 0.5                      # under-relaxation factor, 0 < omega <= 1
tol   = 0.1                      # convergence tolerance, °C

while True:
    cells   = build_models(T)            # step 2: IV curves shifted to T (Ch.1 §3)
    V, I    = solve_circuit(cells,        # step 3: one ngspice solve at the bus voltage
                            v_bus,
                            savecurrents=True)
    P_elec  = V * I                       # step 4: per-cell power; <0 ⇒ heat source
    T_new   = thermal_newton(P_elec)      # step 5: vectorised solve over ALL cells (§2)

    if max(abs(T_new - T)) < tol:         # step 7: nothing moved ⇒ self-consistent
        T = T_new
        break

    T = T + omega * (T_new - T)           # step 6: damped (under-relaxed) update

return T, V, I                            # the self-consistent (temperature, operating point)
One outer iteration = one circuit solve + one thermal solve. Notice how little happens per pass: build the models (cheap array maths), one ngspice run, one element-wise multiply for the powers, one vectorised Newton solve. There is no inner optimiser, no per-cell search. That is what makes replacing thermalEquilibrium's minimize with the §2 solve such a large win.

3.4   Why we damp: under- and over-relaxation

The obvious thing to do at step 6 is set T ← T_new outright — accept whatever the thermal solve says. This is the raw fixed-point update, and on a strongly heating fault it can oscillate or even diverge: the solve overshoots, the next build overcorrects, and the temperature ping-pongs instead of settling. The cure is to move only part of the way toward T_new each pass.

Under-relaxation (0 < ω < 1). Take a fractional step toward the newly computed value instead of the full step: T ← T + ω·(T_new − T). With ω = 0.5 you move halfway; with ω = 0.2 a fifth of the way. It deliberately slows the update to keep it from overshooting. You trade a few extra iterations for stability — almost always a good trade.
Over-relaxation (ω > 1). The opposite: overshoot the computed value on purpose to reach the answer in fewer steps. It can accelerate convergence on well-behaved, gently-coupled problems, but it is risky — on a stiff or strongly heating problem it amplifies exactly the oscillation we are trying to avoid. For this coupling loop we use under-relaxation; over-relaxation is mentioned only so you recognise ω > 1 when you meet it.
T ← T + ω·(Tnew − T)
where
• T = the temperature array we are iterating (one entry per cell, °C)
• Tnew = the array just returned by the thermal solve
• ω = the relaxation / damping factor. ω = 1 is the raw update (full step); 0 < ω < 1 is under-relaxation (partial step); ω > 1 is over-relaxation (overshoot). Setting ω = 0.5 means "believe the new temperature only half-way, then look again".
What damping buys you. A raw update reacts to the full disagreement between old and new temperature every pass, so a violent fault yanks the temperature wildly and the loop chases its own overshoot. Damping feeds back only a fraction of the disagreement, so each pass is a gentle nudge in the right direction. The fixed point is unchanged — at T* = T_new the correction ω·(T_new − T*) is zero for any ω — so damping changes only the path, never the destination.

3.5   Convergence: why only a handful of passes

In practice this loop converges in a handful of outer iterations. The reason is physical: temperature and electrical power are only mildly coupled. A cell warming by ten degrees nudges its IV curve a little, which nudges its dissipation a little, which nudges its temperature a little less, and so on — each round-trip shrinks. There is no runaway feedback for a healthy array, so the disagreement max|T_new − T| falls quickly and the convergence test trips after a few passes.

Because each pass is just one circuit solve plus one cheap vectorised thermal solve (§3.3), a handful of passes is genuinely fast. The cost is dominated by the ngspice solves, and there are only a few of them.

3.6   The hard practical part: reading Vi and Ii back

Everything above assumes step 3 cleanly hands us a voltage and a current for every cell. That assumption is, honestly, where the real engineering risk lives.

Main implementation risk — robustly reading per-cell Vi/Ii from ngspice. For a large netlist, getting each individual cell's terminal voltage and branch current back out of the ngspice raw output is fiddly and easy to get subtly wrong. A mis-parsed node, a current with the wrong sign, or two cells whose results get swapped will quietly poison P_elec and hence every temperature. This is the single biggest place the implementation can break, and it fails silently — the loop still "converges", just to the wrong answer.

Two disciplines make it tractable:

  • Enable savecurrents. This ngspice option records the branch current through each device, so the per-cell currents I_i are actually available to read back rather than inferred.
  • Put each cell's id in its node names. Disciplined, systematic node naming — every node carrying the owning cell's identifier — means the raw output can be parsed back into a per-cell table unambiguously, with no guessing about which voltage belongs to which cell.
Validate on a tiny 2×2 circuit first. Before trusting the read-back on a full array, build a 2×2 cell circuit whose voltages and currents you can work out by hand (or check against a single-cell solve), and confirm the parser returns exactly those numbers in the right order and sign. Only once the 2×2 matches by hand should you scale up. Debugging the parser on a thousand-cell netlist is a nightmare; debugging it on four cells is an afternoon.

3.7   Worked example — one outer iteration narrated EXAM-HOT

Worked example — a single pass of the loop, with a healthy and a failed cell. We start everything at T = 28 °C and walk one outer iteration, focusing on two cells: a healthy one and a reverse-biased failed one.
Step 1 — build at 28 °C and solve the circuit. Both cells' IV curves are evaluated at 28 °C. ngspice solves the array at the bus voltage and (with savecurrents) returns each cell's V_i and I_i.
Step 2 — per-cell power P_elec = V·I. Healthy cell: it is delivering power, so P_elec ≈ +1.1 W (positive — it is a net generator).
Failed cell: it is reverse-biased, so its voltage and current have opposite signs and P_elec ≈ −9.6 W (negative — it is absorbing ~9.6 W, dumping it as heat).
Step 3 — thermal solve. Feed that power vector to the §2 vectorised Newton solve. The healthy cell, dissipating only its small losses, settles near T_new ≈ 39 °C. The failed cell, forced to absorb ~9.6 W, runs away to T_new ≈ 187 °C (these are the §2 results for these powers).
Step 4 — have we converged? The new temperatures (39 °C and 187 °C) differ enormously from the 28 °C we built at — max|T_new − T| is ~159 °C, nowhere near tol. So we do not stop; we take a damped step and iterate.
Step 5 — the damped update on the failed cell (ω = 0.5).
T ← T + ω·(Tnew − T) = 28 + 0.5·(187 − 28) = 28 + 0.5·159 = 28 + 79.5 = 107.5 °C
Instead of leaping to 187 °C in one go, the failed cell moves to 107.5 °C this pass. Next iteration rebuilds at 107.5 °C, re-solves, and the cell continues climbing — approaching 187 °C over the following passes rather than overshooting it.
Sanity check. The arithmetic: 187 − 28 = 159; half of 159 is 79.5; 28 + 79.5 = 107.5. ✓ And the sign logic holds: the negative P_elec on the failed cell is precisely what drives its temperature up, because absorbed electrical power becomes heat. ✓

3.8   Why iterate instead of solving in one shot?

Q·a (foundation). In this loop, what depends on what?
Two dependencies that point at each other. The electrical operating point (each cell's V and I, hence its power) depends on every cell's temperature, because temperature shifts the IV curve (Chapter 1 §3). And every cell's temperature depends on the electrical power it dissipates, because the power it does not export becomes heat (Chapter 1 §7, Chapter 2 §2). Electricity → temperature → electricity: a closed circle.
Q·b (build up). Given that circle, what does it mean to have solved the system — what are we actually looking for?
A fixed point: a temperature field (and matching operating point) that is unchanged by going once around the loop. Build the cells at that temperature, solve the circuit, compute the heat, solve the temperature — and get the same temperature back. That self-consistent state is the physically real answer; any state that changes when you go around the loop is still just a guess.
Q·c (why). If we just set T ← T_new each pass (the raw update), why might the temperature oscillate instead of settling?
Because the raw update reacts to the full disagreement every pass. On a strongly heating fault, the thermal solve overshoots; the next build at that overshot temperature swings the power the other way; the following solve overshoots back. With no damping, each pass corrects by the entire gap and can sail past the answer, so the temperature ping-pongs (or diverges) instead of converging.
Q·d (deep — the target). So why do we iterate between the circuit and the thermal model, with damping, instead of solving them together in one shot?
Solving "in one shot" means stacking every circuit equation and every thermal equation into one giant coupled nonlinear system and solving it simultaneously. That is enormous (thousands of cells), nonlinear on both sides, and welds a SPICE circuit solver to a separate thermal solver that share no common Jacobian — painful to build and brittle to run. Iterating sidesteps all of that: alternate two cheap, well-understood solves (circuit with T fixed; temperature with power fixed) and let them converge to each other. Damping is what makes the alternation safe — by feeding back only a fraction ω of each disagreement, it kills the overshoot from Q·c, costing a few extra passes (which are cheap, §3.5) in exchange for reliable convergence to the same fixed point. We iterate because two cheap damped solves beat one monstrous coupled one.
Pitfall — choosing ω badly. Set ω = 1 (the raw, full-step update) on a strongly heating fault and the loop can overshoot and oscillate — exactly the failure of Q·c, because a 28 °C→187 °C jump in one pass swings the next build's power hard the other way. But set ω too small (say 0.05) and you waste iterations crawling toward the answer in tiny steps, paying for many more ngspice solves than you need. The sweet spot is a moderate value — ω = 0.5 is a sane default here: damped enough to stay stable through a fault, large enough to converge in a handful of passes.
Practice Q3-1. A cell currently built at T = 40 °C has just produced a thermal-solve result of T_new = 150 °C. Using under-relaxation with ω = 0.5, compute the temperature this cell takes this pass. Then repeat with ω = 0.25. Show the arithmetic, and say in one line which choice moves the cell further this pass and what each choice trades off.

Practice Q3-3. The loop's stop test is max|T_new − T| < tol. Outline how you would detect non-convergence — a loop that is oscillating or diverging rather than settling — and name two things you could change to fix it. (Hint: watch the sequence of max|T_new − T| values across passes, and recall §3.4.)
Section recap. A cell's electricity depends on its temperature and its temperature depends on its electricity, so we seek a self-consistent pair — a fixed point the loop maps to itself. The legacy panel.thermalEquilibrium had the right loop but a slow per-panel minimize; we keep the loop and make each thermal step the vectorised Newton solve over all cells (§2). Each pass is build → ngspice solve (with savecurrents) → per-cell P_elec = V·I → thermal solve → damped update T ← T + ω·(T_new − T) → convergence check. Under-relaxation (ω < 1) trades a few passes for stability; ω = 1 can oscillate on a fault, too-small ω wastes passes. It converges in a handful of passes because temperature and power are only mildly coupled. The real risk is reading each cell's V_i/I_i back from ngspice at scale — solved by savecurrents plus cell-id node naming, validated on a 2×2 first.

4Orbit-Driven Fluxes with hapsira

4  Orbit-driven environment with hapsira: real sun, eclipse, albedo and IR

Up to now (§3) we fed the electro-thermal loop a single set of numbers: one sun value, one tilt, one temperature. That is fine for a quick sizing sum, but it quietly assumes the spacecraft sits motionless in eternal sunlight. A real satellite does no such thing. It orbits. As it sweeps around its planet the sunlight it sees, the warmth glowing up from the planet below, and the light bouncing off the cloud-tops all rise and fall — and for part of every orbit the planet's own bulk slides between the satellite and the Sun, plunging the array into darkness and cold. This section teaches how PowerPy builds the time-varying environment from the actual orbit, so that temperature and power come out as curves over time, not as single frozen numbers.

The fluxes that drive the thermal solve — P_Sun (direct sunlight), P_Albedo (reflected sunlight) and P_IR (the planet's own heat) — together with the array tilt, all change continuously around the orbit. Therefore the array temperature T(t) and the electrical power P(t) are time series, not single values. We obtain them by computing the orbit, then evaluating the fluxes at every moment along it.

4.1  Why a single sun value is not enough

Imagine standing on a merry-go-round at dusk with a single lamp at the edge of the playground. As the ride turns you face the lamp (bright), then turn away (your own body in shadow), then the climbing-frame passes between you and the lamp (fully dark). The light on your face is a cycle, not a number. A satellite around a planet lives exactly this cycle once per orbit, and several quantities ride along with it:

  • Direct sun — present for the sunlit part of the orbit, then cut to zero during eclipse (when the planet blocks the Sun).
  • Albedo — sunlight reflected off the planet onto the array. Strongest when the satellite is high over the bright, sunlit face of the planet; nearly absent over the night side.
  • Planetary infrared (IR) — the planet's own thermal glow. Present everywhere (the planet is warm all the way round), but stronger when more of the planet fills the array's view.
  • Tilt — the angle between the array's normal and the Sun line changes as the satellite re-points and moves, so even in full sun the captured flux varies.
Consequence Because every one of these varies, the worst case for the array is not "high noon". It is often the moment just after eclipse exit: the array has been sitting cold in shadow, then is suddenly slammed with full sunlight and asked to deliver power — a thermal and electrical shock. You cannot find that moment with a single sun value; you have to walk the whole orbit. This section's output is exactly that walk, and it sets up a later transient study as a forward hook.

4.2  The tool: hapsira

To know where the satellite is at every instant we need orbital mechanics. PowerPy uses hapsira for this.

hapsira
An open-source Python astrodynamics library: the actively maintained fork of the older poliastro library. (A "fork" is a copy of a project taken over and kept alive by new maintainers after the original slowed down.) The version installed for PowerPy is hapsira 0.18.
astrodynamics
The mathematics of orbits — how a body moves under gravity: where it is, how fast, and where it will be later. (Related word: celestial mechanics.)
ephemeris  /  propagation
An ephemeris is a table of an object's position (and velocity) versus time. Propagation is the act of predicting a future position from the current state and the laws of gravity — i.e. computing that table by stepping the motion forward in time. "Propagate the orbit" means "roll the satellite's position forward and record where it goes."
Why not the old code? The legacy PowerPy used a module eclipse.py built on poliastro and astropy (astropy is a foundational astronomy toolkit — units, time, coordinates). Its albedo function was flagged as broken. The new implementation moves to the maintained hapsira for the orbital geometry and recomputes the albedo flux ourselves with a clean view-factor model (§4.5), so the numbers can be trusted.

4.3  Step 1 — Define the orbit

We first describe the orbit to hapsira as an Orbit object around a central body (the planet we orbit — for example Mars, as in an MSRO-type mission, i.e. a Mars orbiter). There are two equivalent ways to specify it:

  1. From classical orbital elements — six numbers that pin down the size, shape, orientation and current position of the orbit; or
  2. From a state vector — the satellite's position r and velocity v at one instant (3 numbers each).

The six classical elements, one line each:

  • Semi-major axis (a) — half the long axis of the elliptical orbit; sets the orbit's size (and hence its period).
  • Eccentricity (e) — how stretched the ellipse is: e = 0 is a perfect circle, 0 < e < 1 an ellipse; sets the orbit's shape.
  • Inclination (i) — the tilt of the orbit plane relative to the planet's equator, in degrees.
  • Right ascension of the ascending node (RAAN, Ω) — the swivel of the orbit plane about the planet's axis: where the orbit crosses the equator going north.
  • Argument of periapsis (ω) — the rotation of the ellipse within its plane: the angle from the ascending node to the closest point (periapsis).
  • True anomaly (ν) — where the satellite is right now along the ellipse, measured as an angle from periapsis.
Jargon decoder Periapsis = the point of closest approach to the planet; apoapsis = the farthest point. "Ascending node" = the spot where the orbit crosses the planet's equatorial plane heading north. The first two elements (a, e) fix the shape and size; the next three (i, Ω, ω) fix the orbit's orientation in space; the last (ν) fixes the satellite's place on it.

4.4  Step 2 — Propagate

Given the starting Orbit, we ask hapsira to propagate it: step the satellite's position and velocity forward over one or more orbital periods at a chosen time step (say every 30 or 60 seconds). The result is a list of positions versus time — the ephemeris we will hang all the flux calculations on.

Orbital period (time for one lap) of an orbit of semi-major axis a: 
T = 2π · sqrt( a³ / μ )
where a = semi-major axis [m]; μ (mu) = the central body's gravitational parameter [m³/s²], a fixed property of the planet (Mars: μ ≈ 4.28×10¹³ m³/s²). We propagate for at least one full T so the time series covers a complete sunlit-plus-eclipse cycle.
Why propagate at all? Because the very things the thermal solve needs — how much sun, whether we are in shadow, how much of the planet we can see — depend on where the satellite is. No position, no fluxes. Propagation turns "an orbit" into "a sequence of positions", and each position becomes one environment snapshot.

4.5  Step 3 — Per time step, compute the fluxes

At each propagated position (each time step) we evaluate four things: the direct solar flux, the eclipse test, the albedo flux, the IR flux, and the tilt. Take them in turn.

P_Sun — direct solar irradiance (inverse-square law)

The Sun pours out a fixed total power (its luminosity). That power spreads over an ever-larger sphere as you move away, so the irradiance (power per unit area) falls with the square of the distance — the inverse-square law.

E = L_sun / (4π d²)
where E = solar irradiance [W/m²]; L_sun = the Sun's luminosity (total radiated power) [W]; d = distance from the Sun [m]; 4πd² = the surface area of a sphere of radius d. AU = astronomical unit, the mean Earth–Sun distance (≈ 1.496×10¹¹ m).

In practice we rarely plug in L_sun. We anchor to a known point: at 1 AU the irradiance is the solar constant AM0 = 1367 W/m² (AM0 = "Air Mass Zero": the sunlight above the atmosphere, with zero air in the way). Because E ∝ 1/d², scaling AM0 by distance in AU gives the handy working form:

E(d) = 1367 / d²        [d in AU, E in W/m²]
This is the same physics as E = L_sun/(4πd²), just normalised so that d = 1 AU returns 1367 W/m². Farther out (larger d) means less sun.
Critical During eclipse, set P_Sun = 0. The Sun is geometrically blocked, so no direct sunlight reaches the array — full stop.
The eclipse test — umbra and penumbra

Is the satellite in the planet's shadow? The shadow a planet casts is a cone trailing away from the Sun, and it has two zones:

Umbra
The full shadow — the Sun's disc is completely hidden by the planet. No direct sunlight at all.
Penumbra
The partial shadow — the planet covers only part of the Sun's disc, so the satellite gets a dimmed, fractional sun (a brief "partial eclipse" at the edges).

hapsira supplies the geometry — given the satellite, planet and Sun positions it tells us whether (and how deeply) the satellite sits in the shadow cone. Conceptually, the satellite is eclipsed when the planet lies between it and the Sun and it falls inside that shadow cone. When fully in the umbra we zero P_Sun; in the penumbra it can be scaled down in proportion to the visible fraction of the Sun.

P_Albedo — reflected sunlight

Some of the sunlight that strikes the planet bounces back off and a fraction of that lands on the array. Three factors set how much:

Bond albedo (A)
The fraction of all incoming sunlight a planet reflects across all directions and wavelengths (dimensionless, 0–1). Mars ≈ 0.25; Earth ≈ 0.3; cloud-covered Venus ≈ 0.75. "Albedo" just means reflectivity.
Incident solar at the planet
The direct irradiance hitting the planet, i.e. E(d) from the inverse-square law above.
View factor (F)
The fraction of one surface's hemisphere that another surface fills — here, how much of the planet the array "sees" (dimensionless, 0–1). It falls off with altitude (the higher you are, the smaller the planet looks) and with the angle between the sub-solar point and the satellite (over the night side you see little lit surface, so the reflected contribution shrinks). The sub-solar point is the spot on the planet directly beneath the Sun — local noon.
P_Albedo = A · E(d) · F
where A = bond albedo [–]; E(d) = incident solar irradiance at the planet [W/m²]; F = view factor [–]. All the position-dependence lives in F: it shrinks with altitude and as the satellite drifts over the dark side.
P_IR — the planet's own thermal infrared

The planet is warm, so like any warm body it glows in the infrared (IR) (heat radiation, just below visible red). This glow reaches the array from below. We model the planet as an equivalent blackbody at temperature T_planet (a blackbody being an ideal perfect emitter), whose emitted flux follows the Stefan–Boltzmann law, then scale by the same view factor:

P_IR = ( σ · T_planet⁴ ) · F
where σ (sigma) = the Stefan–Boltzmann constant = 5.67×10⁻⁸ W·m⁻²·K⁻⁴; T_planet = the planet's equivalent blackbody temperature [K] (note the fourth power — hotter planets glow far more strongly); F = view factor [–]. Unlike sun and albedo, P_IR does not vanish in eclipse — the planet stays warm on its night side.

This IR is absorbed via the array's rear surface, governed by its rear infrared emittance/absorptance ε (epsilon) — the property describing how readily that surface emits and soaks up thermal radiation (linked into the thermal balance of §3).

tilt — the cosine factor

Finally, tilt is the cosine of the angle between the array's normal (the direction it faces) and the Sun line (the direction to the Sun) at that instant. A flat-on array (angle 0°, cos 0° = 1) catches the full beam; edge-on (90°, cos 90° = 0) catches none. This is Lambert's cosine law, and it modulates the useful solar flux moment by moment.

4.6  Step 4 — Emit a time series of Environment objects

Each time step produces one snapshot of the surroundings: the four fluxes plus tilt plus the resulting temperature. We package each snapshot into PowerPy's existing Environment dataclass — the same structure §3 already understands, which carries albedo, season/irradiance, angles and temperature. Walking the orbit therefore yields a time series of Environment objects, one per step.

We then feed each Environment into the electro-thermal loop of §3. Out comes a time-resolved profile: the array temperature T(t) and the electrical power P(t) right around the orbit — and, by scanning that profile, the worst-case hot-spot over the orbit (e.g. just after eclipse exit, when the array is cold and is then suddenly loaded — the forward hook to a transient study).

The whole pipeline in one breath: Orbit (elements) → propagate (positions vs time) → per-step fluxes (P_Sun with eclipse, P_Albedo, P_IR, tilt) → Environment time series → electro-thermal loop (§3) → T(t), P(t) and the worst case.

4.7  Worked example — solar irradiance at Mars (with units)

Goal: Find the direct solar irradiance at Mars, then the albedo flux on the array, using the inverse-square law and a given view factor.

Step 1 — Get the distance. Mars orbits at about d = 1.52 AU from the Sun. (AU = astronomical unit, the Earth–Sun distance.)
Step 2 — Square the distance. d² = 1.52² = 2.31 (AU², a pure ratio here).
Step 3 — Scale AM0 by inverse-square. Anchor at 1 AU where E = 1367 W/m², then divide by d²: 
E_Mars = 1367 / 1.52² = 1367 / 2.31 ≈ 591 W/m²
Sanity check: Mars should get markedly less sun than Earth. 591 / 1367 ≈ 0.43 → Mars receives roughly 43% of Earth's sunlight. Plausible for a planet half-again as far out. Units: W/m² divided by a dimensionless ratio stays W/m². ✓
Step 4 — Albedo flux. Take Mars bond albedo A = 0.25 and a view factor F = 0.3 at this point in the orbit. Then 
P_Albedo = A · E_Mars · F = 0.25 × 591 × 0.3 ≈ 44 W/m²
Sanity check: 44 W/m² is a small fraction of the 591 W/m² direct beam — reflected light should indeed be a modest top-up, not a rival to direct sun. Units: [–] × [W/m²] × [–] = [W/m²]. ✓

4.8  Socratic ladder — why propagate the orbit at all?

Target question: Why must we propagate the orbit, instead of using one fixed sun value for the whole mission?

Q1. As the satellite travels once around its planet, name at least two things about its surroundings that change.

A. The direct sunlight (full sun on the day side, blocked during eclipse), the reflected sunlight (albedo, strong over the lit side, weak over the dark side), the planet's IR glow reaching it, and the tilt of the array to the Sun. None of these is constant.

Q2. What is an eclipse, and what does it do to the array's power and temperature?

A. An eclipse is the stretch of orbit where the planet blocks the Sun, so the satellite is in the planet's shadow. Direct sun → 0, so the array produces no power and, with the Sun gone, it cools. (The planet's IR continues, slightly slowing the cooling.)

Q3. What is a view factor, and why does it matter for albedo and IR?

A. The view factor is the fraction of the array's hemisphere that the planet fills — how much of the planet the array can "see". Albedo and IR both reach the array from the planet, so both scale with the view factor: see more planet → receive more reflected and emitted flux.

Q4. Why do albedo and IR depend on where the satellite is over the planet?

A. The view factor falls with altitude (higher up, the planet looks smaller) and albedo also depends on the angle to the sub-solar point (over the night side there is little lit surface to reflect). So position changes both how much planet is in view and how much of it is sunlit — hence both fluxes vary along the orbit.

Q5 (the target). So why propagate, rather than use one fixed sun value?

A. Because every flux that drives temperature and power depends on the satellite's position, and the position changes continuously — including a full sunlight-to-eclipse swing each orbit. A single sun value would miss the eclipse entirely (predicting power and warmth that do not exist) and miss the worst-case moment after eclipse exit. Propagating gives the position at every instant, so we can evaluate the true, time-varying environment and get honest T(t) and P(t).

4.9  Pitfalls

Forgetting to zero P_Sun during eclipse. If you leave the direct solar flux on while the satellite is in the planet's shadow, the model will happily predict an impossibly warm, fully powered array sitting in the dark — physically nonsense, and it hides the real cold-soak that makes eclipse exit the worst case. Always gate P_Sun on the eclipse test: in umbra, P_Sun = 0.
Confusing irradiance with the season multiplier. The irradiance E(d) is a physical flux in W/m² (e.g. 591 W/m² at Mars). The "season" multiplier the Environment dataclass carries is a dimensionless scale factor (roughly 1/d²_AU) that says how strong the sun is relative to 1 AU. They encode the same inverse-square physics, but one has units and the other does not. Multiply AM0 (W/m²) by the dimensionless multiplier to recover the irradiance; never treat the bare multiplier as if it were W/m².

5Breakdown and Melt Criteria

5   When does it break? Breakdown and melt criteria

We have spent two sections learning that a reverse-biased cell turns its neighbours' power into heat and runs hot — Chapter 1 §6 told the story, and Chapter 2 §2 nailed a number to it: a failed cell forced to absorb ~9.6 W converged to a front temperature of 187 °C. That is alarming. But "alarming" is not a number a computer can act on. To run the Monte-Carlo failure sweep (§6) we need a precise, mechanical rule the code can apply to every cell, every trial, with no human in the loop: given this cell's converged state, is it destroyed — yes or no? This section defines that rule. The whole point is to convert the intuition "it gets dangerously hot / it is being driven dangerously hard" into a go/no-go flag.

The one idea. A cell (or the substrate under it) is flagged destroyed if either of two independent criteria trips: (1) its converged temperature reaches a structural limit — did it get hot enough to soften or melt the assembly? — or (2) the power it dissipates in reverse bias reaches a power limit — is it being driven backwards hard enough to damage it, regardless of the steady temperature? Temperature captures the thermal outcome; reverse-power captures the electrical stress. We use both, OR-ed together, because either one alone can miss a failure the other catches.

5.1   Two questions, two criteria

There are two genuinely different ways a cell can be ruined, and they ask different questions. The first is about the steady thermal state: once the electro-thermal loop of §3 has converged, how hot did this cell (and the substrate it is bonded to) actually get? If that settled temperature is high enough to soften the structure or melt the bond, the cell is destroyed by heat. The second is about electrical dose: how much power is being forced through this cell while it is reverse-biased? A large reverse dissipation can damage a cell through very localised heating, cracking, or interconnect failure even if the cell-averaged steady temperature never looks frightening. These are not the same question, and — as §5.3 shows with concrete numbers — a cell can fail one test while comfortably passing the other.

Why a flag at all? The Monte-Carlo sweep (§6) runs thousands of random failure patterns. For each, it must aggregate "how many cells / which patterns destroyed the panel?" That aggregation needs a Boolean per cell, not a temperature to be eyeballed. The criteria here are exactly the function that turns each cell's converged physical state into the destroyed: True/False that §6 counts and §7 stores.

5.2   Criterion 1 — the temperature limit

The first criterion compares the cell's converged front temperature against a material limit:

destroyed (thermal) ⇔  Tfront ≥ Tlimit
where
• Tfront = the cell's (or substrate's) converged front-face temperature from the §3 loop, in °C
• Tlimit = the temperature at or above which the bonded assembly is considered destroyed, in °C — a project input, not a universal constant

The subtle part is choosing Tlimit, and to choose it you must distinguish two very different temperatures of a material.

Melting point. The temperature at which a solid turns to liquid. For the aluminium that forms the substrate's honeycomb core, this is about 660 °C. It is a clean, well-defined physical constant — and, for our purposes, a trap: by the time aluminium actually melts, the structure failed long ago.
Softening / structural (bond) limit. The much lower temperature at which the assembly stops doing its job — where aluminium loses mechanical strength and, critically, where the adhesive bond holding the cell to the substrate begins to fail. Aluminium softens and creeps well below its melting point, and the bonding adhesives degrade lower still. For a bonded solar-panel assembly a conservative structural limit is of order ~150–200 °C. This — not the melting point — is the number that belongs in Tlimit. The exact value is a project input set by the materials and bond used.
The substrate is aluminium honeycomb — but use its structural limit, not its melt. The core melts at ~660 °C, yet the panel is destroyed far earlier: the bond lets go and the aluminium softens somewhere around 150–200 °C. Sizing Tlimit at 660 °C would declare a panel "fine" at 400 °C — a temperature at which it has, structurally, already failed. Conservative engineering uses the lower limit.

5.3   Criterion 2 — the reverse-power limit

The second criterion ignores temperature entirely and looks at how hard the cell is being driven backwards. It only applies to a cell that is actually reverse-biased — one whose terminal voltage has gone negative because its neighbours are forcing current through it (Chapter 1 §6, and the negative P_elec of §3).

Prev = |Vi · Ii|  only when Vi < 0;     destroyed (reverse-power) ⇔  Prev ≥ Plimit
where
• Vi, Ii = the cell's converged terminal voltage (V) and branch current (A) from the §3 solve
• Prev = the magnitude of the power dissipated in reverse bias, in W. The absolute value is taken because we care about the amount of power being dumped as heat, not its book-keeping sign
• Plimit = the per-cell reverse dissipation the assembly can survive, in W — again a project input
• the gate Vi < 0 is essential: a forward cell (V > 0) is generating power, and no amount of forward power should ever trip this test (see the pitfall in §5.6)

Why bother with this when we already have a temperature test? Because raw dissipation and steady temperature are not the same physics. Prev measures the electrical stress — the rate at which energy is being forced into the cell — and damage can be driven by that stress in ways the cell-averaged converged temperature does not reveal. Energy concentrated into a tiny defect, a cracked interconnect, or a few square millimetres of silicon can ruin a cell whose bulk temperature, smeared over its whole area by the lumped thermal model, still reads modest. Reverse-power catches the dose; temperature catches the outcome.

5.4   Why both — and the cases each one catches alone EXAM-HOT

If the two criteria always agreed, one would be redundant. They do not. Here are the two asymmetric cases — each a real failure that one test catches and the other misses — which together are the entire justification for OR-ing them.

Case A — temperature catches what reverse-power misses (the poorly-radiating cell). Imagine a cell whose reverse dissipation is only moderate — say Prev = 1.2 W, comfortably below a Plimit = 2 W. The reverse-power test says PASS. But suppose this cell radiates heat poorly (a low-emissivity patch, a blocked radiating path, a hot neighbour on every side). With nowhere to dump even 1.2 W, its converged Tfront climbs to, say, 170 °C ≥ Tlimit = 150 °C — the temperature test trips.
What this proves. Low electrical dose, but a hot steady state because the heat cannot escape. Reverse-power alone (1.2 W < 2 W) would have waved this cell through as healthy; the temperature criterion is what catches it. A modest Prev can still cook a cell that cannot radiate. ✓
Case B — reverse-power catches what temperature misses (intense, localised heating). Now the opposite: a cell driven hard backwards, Prev = 9.6 W ≥ Plimit = 2 W — the reverse-power test trips immediately. But imagine the heating is extremely localised (a pinhole short, a tiny avalanche site): the energy ruins a few square millimetres of silicon, yet the cell-averaged converged Tfront that the lumped thermal model reports is only, say, 120 °C < Tlimit = 150 °C — the temperature test passes.
What this proves. High electrical dose, but a whole-cell steady temperature that looks survivable because the lumped model smears the hot-spot over the full area. Temperature alone would have called this cell healthy; the reverse-power criterion is what catches it. ✓
That is why it is "either trips." Case A is a thermal failure with a passing reverse-power number; Case B is an electrical-dose failure with a passing temperature number. Neither test is a superset of the other, so the only safe go/no-go rule is the logical OR: flag the cell destroyed if either criterion trips. For a safety check you take the union of the dangers, never the intersection.

5.5   The criteria in action: a bypass diode, and the same cell with and without it EXAM-HOT

The two criteria earn their keep by quantifying exactly when the panel's protection is enough. Recall the protection mechanism from Chapter 1 §5–§6: a bypass diode sits across a cell and, if that cell is driven into reverse, conducts the string current around it — clamping the cell's reverse voltage to roughly the diode's forward drop, about −0.7 V. Remove that diode and a reverse-biased cell can be pushed all the way to its own reverse breakdown voltage, of order −20 V, before it conducts. Same current either way — but a factor of nearly 30 in voltage, and therefore in dissipated power. We will run the same physical cell through the criteria twice: once protected, once not.

Worked example (a) — the protected cell (bypass diode present). A failed cell is carrying the string current I = 0.48 A. Its bypass diode clamps its voltage to V = −0.7 V. Limits for this assembly: Plimit = 2 W, Tlimit = 150 °C.
Step 1 — reverse-power criterion. The cell is reverse-biased (V = −0.7 V < 0), so the test applies.
Prev = |V · I| = |(−0.7 V)(0.48 A)| = |−0.336 W| = 0.34 W
Compare: 0.34 W < Plimit = 2 W ⇒ PASS on reverse-power.
Step 2 — temperature criterion. Dissipating only 0.34 W, this cell barely warms; its converged front temperature is Tfront = 45 °C. Compare: 45 °C < Tlimit = 150 °C ⇒ PASS on temperature.
Verdict. PASS on both criteria ⇒ cell not destroyed. The bypass diode did its job: by holding the reverse voltage to −0.7 V it kept dissipation to a third of a watt, far below anything that could overheat or over-stress the cell. ✓
Even the passing cell shows why we still need the temperature test. Example (a) passes both, yet it quietly makes the case for keeping criterion 1. That same modest 0.34 W is harmless only because this cell can radiate it away to settle at 45 °C. Put an identical 0.34 W into a cell that radiates poorly (Case A's mechanism) and its Tfront could still climb past 150 °C — a failure the reverse-power test, seeing only 0.34 W, would never flag. The low Prev guarantees nothing about the steady temperature; that is precisely the gap the temperature criterion exists to close.
Worked example (b) — the unprotected cell (no bypass diode). The same failed cell, same string current I = 0.48 A — but now with no bypass diode, so it is pushed to its reverse-breakdown voltage V = −20 V. Same limits: Plimit = 2 W, Tlimit = 150 °C.
Step 1 — reverse-power criterion. Still reverse-biased (V = −20 V < 0):
Prev = |V · I| = |(−20 V)(0.48 A)| = |−9.6 W| = 9.6 W
Compare: 9.6 W ≥ Plimit = 2 W ⇒ FAIL on reverse-power (it exceeds the limit nearly fivefold).
Step 2 — temperature criterion. This is the very same 9.6 W that drove the §2/§3 worked example: forced to absorb it, the cell's converged front temperature runs away to Tfront = 187 °C. Compare: 187 °C ≥ Tlimit = 150 °C ⇒ FAIL on temperature.
Verdict. FAIL on both criteria ⇒ cell destroyed. Here the two tests agree — the dose is so brutal that it both over-stresses the cell electrically and cooks it. Note this is one cell and one diode telling the whole story: clamping −20 V down to −0.7 V collapses 9.6 W to 0.34 W, and turns a double failure into a double pass. The criteria are what let us state that the protection was the difference between destroyed and fine. ✓

5.6   What gets recorded per cell

The criteria are not evaluated in a vacuum — they are computed as part of a per-cell record written after the §3 loop converges. For every cell, the code stores:

  • Tfront — converged front-face temperature (°C);
  • Trear — converged rear-face temperature (°C);
  • Pelec — the signed electrical power V·I (W): positive if the cell generates, negative if it absorbs (§3);
  • the reverse flag — whether V < 0 (is this cell reverse-biased at all?);
  • Prev — the reverse dissipation |V·I| when reverse-biased, else 0 (W);
  • which criterion (if any) tripped — none, temperature, reverse-power, or both.
This record is the bridge to the rest of the chapter. It is precisely what the data layer (§7) persists for each trial, and what the Monte-Carlo sweep (§6) aggregates across thousands of random failure patterns to answer "which patterns destroy the panel, and how often?" The destruction flag computed here is one column of that record; the raw T, P and flags are kept alongside it so later analysis can ask questions the single Boolean cannot answer.

5.7   Connecting the criteria to the bypass diode

Pulling §5.5 together as a principle: the bypass diode is the panel's defence against reverse-bias destruction, and the two criteria are how we measure whether that defence held.

The diode sets Prev; the criteria judge it. With a bypass diode, reverse voltage is clamped to ~−0.7 V, so Prev = 0.7 V × I — for a 0.48 A string, 0.34 W, usually well under Plimit. Without it, the cell can reach reverse breakdown ~−20 V, so Prev = 20 V × I — 9.6 W, far over Plimit. The criteria quantify exactly when the protection is or is not enough: the same cell flips from a double PASS to a double FAIL the moment you remove its diode.

5.8   Socratic ladder — why two criteria, not one?

Q·a (foundation). What does the temperature criterion actually measure, and against what does it compare?
It measures the cell's (or substrate's) converged steady-state front temperature Tfront — the settled outcome of the §3 electro-thermal loop, after the cell has finished heating up. It compares that against a structural temperature limit Tlimit (the softening/bond limit of the aluminium-honeycomb assembly, ~150–200 °C — not the ~660 °C melting point). In one phrase: did it end up hot enough to soften or unbond the structure? It is a test on the thermal result.
Q·b (build up). What does the reverse-power criterion measure instead, and why is it a different question?
It measures the electrical stress / dose: Prev = |V·I|, the rate at which power is being forced into the cell while it is reverse-biased (V < 0), compared against Plimit. It is a different question because dissipation is not temperature: how hard the cell is driven backwards is not the same as how hot its bulk steadily becomes. A large Prev concentrated in a tiny region can damage the cell even if the area-averaged converged temperature stays modest. In one phrase: how hard is it being driven backwards? It is a test on the electrical cause.
Q·c (the crux). Give one failure each criterion catches that the other would miss.
Temperature catches, reverse-power misses: a poorly-radiating cell with a modest Prev below Plimit (say 1.2 W < 2 W → reverse-power says PASS), whose heat cannot escape so its Tfront climbs past Tlimit (e.g. 170 °C ≥ 150 °C). Low dose, hot steady state — only the temperature test flags it (Case A). Reverse-power catches, temperature misses: an intensely localised fault with Prev ≥ Plimit (9.6 W → reverse-power says FAIL) that ruins a few mm² of silicon, while the cell-averaged Tfront the lumped model reports stays below Tlimit (e.g. 120 °C < 150 °C). High dose, survivable-looking bulk temperature — only the reverse-power test flags it (Case B).
Q·d (deep — the target). Given those two cases, why is "either trips" the right rule for a go/no-go design check — rather than "both must trip", or just picking one criterion?
Because the two tests detect non-overlapping dangers, and for a safety decision you must take the union of all the ways the panel can die, not the intersection. Picking one blinds you to whichever failure mode that test cannot see — drop the temperature test and Case A's poorly-radiating cell sails through; drop reverse-power and Case B's localised fault sails through. Requiring "both must trip" (logical AND) is even worse: it would only flag a cell that is simultaneously over-temperature and over-power, so it would clear both Case A and Case B — each a real destruction — because each trips only one test. The conservative, defensible go/no-go rule is the logical OR: if any credible criterion says "destroyed," treat it as destroyed. A design check should err toward catching failures, and OR is what does that.

5.9   Pitfalls

Pitfall 1 — forgetting the sign gate on reverse-power. The reverse-power test must compare |V·I| against Plimit only when V < 0. Sign decides everything here: a forward cell delivering 9.6 W of useful power is perfectly healthy — that is the array doing its job — whereas a reverse cell dissipating 9.6 W is destroyed. Same magnitude, opposite meaning. If you drop the V < 0 gate and test |P| unconditionally, you will flag every strong generator as a failure and the sweep becomes nonsense. Gate first on the reverse flag, then take the magnitude.
Pitfall 2 — using the aluminium melting point as Tlimit. The honeycomb core melts at ~660 °C, but the assembly is destroyed far earlier — the bond fails and the aluminium softens around ~150–200 °C. Setting Tlimit = 660 °C would pronounce a 400 °C panel "healthy" when it has structurally failed. In our examples Tlimit = 150 °C is the structural limit — roughly four times below the melt — and that lower number is the one that matters. Conservative wins: melt is the wrong limit.

6The Monte-Carlo Failure Sweep

6   The Monte-Carlo failure sweep: which failures, and how bad

The previous sections built the machinery for analysing one damaged array: Section 1 gave us a Circuit we can reach into and break by name, Section 3 gave us the electro-thermal loop that finds the self-consistent temperatures and operating point, and Section 5 gave us the breakdown criteria that decide whether any cell has been pushed past its safe limit. Each of those answers the question "given this fault, what happens?". This section answers the question one level up: which faults should we even ask about, and how do we turn thousands of individual fault answers into trustworthy statements about the whole design? That is what the Monte-Carlo failure sweep is for. It is the outermost driver — the thing that decides which failures to simulate, calls the §1+§3+§5 stack once per failure, and gathers the results into rankings and distributions.

6.1   Recap: why we sample instead of enumerate (Chapter 1 §8)

Chapter 1 §8 introduced the Monte-Carlo idea and the wall it is built to get around. Both halves matter here, so let us restate them.

Monte-Carlo method. A way to estimate a quantity you cannot compute exactly by running many random trials and averaging their outcomes. Instead of evaluating every possibility, you sample a manageable number of them at random and let the average of the samples stand in for the true answer. The more trials you run, the closer the estimate.
The 1/√N error law. The statistical error of a Monte-Carlo estimate shrinks in proportion to 1/√N, where N is the number of trials. Concretely: to halve the error you must quadruple the number of runs (because √(4N) = 2√N). This is slow but utterly reliable — it does not care how complicated each trial is. We make this precise in §6.7.
The one idea. An array of N cells has 2N possible failure combinations (each cell is either alive or dead) — for even a few hundred cells that number is larger than anything we could ever simulate. So we do not enumerate failures; we sample them. The whole section is one driver that picks a failure pattern, runs the coupled solve, records what happened, resets, and repeats — then summarises the pile of results. Choosing which patterns to sample, and how many, is the real skill.
Why 2N is hopeless, in one number. Suppose N = 600 cells (a realistic array from §1). Then 2600 is roughly 10180 — vastly more than the number of atoms in the observable universe (~1080). Enumerating every failure combination is not "slow"; it is physically impossible. Sampling is not a shortcut we take for convenience — it is the only road that exists.

6.2   What one RUN is

The atom of the whole sweep is a single run: one failure pattern, simulated end to end, with its outcomes written down. Everything else is just "do many runs and look at the pile". Here is exactly what one run does.

  1. Pick a failure pattern. Choose a set of cell ids to break — one cell, or k cells, or each cell with some probability, depending on the sampling mode (§6.4).
  2. Fail those cells. For each chosen id, call circuit.fail(id,"dead"), which marks that cell as a dead element in the netlist (Section 1 gave us per-cell addressability precisely so we could do this).
  3. Solve the coupled model. Run the electro-thermal loop (§3): build cells at their temperatures, solve the circuit, form per-cell power, solve the thermal field, iterate to the self-consistent state.
  4. Record the outcomes. From the converged state, write down four numbers (and one location):
    • total array power — how much the whole array still delivers, in W;
    • maximum cell temperature — the hottest cell, in °C;
    • number of hot-spots — how many cells exceed a hot-spot threshold;
    • breakdown tripped? — whether any cell tripped a §5 breakdown criterion, and where (which cell/line), so we can map vulnerability, not just count it.
  5. Reset. Un-fail the cells (restore the circuit to pristine) so the next run starts from a clean array, not an accumulating wreck.
  6. Repeat. Go back to step 1 for the next pattern.
Pitfall — forgetting to reset. Step 5 is easy to drop and fails silently. If you do not restore the circuit between runs, the dead cells from run 1 are still dead in run 2, so run 2 is really testing 3 failures, run 3 tests 6, and your "single-cell failure sweep" is quietly a "cumulative catastrophe sweep". Every run must start from the same pristine array. Either reset explicitly, or rebuild a fresh Circuit per run.

6.3   The three sampling modes — what each one is for

The pattern in step 1 is the only thing that changes between the three modes the project wants. They are not interchangeable: each one is shaped to answer a different question. Define each, then read the contrast table.

Mode 1 — POSITION sweep. Fail one cell at a time, each location in turn (or each line in turn), and run the solve once per location. It is deterministic — there is no randomness, you simply march through every position — and costs O(N) runs (one per cell, or one per line). It answers: "where is the array most vulnerable?" — which single position, if it fails, hurts most, and which hurts least.
Mode 2 — COUNT sweep. Fail k cells at once, for k = 1, 2, 3, …, choosing the k cells at random and repeating several random draws at each k. By plotting a damage metric against k you see how damage grows with the number of simultaneous failures. It answers: "how many failures can we tolerate?" — at what failure count does power loss or peak temperature cross an unacceptable line.
Mode 3 — RANDOM sampling. Draw many random failure patterns — each cell fails independently with some probability p, or each pattern is k randomly chosen cells — and record the metric for every one. With enough patterns you build the full statistical distribution of power loss / max temperature / hot-spot count. It answers: "what is the spread of outcomes?" — not just the average damage but its variability, the tail, the worst-case-you-might-realistically-see.

The count and random sweeps are statistical — they cannot ask every question (there are too many), so they sample, and in exchange they can speak about combinations and spreads the position sweep can never reach. You pick the mode by matching it to the question you actually have.

6.4   Outputs and analysis: ranking, histograms, robustness

A sweep produces a pile of recorded outcomes; the analysis turns that pile into decisions. Three things come out of it.

  • Ranking — most vs least damaging. Sort the positions (mode 1) or patterns (modes 2/3) by a damage metric — say power loss, or peak temperature. The top of the list is the array's weak spot (the cell or line you most want to protect or add redundancy around); the bottom is where a failure barely matters. A position sweep naturally produces a map of vulnerability by location.
  • Histograms. For the random sweep, draw a histogram of each recorded metric — power loss, max temperature, hot-spot count. A histogram is a bar chart of how often each outcome range occurs; it shows the whole shape of the result, including the average, the spread, and the dangerous tail, rather than collapsing everything to one number.
  • Robustness comparison between layouts. Run the same sweep on two different Circuit configurations — say a 1-line-per-module layout versus a 2-lines-per-module (redundant) layout — and compare their rankings and histograms. The layout whose distribution sits at lower power loss / lower peak temperature for the same failures is the more robust design. This is how the sweep stops being a description of one array and becomes a tool for choosing between designs.
Comparing layouts is the whole point. A single histogram tells you how one design behaves under failure. Two histograms, from the same sweep run on two designs, tell you which design to build. Keep this in mind through §6.5 and §6.8 — almost every reproducibility rule below exists so that a layout-vs-layout comparison is honest.

6.5   Reproducibility: fix the seed

The count and random modes use a pseudo-random generator to choose which cells fail. A "pseudo-random" generator is not truly random — it is a deterministic recipe that produces a stream of numbers that look random. That recipe has a starting value, and controlling it is how we make a "random" experiment repeat exactly.

Seed. The starting value of a pseudo-random generator. The generator turns the seed into a long sequence of pseudo-random numbers; the same seed always produces the same sequence. So fixing the seed makes a Monte-Carlo run reproducible: re-run it and you get bit-for-bit the same failure patterns, the same results, the same histogram.
Why fixing the seed is essential — two reasons. (1) Fair comparison. To compare two layouts honestly you must hit them with the identical set of failure patterns. Same seed → same patterns → any difference in the results is a difference in the designs, not in the dice. (2) Debugging. When a run does something strange, a fixed seed lets you reproduce the exact failing pattern and inspect it, instead of chasing a bug that vanishes on the next run.

6.6   How many runs? Convergence and the standard error

For the random sweep we must decide when to stop. Run too few patterns and the estimated mean (say, mean power loss) is noisy and might mislead; run too many and we waste compute. The §8 1/√N law, made precise, tells us exactly how many we need.

Sample standard deviation s. A measure of how spread out the individual run outcomes are around their average. If every run gives nearly the same power loss, s is small; if outcomes scatter widely, s is large. It describes the spread of one run's result.
Standard error (of the mean), SE. How uncertain the estimated average is — i.e. how far the mean of your N runs is likely to be from the true mean. It is not the spread of individual runs; it is the spread of the average, and it shrinks as you add runs: SE = s/√N. This is the 1/√N law of §8 written out — the error of the estimated mean falls like 1/√N.

• s = sample standard deviation of the per-run outcomes (e.g. W)
• N = number of runs.
Rearranged to solve for the runs needed to reach a target standard error SEtarget:
N = (s / SEtarget)2.  Equivalently, scaling from a known run: Nnew = Nold · (SEold / SEtarget)2.
Worked example A — runs needed for a target standard error. EXAM-HOT We have run 100 random patterns and the estimated mean power loss has a standard error of 8 W. We want to tighten that to 2 W. How many runs do we need in total?
Step 1 — identify what we know, with units. Nold = 100 runs; SEold = 8 W; SEtarget = 2 W.
Step 2 — use the scaling form. Because SE = s/√N and s does not change as we add runs, the runs scale as the inverse square of the error:
N = Nold · (SEold / SEtarget)2 = 100 · (8 W / 2 W)2
Step 3 — arithmetic. 8 / 2 = 4; 42 = 16; 100 · 16 = 1600 runs.
Sanity check. We cut the error by a factor of 4 (from 8 W to 2 W). The 1/√N law says error falls like 1/√N, so cutting error 4× needs 42 = 16× the runs: 16 × 100 = 1600. ✓ The units cancel inside the ratio (W/W), leaving a pure count, as a number of runs must be. ✓

In practice you do not even need s up front: watch the running estimate. Plot the estimated mean (and a band of ±SE around it) as runs accumulate. Early on it lurches; as N grows it settles into a narrowing band. When the band is tighter than the precision you need — and the estimate has stopped drifting — you have run enough. That stabilising curve is the convergence you are looking for.

6.7   Cost and parallelism: why a big sweep is affordable

A sweep of thousands of runs sounds expensive. Two facts make it cheap. The first is about structure; the second is about where the time goes.

Embarrassingly parallel. A workload made of pieces that are completely independent — no piece needs any result from another, and they share no state. Such a workload splits across processors with almost no coordination overhead, giving a near-linear speed-up (8 cores ≈ 8× faster). The name is not an insult; it means the parallelism is so obvious it is "embarrassing" how little cleverness it takes.
Each run is independent → the sweep is embarrassingly parallel. One run fails some cells, solves, records, and resets — it touches nothing any other run needs. So you can hand different runs to different CPU cores and they never have to talk. A library such as joblib farms the runs across cores for you, and the wall-clock time drops by roughly the number of cores.
The cost driver, and why §2 matters. Each run is one (or a few) circuit solves plus the thermal solves of the §3 loop. The thermal solves are the vectorised Newton solves from Section 2 — they update every cell at once and are cheap. So the per-run cost is dominated by the circuit solve, and the thermal side stays light no matter how many runs you do. That vectorisation from §2 is precisely what makes a thousand-run sweep affordable rather than ruinous: a slow per-cell thermal solve multiplied by thousands of runs would sink the whole exercise.
Worked example B — sizing a position sweep and its wall-clock. EXAM-HOT An array has 600 cells. We want a full position sweep: fail every single cell, one at a time, and solve. (a) How many runs? (b) If one run takes 0.5 s, what is the wall-clock on 1 core versus 8 cores?
Step 1 — number of runs. A position sweep fails one cell at a time, every location in turn → one run per cell → 600 runs. (This is the O(N) cost from §6.3 with N = 600.)
Step 2 — wall-clock on 1 core. Runs are done one after another: 600 runs × 0.5 s/run = 300 s (= 5 minutes).
Step 3 — wall-clock on 8 cores. The runs are independent (embarrassingly parallel), so 8 cores share the work: 300 s / 8 = 37.5 s ≈ 38 s.
Sanity check. 600 × 0.5 = 300 ✓. 300 / 8 = 37.5 ✓. Note 37.5 s is the ideal linear speed-up; real parallelism carries a little overhead (spawning workers, gathering results), so expect about 38 s or a touch more — near-linear, not perfectly linear. That gap between "37.5" and "~38" is exactly the coordination cost the word embarrassingly promises will be small. ✓

6.8   Variance reduction: the same accuracy with fewer runs

The 1/√N law is harsh — quadrupling runs to halve the error. Two standard techniques get the same accuracy from fewer runs by sampling more cleverly than "uniformly at random".

Stratified sampling. Divide the space of failure patterns into regions (strata) — e.g. by failure count, or by array region — and deliberately spend a planned share of runs in each stratum, instead of leaving coverage to chance. This guarantees every region is represented and removes the luck of random under-sampling, shrinking the variance of the estimate.
Importance sampling. Spend more runs where the outcome matters most — here, where damage is likely or severe — and fewer where little happens, then re-weight the results to stay unbiased. By concentrating effort on the patterns that drive the answer, you reach the same accuracy with fewer total runs than uniform sampling.

6.9   Socratic ladder — choosing which failures to trust

Q·a (foundation). We have three sampling modes. In one line each, what question does each mode answer?
Position sweep → "where is the array most vulnerable?" (which single location hurts most/least). Count sweep → "how many simultaneous failures can we tolerate?" (how damage grows with the number of dead cells). Random sampling → "what is the spread of outcomes?" (the full distribution of power loss / max-T / hot-spot count, not just the average).
Q·b (build up). When is a deterministic position sweep on its own enough — and when is it not?
It is enough when your real risk is a single cell or line failing in isolation, and you want a complete, no-luck-involved map of which single failure is worst. It is exhaustive over single faults and deterministic, so it can never miss one and never needs a seed. It is not enough when failures come in combinations (two, three, many at once) or when you need to know the spread of outcomes — a position sweep asks only single-fault questions and is silent about everything else.
Q·c (why). Once we admit multiple simultaneous failures, why must we turn to random sampling rather than just enumerating them?
Because the number of multi-cell failure combinations explodes: there are 2N patterns in all (≈10180 for 600 cells), far beyond any enumeration. We cannot ask every combination, so we sample random patterns and let their distribution estimate the truth. Random sampling is the only way to say anything about the spread of multi-failure outcomes within a finite compute budget.
Q·d (build up). Having chosen random sampling, how do we know we have run enough patterns?
Track the standard error of the estimated mean, SE = s/√N, and watch the running estimate. Keep adding runs until SE is below the precision you need and the running mean has stopped drifting (it settles into a narrowing band). If you need error SEtarget and a pilot of Nold runs gave SEold, you need N = Nold(SEold/SEtarget)2 runs — e.g. 100 runs at 8 W → 1600 runs for 2 W.
Q·e (deep — the target). Given we can't simulate all 2N failures, how do we choose which failures to simulate so the answer is trustworthy?
We match the sampling to the question, then run enough and pin the randomness. (1) Pick the mode by the question: a position sweep when single-fault vulnerability is the concern (deterministic, O(N), exhaustive over single faults); a count sweep to chart damage against the number of simultaneous failures; random sampling to get the distribution of outcomes. (2) For the random modes, run until the standard error is below your needed precision and the running estimate has stabilised — not a fixed guess, but a convergence you watch. (3) Fix the seed so the chosen patterns are reproducible: the experiment repeats exactly, two layouts are hit with identical patterns (so any difference is the design, not the dice), and any oddity is debuggable. (4) If runs are scarce, use stratified or importance sampling to cover regions deliberately or concentrate effort where damage lives, buying the same accuracy from fewer runs. Trustworthiness comes from choosing the right question-shaped mode, running until the error is provably small, and making the randomness controlled and repeatable — not from blindly throwing random faults and hoping.

6.10   Two traps that fake a result

Pitfall — comparing two layouts with different seeds. If layout A is swept with seed 1 and layout B with seed 2, the two designs face different random failure patterns. Now any difference in their histograms mixes two causes — the design difference you care about and the random-pattern difference you don't — and you cannot tell them apart. The conclusion "B is more robust" might just be "B got an easier set of failures". Always use the same seed for both layouts so they meet the identical patterns and the only variable left is the design.
Pitfall — reporting a single worst case from too few random runs. Run only a handful of random patterns and the worst one you happen to see is dominated by luck — run again with a different seed and the "worst case" changes. A worst case from too few runs is an anecdote, not a result. Either run enough patterns that the worst-case region is well sampled (watch the tail of the histogram settle), or — better for true worst-single-faults — use the deterministic position sweep, which finds the actual worst single location with no luck involved.

7Representing and Storing the Results

7   Representing the results: storing and presenting a Monte-Carlo sweep

Section 6 built the sweep: it picks a failure pattern, runs the coupled electro-thermal solve, records what happened, resets, and repeats — thousands of times. But it ends with an awkward, very practical question, and it is the exact question the colleague raised on the project: where do all those numbers go, and how do we look at them? A finished sweep is not one answer — it is a mountain of answers, one little bundle of numbers for every cell in every run. This section is about the shape of that mountain (how the data is laid out), where it should live (the storage format), and what the engineer actually looks at to make a decision (the figures and tables). The punchline, argued carefully below, is: keep the raw results in a compact binary table as the single source of truth, and treat Excel as a generated human-facing summary on top — never as the place the real data lives.

The one idea. A sweep of R runs over N cells produces a result for every (run, cell) pair — that is R×N rows, e.g. 1000 runs × 600 cells = 600,000 rows. Store that as a tidy/long table (one row per run×cell) in a compressed columnar binary format (Parquet or HDF5) — this is the source of truth. From it, generate the things humans read: a heat-map of where failure hurts most, histograms of the outcome distributions, ranking tables, and an Excel summary workbook. Excel is the export, not the database. The raw tensor is too big and too query-heavy for a spreadsheet, and a spreadsheet is the wrong tool for it.

7.1   The data shape: how big is the mountain?

Start by being honest about the size, because the size is what kills the naive "just put it in Excel" answer. Each run does not produce one number — it produces a small record for each cell: that cell's voltage, current, electrical power, front and rear temperature, and a few flags (was it reverse-biased? a hot-spot? did a §5 criterion trip?). So one run with 600 cells already produces 600 little records. Do that for 1000 runs and you have 1000×600 = 600,000 records. The result of a sweep is a big table, and the first design decision is how to lay that table out.

Two numbers fix the scale. The number of rows is R×N (runs times cells). The width is the number of variables you record per row — here 12 columns (listed in §7.2). So the raw sweep is an (R×N) × 12 table. With 1000 runs and 600 cells that is 600,000 × 12 ≈ 7.2 million individual numbers. That is not "a big spreadsheet" — it is a small database. Treat it like one.

7.2   Tidy / long format — one row per (run, cell)

There are two ways to arrange a per-cell-per-run result, and choosing the right one makes every later step (filtering, grouping, plotting) either trivial or painful.

Tidy / long format. A table arranged so that each row is one observation and each column is one variable. For our sweep an "observation" is one cell in one run, so there is one row per (run, cell). The columns hold the identifying keys (which run, which cell) and the measured variables for that cell. The table is "long" because it is tall and narrow: many rows, few columns.

Concretely, the long table for the sweep has exactly these 12 columns — memorise this list, because the worked example sizes it and the practice asks you to reproduce it:

Wide format (the contrast). A table with one row per run and a separate column for every cell×variable — e.g. V_cell0, V_cell1, …, V_cell599, T_front_cell0, …. With 600 cells and the per-cell variables, that is hundreds of columns per run. The same data, but laid out short-and-fat instead of tall-and-thin.

The two formats carry the same information, so why prefer long? Because long format matches the three operations you do constantly — filter, group, aggregate — and wide format fights them.

  • Filter by cell. "Give me every hot-spot across the whole sweep." In long format that is one condition on one column: hotspot_flag == true. In wide format the hot-spot information is smeared across 600 differently-named columns (hotspot_cell0 … hotspot_cell599), so the same question needs you to name or scan all of them.
  • Group. "For each cell_id, what is its average T_front over all runs?" In long format you group by the single cell_id column. In wide format "cell" is not a column at all — it is encoded in column names — so you cannot group by it without first reshaping the table.
  • Aggregate. "Worst P_elec per run." In long format you group by run_id and take the min. In wide format you would reduce across hundreds of named columns by hand.
Why long wins, in one sentence. In long format the things you want to slice by — which run, which cell — are values in columns, so filtering and grouping are one-liners; in wide format those same things are hidden in column names, so every slice needs a reshape first. Make the thing you query a column, not a name.
"But long format repeats the metadata — isn't that wasteful?" Yes, the run-level keys run_id, mode, failed_ids are identical on all 600 rows of a given run, so long format looks like it stores them 600× over. Hold that thought — §7.3 shows the columnar binary store crushes those repeated values to almost nothing, so the "waste" is only apparent. This is exactly why we do not store long format as raw CSV/Excel, but in a format built for repetition.

7.3   Storage: a columnar binary store as the source of truth

Now the central recommendation. The raw long table lives in a compressed columnar binary format; Excel is only a derived export. Two formats are the standard choices.

Parquet. A compressed, columnar on-disk table format. "Columnar" means it stores each column's values together (rather than row-by-row), which lets it (1) read only the columns you ask for — read just T_front without touching the other 11 — and (2) compress each column hard, because values within a column are similar. The result is a file that is a small fraction of the size of the equivalent CSV or Excel and is fast to read selectively.
HDF5. A hierarchical binary container for large numerical arrays and tables. Think of it as a file system inside one file: you can store many named datasets (arrays, tables) in a tree, with metadata attached, all in an efficient binary layout. Like Parquet it is built for large numerical data and is far more compact and faster than text. Either Parquet or HDF5 is a fine primary store for the sweep.
Columnar encoding is what makes long format cheap. Recall the worry from §7.2 — long format repeats run_id/mode/failed_ids on every cell row. A columnar store keeps each column contiguous, so a column that reads 0,0,0,…(600×),1,1,1,… is stored once with a repeat count (run-length / dictionary encoding), not 600 times. The apparent redundancy of long format collapses to near-zero on disk. So the format that is easiest to query (long) is also, once columnar-compressed, cheap to store — you do not have to trade one for the other.
pandas DataFrame. The in-memory table PowerPy works with. A DataFrame is a labelled 2-D table (named columns, indexed rows) that reads and writes Parquet and HDF5 directly (df.to_parquet(...) / pd.read_parquet(...), and the HDF5 equivalents) and does the groupby/aggregate operations from §7.2 efficiently. The workflow is: sweep → build a DataFrame → write it to Parquet/HDF5 → later, read back (all of it, or just the columns/rows you need) to make figures.

Now the case against Excel as the primary store — three concrete reasons:

A natural layout is one sheet per sweep (the per-run summary metrics — total power, max-T, hot-spot count, breakdown count) plus a ranking sheet (positions/patterns sorted most- to least-damaging). That is dozens to a few thousand rows — exactly what a spreadsheet is for. The key word is generated: the workbook is a derived view that you can throw away and regenerate from the store at any time.

7.4   Aggregations and presentation — what the engineer actually looks at

Nobody reads 600,000 rows. The store exists so that a handful of summaries can be computed from it. Three kinds of figure/table do the real communicating, and the first two come straight from the analyses named in §6.

Heat-map. A 2-D grid in which each cell is coloured by the value of a number — colour stands in for magnitude (e.g. a blue→red scale, red = hot/large). It lets the eye find extremes and patterns in a whole field at a glance, without reading individual numbers.

For the sweep, the most useful heat-map maps failure position → resulting maximum temperature: lay the array out as a grid, and colour each position by the peak temperature that results when that position fails (from the position sweep, §6). The hot-red squares are the positions where a failure is most damaging — the array's weak spots — visible instantly.

  • Histograms of the random-sweep outcomes — one each for total power loss, maximum temperature, and hot-spot count across runs. These are exactly the distributions from §6: a histogram is a bar chart of how often each outcome range occurs, so it shows the whole shape — the average, the spread, and the dangerous tail — rather than collapsing the sweep to a single number.
  • Ranking tables. (1) Most- and least-damaging failure positions/patterns: sort by a damage metric (power loss, peak temperature) so the top of the list is what to protect and the bottom is what barely matters. (2) Per-layout robustness comparison: run the same sweep on two Circuit layouts and tabulate their metrics side by side — the layout with lower power loss / lower peak temperature for the same failures is the more robust design.
Every figure is computed, not stored. The heat-map, the three histograms, and the ranking tables are all groupby/aggregate operations on the long table — group by cell_id for the position heat-map, take per-run maxima for the histograms, sort for the rankings. You store the raw long table once and derive every figure from it; you never store the figures' underlying numbers separately, because then they could drift out of sync with the raw data.

7.5   Reproducibility metadata — store it WITH the results

A results table is only trustworthy if you can regenerate any figure from it months later. That needs more than the numbers — it needs the recipe that produced them. Store, alongside the raw table, the four things that pin the experiment down:

  • the random seed — so the exact failure patterns of the random/count modes can be reproduced bit-for-bit (§6.5);
  • the Circuit parameters — the layout and component values that define which array was swept;
  • the substrate — the thermal backing (materials/geometry) the §3 loop used;
  • the orbit — the illumination/eclipse conditions that set the inputs.
Seed + Circuit + substrate + orbit = a regenerable experiment. With those four stored next to the results (HDF5 makes this natural — attach them as metadata in the same container; with Parquet keep a small sidecar config), any figure can be rebuilt from scratch and any number re-checked. Without them, the table is a pile of numbers no one can defend: "where did this histogram come from?" has no answer. Reproducibility metadata is cheap to store and the whole reason the results can be trusted.

7.6   Worked example — sizing the raw table, and the Excel verdict

Worked example — how big is a sweep, and does it fit in Excel? EXAM-HOT A sweep runs R = 1000 runs over an array of N = 600 cells, recording the 12 columns of §7.2. (a) How many rows? (b) Roughly how much memory, at ~8 bytes per number? (c) Does it fit in a single Excel sheet — and what does that tell us about where to store it?
Step 1 — rows = runs × cells. One row per (run, cell), so
rows = R × N = 1000 × 600 = 600,000 rows.
Step 2 — memory ≈ rows × columns × bytes-per-number. Each row has 12 columns; take ~8 bytes per number (a standard 64-bit float):
bytes ≈ 600,000 × 12 × 8 = 57,600,000 ≈ 58 MB in memory.
Written out as compressed Parquet this shrinks to just a few MB — the columnar encoding (§7.3) squeezes the repeated keys and similar values hard.
Step 3 — compare against Excel's row limit. Excel caps a sheet at 1,048,576 rows (= 220). Our table has 600,000 rows, and 600,000 < 1,048,576, so it does fit — numerically. But that is the wrong test. 600,000 rows is slow to open and filter, and a spreadsheet is not a database (§7.3). And the headroom is thin: push to R = 2000 runs and
rows = 2000 × 600 = 1,200,000 > 1,048,576 ⇒ overflows Excel.
(The crossover is at about 1748 runs for 600 cells.)
Sanity check and conclusion. 1000×600 = 600,000 ✓; 600,000×12×8 = 57.6 M bytes ≈ 58 MB ✓; 2000×600 = 1.2 M > 1.05 M ✓. The honest chain: at 1000 runs the table fits in Excel but Excel is still the wrong store (slow, not a database); double the runs and it does not even fit. Either way the answer is the same — store the raw table in Parquet/HDF5, and make Excel only a summary export. The size argument does not depend on overflow; overflow just makes it unanswerable to pretend otherwise.

7.7   Socratic ladder — why Parquet/HDF5 for raw, Excel for export

Q·a (foundation). What is the difference between long and wide format for a sweep's results?
Long (tidy): one row per (run, cell), one column per variable — tall and narrow; "which run" and "which cell" are values in columns (run_id, cell_id). Wide: one row per run, with a separate column for every cell×variable (hundreds of columns) — short and fat; "which cell" is hidden in column names. Same information, opposite layout.
Q·b (build up). How big does the raw table get, and why is that the heart of the problem?
R×N rows — e.g. 1000 runs × 600 cells = 600,000 rows, ×12 columns ≈ 7.2 million numbers (~58 MB in memory). It is the heart of the problem because the size is what rules out a spreadsheet as the store: this is a small database, not a worksheet.
Q·c (why). What exactly is Excel's row limit, and does the sweep hit it?
Excel allows 1,048,576 rows per sheet (220). A 1000×600 sweep (600,000 rows) fits — but it is close, and a 2000×600 sweep (1,200,000 rows) exceeds it. Even when it fits, 600k rows is slow and a spreadsheet is not built to be queried like a database.
Q·d (build up). What does a columnar binary format (Parquet/HDF5) buy you that CSV/Excel does not?
Three things. (1) Selective reads — pull just the columns you need (e.g. only T_front) without touching the other 11. (2) Hard compression — storing each column contiguously lets repeated keys and similar values compress to a tiny fraction (run-length/dictionary encoding), so long format's repeated run_id/mode costs almost nothing; the file is a few MB vs tens of MB. (3) Scale and speed — effectively unlimited rows, fast reads, built to be a data store. It makes the easy-to-query long format also cheap to store.
Q·e (deep — the target). So why store the raw results in Parquet/HDF5 and treat Excel as only an export?
Because the two have different jobs, and matching each to its job keeps the data correct. The raw results are a large, query-heavy long table (600k+ rows, 12 columns) that you will filter, group and aggregate repeatedly to make heat-maps, histograms and rankings — that is precisely what a columnar binary store is for: unlimited rows, selective fast reads, compression that crushes long format's repeated keys, and a natural home for the reproducibility metadata (seed, Circuit, substrate, orbit). Excel cannot do that job — it caps near a million rows, is slow on that scale, and is not a database. But Excel is the right tool for the human-facing summary: a generated workbook with one sheet per sweep plus a ranking sheet, dozens to a few thousand rows, produced from the store. So the store is the single source of truth (everything derives from it and nothing drifts), and Excel is a disposable, regenerable view on top. Raw in Parquet/HDF5, summary in Excel — the right tool for each, with one authoritative copy of the data.

7.8   Two traps that corrupt the results

Pitfall — editing the Excel summary and treating it as the source of truth. The summary workbook is a derived view, generated from the Parquet/HDF5 store. If you open it, hand-edit a number (fix a typo, tweak a ranking, annotate a cell) and then treat that edited workbook as the real data, you have forked the truth: the store and the spreadsheet now disagree, and the next time anyone regenerates the workbook your edits vanish — or worse, are silently kept and contradict the raw data. Never edit the summary as if it were the data. Fix the inputs or the analysis and regenerate the workbook from the store. The store is authoritative; the workbook is throwaway.
Pitfall — storing wide format, then struggling to filter by cell. Wide format (one row per run, a column per cell×variable) looks tidy at first — one row per run feels natural. But the moment you ask a per-cell question — "every hot-spot in the sweep", "average T_front for cell 317", "worst P_elec per cell" — you discover "cell" is not a column you can filter or group by; it is smeared across hundreds of column names. Every such query then needs a reshape (melt) back to long first. Store long from the start. Make run_id and cell_id real columns so the queries you will actually run are one-liners.

8   Putting it together: the full pipeline, end to end

Seven sections in, we have built seven separate machines: a parametric, fault-addressable Circuit (§1); a vectorised thermal solver (§2); the electro-thermal coupling loop that makes electricity and temperature agree (§3); an orbit-driven environment from hapsira (§4); a pair of breakdown criteria (§5); a Monte-Carlo failure sweep (§6); and a long-format results store with Excel export (§7). Each was taught on its own. This final section does the one thing none of the earlier ones could: it assembles them into a single workflow and shows how one concrete engineering question flows through every module from end to end. The question we will trace is deliberately sharp — "which single-cell failure is most likely to melt the substrate?" — and by the end you will be able to name, for each module, exactly what it contributes to the answer.

The one idea. The seven modules are not seven tools you pick from; they are seven stages of one assembly line. The array shape (§1), the substrate (a small new loader), and the orbit/environment (§4) are built once; then the Monte-Carlo driver (§6) turns a crank thousands of times, and on every turn a single failure pattern is pushed through the same fixed sequence — circuit.fail (§1) → electro-thermal loop (§3, which calls the vectorised thermal solve §2) → breakdown criteria (§5) → a row in the results store (§7) — and at the end the store is aggregated into heat-maps, histograms, rankings and an Excel summary (§7). Understanding the pipeline is understanding where each section lives in that sequence.

8.2   The end-to-end flow, stage by stage

Now the assembly line itself. Read it once as five numbered stages; each stage names the module and section it belongs to. Stages 1–2 happen once; stages 3–4 are the body of the Monte-Carlo loop and repeat once per run; stage 5 happens once at the end.

  1. Build the array (once) — §1. Declare the topology and clone the prototype cell into the whole tree: 
    circuit = Circuit.fromPrototype(cell,
              cells_per_line, lines_parallel,
              modules_per_block, n_blocks)
    This fixes the shape, assigns every cell its hierarchical id (B3.M2.L1.C7), and fills the flat circuit.cells registry that fault injection will reach into.
  2. Load the substrate and define the orbit (once) — §4 (+ substrate.py). Pull the substrate's α, ε, conductivity and thickness from getSubstrateData; define the orbit with hapsira and generate the Environment time series — P_Sun, P_Albedo, P_IR and tilt at every orbit point, including the eclipse where direct sun drops to zero. From this series we can pick the worst orbit point (the harshest combination of fluxes) to stress the array.
  3. For each Monte-Carlo run: plant a failure and solve — §6 → §1 → §3 → §2 → §5. The driver (§6) chooses a failure pattern; then, for that run:
    • plant it — circuit.fail(id, "dead") degrades the chosen cell(s) by id (§1);
    • solve it — at the chosen orbit point(s), run the electro-thermal coupling loop (§3), which repeatedly calls the vectorised thermal solver (§2) until temperature and the operating point agree (the self-consistent fixed point);
    • judge it — apply the breakdown criteria (§5) to every cell's converged state: destroyed if Tfront ≥ Tlimit or (reverse-biased and Prev = |V·I| ≥ Plimit).
  4. Record per cell, then reset and repeat — §7 → §6. Write the per-cell record — Tfront, Trear, signed Pelec, reverse flag, Prev, which criterion tripped — as rows in the long-format store (§7), tagged with the run id and the failure pattern. Un-fail the cell so the next run starts from a pristine array (§6's reset rule), and loop back to stage 3.
  5. Aggregate (once) — §7. When every run is done, aggregate the store: heat-maps of vulnerability by location, histograms of power-loss / max-temperature / hot-spot count, and a ranking of failure positions by damage. Export an Excel summary the engineer can read and act on.
The shape of the whole thing, in one breath. Build once (array §1, substrate, environment §4) → loop many times (fail §1, couple §3 calling thermal §2, judge §5, record §7) → aggregate once (rank, histogram, Excel §7). The "loop many times" body is identical every iteration; only the failure pattern the driver hands in (§6) changes. That is the entire system.

8.4   A worked end-to-end scenario, with the chapter's numbers EXAM-HOT

Let us run the question on a concrete array and read the result off the established numbers. We reuse the chapter's typical shape and the §2/§5 thermal results exactly, so the scenario is fully traceable.

Worked scenario — the most-damaging single-cell failure on a 600-cell array.
Step 1 — build the array (§1). Take the chapter's typical topology: cells_per_line=10, lines_parallel=1, modules_per_block=3, n_blocks=20. Total cells = 10×1×3×20 = 600. Series depth (what sets the bus voltage) = cells_per_line × modules_per_block = 10×3 = 30 cells ⇒ Vbus = 30 × 2.35 V ≈ 70.5 V — the project's ~70 V bus. Note lines_parallel = 1: each line is the only current path in its module — there is no line-level redundancy. Hold that fact; it is the whole story.
Step 2 — conditions (§4). Load the substrate (α, ε, conductivity, thickness via getSubstrateData) and build the orbit's Environment with hapsira. From the time series, pick the worst orbit point — the harshest sun/tilt combination — and run the sweep there, where failures bite hardest.
Step 3 — choose mode and sweep (§6). The question is single-fault vulnerability, so use the position sweep: fail one cell at a time across all 600 locations — 600 deterministic runs, no seed needed.
Step 4 — the decisive run (§1→§3→§2→§5). Consider the run that kills the cell at a particular series position. circuit.fail(id,"dead") sets that cell's photocurrent to zero (§1). But a dead cell in a series line does not simply vanish: the other cells in its line still try to push the string current through it, forcing it into reverse bias. With no effective bypass, it is driven to its reverse-breakdown voltage V = −20 V while still carrying the string current I = 0.48 A. The electro-thermal loop (§3), calling the vectorised solver (§2), converges to:
Prev = |V·I| = |(−20 V)(0.48 A)| = 9.6 W  →  Tfront ≈ 187 °C
Now apply the §5 criteria: Prev = 9.6 W ≥ Plimit = 2 W ⇒ FAIL on reverse-power; Tfront = 187 °C ≥ Tlimit = 150 °C ⇒ FAIL on temperature. Both criteria trip — this position is flagged destroyed and, being among the hottest/most-stressed, ranks most-damaging.
Step 5 — the rest of the array, and the power (§5→§7). The healthy cells, meanwhile, sit near ~39 °C (the §2 healthy-cell figure, 38.89 °C) — cool, generating, well clear of both limits. But the failed cell's whole line has stopped generating (its current path is broken), so the array's total power drops by that lost string's share — the contribution of one line out of the array. Every per-cell number goes into the long-format store (§7).
What the engineer learns. The ranking (§7) names the series position whose single-cell death drives 9.6 W / 187 °C and trips both criteria as the array's weak spot. Two fixes follow directly: (1) add a bypass diode at that position — from §5, clamping −20 V to −0.7 V collapses 9.6 W to 0.34 W and the cell settles cool, a destroyed-to-fine flip; or (2) change the layout — raise lines_parallel from 1 to 2, the redundant module §6 compared, so a dead cell's line is no longer the only path and the others carry on. The pipeline did not just say "it gets hot"; it said where, how hot, why, and what to do. ✓
Why lines_parallel = 1 was the whole story. With a single line per module, one dead cell breaks the only path and forces itself into −20 V reverse breakdown — the 9.6 W / 187 °C catastrophe. The §1 topology knob, the §5 criteria, and the §6 layout comparison all meet at this one cell: the array is fragile because of a wiring choice, and the fix is a different wiring choice. That is the chapter's argument in a single cell.

8.5   Honest status: what is proven, what is pending

A study-notes chapter should be candid about which parts already stand and which are still a plan. Here is the honest accounting.

Proven now. The vectorised thermal solver (§2, thermal_vectorized.py) is pure numpy — it depends on no external simulator — and it is verified against a hand-worked single-cell energy balance: the §2 worked example (65.26 °C idle, 38.89 °C extracting, 187.48 °C dissipating) was reproduced by hand, step by Newton step, with the residual collapsing to ~10−11. Where the solver overlaps a closed-form balance, it agrees. That stage is trustworthy today.
Pending validation. The circuit / ngspice path (§1) and the Monte-Carlo machinery (§6) depend on the existing Chapter-1 cell model and on ngspice actually running and reporting correctly. Those are validated in two steps, smallest first:
  • Tiny 2×2 circuit first. Build a 4-cell Circuit (cells_per_line=2, lines_parallel=2 → one tiny block), run it through ngspice, and read back each cell's V and I to confirm the per-cell operating point is extracted correctly and matches a hand calculation. This is the smallest test that exercises the netlist build, the run, and the read-back together.
  • Then end-to-end. Only once the 2×2 read-back is trusted do we wire the full pipeline (§1–§7) and run a real sweep, checking that healthy cells land near ~39 °C and a forced reverse cell reproduces the ~187 °C / 9.6 W signature from §2/§5.
The main implementation risk — per-cell V,I extraction from ngspice. Everything downstream rides on one fragile step: pulling each cell's converged voltage and current back out of the ngspice result, correctly, by id. The §3 loop needs that per-cell P_elec vector to compute heat; §5 needs each cell's V (and its sign) to apply the reverse-power criterion. If the read-back maps a current to the wrong cell, or drops a sign, then one circuit solve quietly corrupts the whole per-cell power vector — and every temperature, every breakdown flag, every ranking built on it is wrong, silently. This is exactly why the 2×2 read-back test exists: prove the extraction on four cells you can check by hand before trusting it on six hundred.

8.8   Two pipeline-level pitfalls

Pitfall 1 — sweeping at a benign orbit point and declaring victory. The §4 environment is a time series: the fluxes and tilt rise and fall around the orbit, so the same failure is far more dangerous at some orbit points than others. If you run the whole sweep at a benign point — full sun, easy tilt, nothing stressed — even the worst single-cell failure may stay under the limits, and you will conclude "the array is safe." It is not; you simply never tested it when it was hot. Always identify the worst-case orbit point (or sweep across the harsh part of the orbit) before drawing any safety conclusion. A clean sweep at the easiest moment of the orbit proves nothing about the hardest.
Pitfall 2 — getting the per-cell read-back wrong poisons everything downstream. One circuit solve produces the entire per-cell P_elec vector that the §3 loop feeds to the §2 thermal solve, and the per-cell V that §5's reverse-power criterion gates on. Get that extraction wrong — a current attributed to the wrong id, a dropped sign on V — and the error does not stay local: every temperature, every breakdown flag, and every ranking computed from that vector is wrong, and wrong silently (the numbers still look plausible). This is the same risk flagged in §8.5; it is why the 2×2 read-back test must pass before the pipeline is trusted. The pipeline is only as correct as its weakest hand-off, and this is the weakest.

Q1-1. A line with cells_per_line = 4, cells of Vmp = 2.35 V and Imp = 0.48 A; then a module of lines_parallel = 2 such lines.

(a) Series or parallel? A line is a series group — cells_per_line cells wired end-to-end. (Series is the line's default combine; its job is to build voltage.)

(b) The line's max-power voltage and current. In series, voltages add and the current is common to all cells in the chain.
Vmp,line = 4 × 2.35 V = 9.4 V.   Imp,line = 0.48 A (unchanged — putting cells in series does not increase current; the same current flows through every cell in the chain).

(c) The module (2 lines in parallel). A module is a parallel group, so its children (the lines) all sit between the same two rails: voltage stays that of one line and currents add.
Vmp,module = 9.4 V (same as one line).   Imp,module = 2 × 0.48 A = 0.96 A.

(d) What adds where. Voltage adds in series; current adds in parallel. (And the partner quantity is the one held common: current is common down a series chain, voltage is common across a parallel rail pair.)
Sanity check: power tracks correctly either way — one line is 9.4 V × 0.48 A = 4.512 W, two such lines in parallel is 9.4 V × 0.96 A = 9.024 W = exactly twice one line, as two identical lines must give. ✓

Section 2 — Vectorising the Thermal Solve

Q2-1. Single cell, same substrate/constants as worked example 2.1 (CA = 3 W/K, εFAσ = 1.531×10−10, εRAσ = 1.514×10−10, A = 0.003 m²), but shaded: front absorbed flux αF·A·PSun·tilt = 2.000 W, Pelec = 0, rear fluxes = 0, start T1 = T2 = 300 K.

(a) Front residual, and its value at 300 K. With the rear fluxes zero and at the start T1 = T2 (so the conduction term CA(T2−T1) = 0), the front residual is
f1 = αFAPSun·tilt − εFAσ(T14 − Tsp4) − Pelec + CA(T2 − T1).
Evaluate at 300 K (the Tsp4=2.7⁴ piece is negligible):
f1 = 2.000 − (1.531×10−10)(300)4 = 2.000 − 1.240 = +0.760 W.
(Net heat in is only 0.76 W now — much less than the 2.74 W of the fully-lit cell, because the cell absorbs less but still radiates the same 1.24 W at 300 K.)

(b) Rear residual at 300 K. The rear face absorbs nothing (PAlbedo = PIR = 0) and the conduction term is again zero at the start, so
f2 = 0 − (1.514×10−10)(300)4 = 0 − 1.226 = −1.2263 W
— identical to worked example 2.1, since the rear side is unchanged by shading the front.

(c) One Newton step. The Jacobian does not depend on the fluxes, only on the temperatures, so at 300 K it is exactly the one from example 2.1:
a = −3.0165,  b = c = 3.0,  d = −3.0164,  and det = ad−bc = 9.0989−9 = 0.0989.
Apply the closed-form update with the new residuals (f1 = 0.760, f2 = −1.2263):
ΔT1 = −(df1 − bf2)/det = −[(−3.0164)(0.760) − (3.0)(−1.2263)]/0.0989 = −[−2.292 + 3.679]/0.0989 = −(1.387)/0.0989 = −14.02 K.
ΔT2 = −(−cf1 + af2)/det = −[(−3.0)(0.760) + (−3.0165)(−1.2263)]/0.0989 = −[−2.280 + 3.699]/0.0989 = −(1.420)/0.0989 = −14.35 K.
New temperatures after one step:
T1 = 300 − 14.02 = 285.98 K = 12.83 °C;   T2 = 300 − 14.35 = 285.65 K = 12.50 °C.
Sanity check: the corrections are negative this time — the cell must cool from 300 K, because at 300 K it is already radiating more (1.24 + 1.23 = 2.47 W total) than the 2.000 W it now absorbs, so net it is losing heat. That is the opposite sign to the fully-lit example 2.1 (where the cell warmed), and exactly what shading should do. Iterating to convergence lands the cell near 13 °C front / 10 °C rear — a cold, lightly-lit cell, as expected.
Q2-2. Why one vectorised Newton beats N fsolve calls. Calling scipy.fsolve once per cell inside a Python for loop pays two avoidable costs on every cell: (1) Python interpreter loop overhead — each iteration of a Python loop is far slower than the same arithmetic in compiled code, and you pay it N × (Monte-Carlo trials) times; and (2) a full general-solver spin-up per cell — fsolve is a black box that does not know our equations, so each call re-initialises, probes the function, and typically estimates derivatives numerically, repeating that setup work from scratch for every cell even though all cells obey the same two equations. The vectorised Newton avoids both: it exploits the known analytic Jacobian and closed-form 2×2 inverse, and it updates all N cells with a handful of numpy array operations that run in compiled C — so the only remaining loop is over ~5–8 iterations, not over cells. (It is also more solver work that is shared rather than duplicated: one residual/Jacobian evaluation covers the whole array.)
Q2-3. What lateral conduction does to the Jacobian.
(i) Structure. If each cell also conducts heat sideways to its nearest neighbours, then each cell's residuals now depend on its neighbours' temperatures, so partial derivatives linking one cell to another become non-zero. The global Jacobian is therefore no longer block-diagonal: the off-block positions that were all zero now carry the lateral-conduction coupling terms. With nearest-neighbour coupling the non-zeros sit just outside the diagonal blocks, so the matrix becomes banded (a sparse, coupled structure) rather than a clean set of independent 2×2 blocks.

(ii) Validity of the elementwise update. The simple closed-form 2×2-inverse update is no longer valid, because it assumes each cell's 2×2 block can be inverted in isolation — which is only true when the blocks are uncoupled. Once the blocks are coupled you cannot solve each cell separately; you must solve the whole linear system JΔx = −f together at each Newton step. In practice you would use a banded or sparse linear solver (e.g. a banded LU factorisation, or scipy.sparse.linalg) that exploits the banded structure so it is still efficient, rather than the per-cell elementwise inverse. In short: drop the "no sideways heat flow" assumption and you lose the vectorisation trick, trading it for a (still tractable) banded solve.

Section 3 — The Electro-Thermal Coupling Loop

Q3-1. Damped update from T = 40 °C toward T_new = 150 °C.
The disagreement is the same for both cases: T_new − T = 150 − 40 = 110 °C.
With ω = 0.5. T ← T + ω·(T_new − T) = 40 + 0.5·110 = 40 + 55 = 95 °C.
With ω = 0.25. T ← 40 + 0.25·110 = 40 + 27.5 = 67.5 °C.
Which moves further, and the trade-off. ω = 0.5 moves the cell further this pass (to 95 °C vs 67.5 °C), because a larger ω accepts more of the computed change. The trade-off: the larger ω converges in fewer passes but risks overshoot/oscillation on a strongly heating fault; the smaller ω is more cautious and stable but needs more passes (more ngspice solves) to crawl to the same fixed point. Either way the destination is unchanged — at the fixed point T_new = T, the correction is zero for any ω; ω only sets the path.
Q3-2. Why a negative P_elec raises temperature.
P_elec = V·I is the electrical power flowing at the cell's terminals. For a healthy cell V and I have the generator sign convention and the product is positive — the cell exports power to the rest of the circuit. A reverse-biased cell is being driven backwards by its neighbours, so its voltage and current take opposite signs and the product goes negative (≈ −9.6 W): the cell is no longer a source but a sink, absorbing ~9.6 W of electrical power from the string. That absorbed power does not vanish — by energy conservation it is dissipated inside the cell as heat. So a negative P_elec feeds the thermal solve as a heat input, which drives the cell's temperature up (in the worked example, all the way to ~187 °C). Positive P_elec = power leaving = cooler; negative P_elec = power absorbed as heat = hotter.
Q3-3. Detecting non-convergence, and how to fix it.
How to detect it. Record the residual r_k = max|T_new − T| at every pass k and watch the sequence rather than a single value. A healthy run shows r_k falling monotonically toward zero (and tripping r_k < tol within a handful of passes). Non-convergence shows up as a residual that does not shrink: it oscillates (alternating up/down, often roughly constant in size — the ping-pong of §3.4), grows without bound (divergence), or plateaus above tol and stops improving. The practical guards: cap the loop with a maximum-iteration count and flag failure if r_k has not crossed tol by then; and flag oscillation/divergence if r_k stops decreasing (e.g. r_k ≥ r_{k−1} for several consecutive passes).
Two fixes. (1) Reduce ω — apply more under-relaxation (e.g. drop ω from 1.0 or 0.5 toward 0.2), feeding back a smaller fraction of each disagreement so the loop stops overshooting; this is the direct cure for oscillation from a too-large step on a strong fault (§3.4, and the ω pitfall). (2) Improve the starting guess and/or loosen an over-tight tolerance — a better initial T (e.g. seed faulted cells warm rather than at 28 °C) shortens the path, and a tol set absurdly tight relative to the solvers' own accuracy can make a converged loop look stuck, so right-size it. (A third, if it still fights you: re-examine step 3 — a sign error or mis-parse in the per-cell V/I read-back from ngspice (§3.6) can manufacture a fake instability, so validate the read-back on a 2×2 before blaming the loop.)

Section 4 — Orbit-Driven Fluxes with hapsira

Q4-1. Solar irradiance at Venus (d = 0.72 AU). Apply the inverse-square scaling anchored at AM0 = 1367 W/m² at 1 AU.
Square the distance: d² = 0.72² = 0.5184 (AU², a pure ratio).
Scale: E = 1367 / 0.5184 ≈ 2637 W/m².
Comparison: this is larger than Earth's 1367 W/m² — about 2637 / 1367 ≈ 1.93×, i.e. nearly double. That is expected: Venus is closer to the Sun than Earth (0.72 AU < 1 AU), and by the inverse-square law a smaller distance means a larger irradiance. (Units: W/m² ÷ dimensionless ratio = W/m². ✓)
Q4-2. Albedo flux over Earth. Use P_Albedo = A · E · F.
Substitute: P_Albedo = 0.30 × 1367 × 0.20.
Step the arithmetic: 0.30 × 1367 = 410.1 W/m²; then 410.1 × 0.20 = 82.0 W/m².
Answer: P_Albedo ≈ 82 W/m².

Q4-3. Array temperature entering and leaving eclipse.
Entering eclipse: the planet moves between the satellite and the Sun, so P_Sun drops to 0 and the electrical power output falls to zero — the array makes no electricity in shadow. With the dominant heat input (direct sun) removed, the array cools (radiates heat to space faster than it gains it). It does not cool to absolute zero, because the planet's P_IR keeps glowing onto it from below (the planet stays warm on its night side), which slows the cooling — but the trend is a steady cold-soak through the eclipse.
Leaving eclipse (eclipse exit): the satellite emerges into sunlight, P_Sun jumps back to its full value, electrical power is restored, and the now-cold array is suddenly illuminated and loaded again. P_IR has been present throughout and changes only with view factor.
Worst-case hot-spot: the moment just after eclipse exit. The array has been cold-soaked in shadow and is then hit with full sun and asked to deliver power at once — the abrupt swing in flux and load makes this the critical (and inherently transient) case, which a single fixed-sun model could never reveal. This is exactly the forward hook to a transient study.

Section 5 — Breakdown and Melt Criteria

Q5-1. Limits: Plimit = 2 W, Tlimit = 150 °C. The reverse-power test applies only when V < 0.
(i) V = −0.7 V, I = 0.50 A, Tfront = 52 °C. Reverse-biased (V < 0), so Prev = |(−0.7 V)(0.50 A)| = 0.35 W. Reverse-power: 0.35 W < 2 W ⇒ PASS. Temperature: 52 °C < 150 °C ⇒ PASS. Overall: not destroyed (this is a healthy diode-clamped reverse cell).
(ii) V = +2.3 V, I = 0.46 A, Tfront = 61 °C. V > 0 — the cell is forward, generating power, so the reverse-power test does not apply: Prev is recorded as 0 W and the criterion is PASS by the sign gate (the +1.06 W it delivers is useful output, not a fault). Temperature: 61 °C < 150 °C ⇒ PASS. Overall: not destroyed. (Trap: had you ignored the sign gate and tested |V·I| = 1.06 W you would still get PASS here, but the reasoning must be "forward cell, test skipped," not "1.06 < 2.")
(iii) V = −15 V, I = 0.40 A, Tfront = 140 °C. Reverse-biased, so Prev = |(−15 V)(0.40 A)| = 6.0 W. Reverse-power: 6.0 W ≥ 2 W ⇒ FAIL. Temperature: 140 °C < 150 °C ⇒ PASS. Overall: destroyed — and it is the reverse-power criterion that condemns it. This is a Case-B-style cell: its bulk temperature (140 °C) sits just under the limit and would have cleared the temperature test alone, but the 6 W reverse dose trips the OR. Only cells (i) and (ii) survive; (iii) is flagged, caught solely by reverse-power.
Q5-2. Why a bypass diode usually keeps Prev under Plimit.
A bypass diode wired across the cell turns on as soon as the cell is driven into reverse and carries the string current around it, clamping the cell's reverse voltage to about the diode's forward drop, ~−0.7 V. The reverse dissipation is then only Prev = |V·I| ≈ 0.7 V × I — for a string current of, say, 0.48 A that is about 0.34 W, comfortably below a typical Plimit of order a couple of watts. Without the diode there is nothing to clamp the voltage, so the cell is pushed all the way to its reverse-breakdown voltage (~−20 V) before it conducts, giving Prev = 20 V × I ≈ 9.6 W — nearly 30× larger and far over Plimit. The diode wins simply by holding the voltage small: the current is essentially the same either way, so the ~30× voltage ratio is the whole difference between a third of a watt and ten watts.
Q5-3. Why Tlimit should be the structural limit (~150–200 °C), not the melting point (~660 °C).
The melting point is where the aluminium honeycomb core turns to liquid, but the panel is destroyed long before that. The relevant failure mechanism below the melting point is the loss of the bonded assembly's integrity: the aluminium softens and creeps (losing mechanical strength) and, decisively, the adhesive bond holding each cell to the substrate degrades and lets go, somewhere around 150–200 °C. Once the bond fails or the core slumps, the panel has failed as a structure even though no metal has melted. For a go/no-go design check, "conservative" means choosing the limit that flags destruction as early as the real failure occurs — i.e. the lower structural/bond limit — so the check never declares a structurally-failed panel "healthy." Using 660 °C would do exactly that wrong thing: it would pass a 400 °C panel that has, in reality, already come unbonded. The exact value is not a universal constant; it is a project input set by the specific aluminium, adhesive and bond construction used in the assembly, supplied by the materials/structures engineers.
Q5-4. Own worked numbers for the two disagreement cases (any internally-consistent answer is acceptable; one valid set below). Take limits Plimit = 3 W, Tlimit = 160 °C.
(a) FAILS on temperature, PASSES on reverse-power (poorly-radiating cell). Let V = −4 V, I = 0.50 A ⇒ Prev = |(−4 V)(0.50 A)| = 2.0 W. Reverse-power: 2.0 W < 3 W ⇒ PASS. Now suppose this cell sits in a low-emissivity patch hemmed in by hot neighbours, so even 2 W cannot escape and its converged Tfront = 175 °C. Temperature: 175 °C ≥ 160 °C ⇒ FAIL. Overall destroyed, caught only by temperature. Physical situation: modest electrical dose, but the heat has nowhere to go, so the steady temperature runs over the limit.
(b) FAILS on reverse-power, PASSES on temperature (intensely localised fault). Let V = −18 V, I = 0.45 A ⇒ Prev = |(−18 V)(0.45 A)| = 8.1 W. Reverse-power: 8.1 W ≥ 3 W ⇒ FAIL. But the energy goes into a pinpoint avalanche site of a few mm², so the cell-averaged converged temperature the lumped model reports is only Tfront = 130 °C. Temperature: 130 °C < 160 °C ⇒ PASS. Overall destroyed, caught only by reverse-power. Physical situation: high electrical dose concentrated in a tiny region ruins the cell while the area-smeared bulk temperature still looks survivable.
Both cases confirm the OR rule: each is a genuine destruction that a single criterion would have cleared.

Section 6 — The Monte-Carlo Failure Sweep

Q6-1. Runs needed to tighten the standard error from 6 °C to 1.5 °C.
(a) Known values (with units). Nold = 50 runs; SEold = 6 °C; SEtarget = 1.5 °C.
Since SE = s/√N with s fixed, runs scale as the inverse square of the error:
N = Nold · (SEold / SEtarget)2 = 50 · (6 °C / 1.5 °C)2.
6 / 1.5 = 4; 42 = 16; 50 · 16 = 800 runs in total.
(b) Rule of thumb. Error falls like 1/√N, so to cut the error by a factor f you need f2 times as many runs — here f = 4, so 42 = 16× the runs. (Check: the °C units cancel inside the ratio, leaving a pure run count, as it must.)
Q6-2. Matching each question to its sampling mode.
(i) "Which single cell, if it dies, costs us the most power?" → POSITION sweep. It asks about a single failure and wants the worst location; the position sweep fails one cell at a time over every location (deterministic, O(N)) and ranks them, finding the worst single cell with no luck involved.
(ii) "If cells fail with probability 2% each, what is the distribution of array power we should expect?" → RANDOM sampling. The question explicitly asks for a distribution under a per-cell failure probability; draw many random patterns (each cell dead w.p. 2%) and histogram the resulting array power to get its spread.
(iii) "How many simultaneous dead cells before peak temperature crosses the breakdown limit?" → COUNT sweep. It asks how damage grows with the number of simultaneous failures; fail k random cells for k=1,2,3,… and find the k at which peak temperature crosses the §5 limit.
Q6-3. Position sweep on an 800-cell array, 0.4 s per run.
(a) Runs and 1-core wall-clock. A position sweep fails one cell at a time over every location → one run per cell → 800 runs. On a single core the runs are sequential: 800 runs × 0.4 s/run = 320 s (≈ 5 min 20 s).
(b) 4-core wall-clock and why it is slightly longer in practice. The runs are independent — each fails its cell, solves, records and resets, sharing no state with any other run — so the sweep is embarrassingly parallel and 4 cores share the work: ideal time = 320 s / 4 = 80 s. The real time is a little longer than 80 s because parallelism carries small overheads — spawning the worker processes, distributing the runs, and gathering the results back (e.g. via joblib) — so the speed-up is near-linear, not perfectly linear. (Check: 800 × 0.4 = 320 ✓; 320 / 4 = 80 ✓.)

Section 7 — Representing and Storing the Results

Q7-1. Sizing the raw long table for R = 2500 runs over N = 600 cells, 12 columns.
(a) Rows = runs × cells. One row per (run, cell):
rows = R × N = 2500 × 600 = 1,500,000 rows.
(b) Memory ≈ rows × columns × bytes-per-number, at 12 columns and ~8 bytes per 64-bit number:

Q7-2. Choosing the right artifact — Parquet/HDF5 store vs Excel summary workbook.
(i) A printable one-page table of the ten most-damaging failure positions for a report. → Excel summary workbook. It is a tiny (ten-row), human-facing, printable view — exactly the ranking-sheet job Excel is for. Generate it from the store; do not treat the spreadsheet as the data.
(ii) Recompute the max-temperature histogram after changing the hot-spot threshold, over all 700,000 rows. → Parquet/HDF5 store. (The result could then be exported to an Excel sheet for a colleague to read, but the filtering happens against the store.)
Q7-3. The 12 columns of the long (tidy) results table, and the rows one run contributes.
The 12 columns (one row per (run, cell)):
  1. run_id — which run the row belongs to (key, run).
  2. mode — sampling mode that produced the run: position / count / random (run-level key).
  3. failed_ids — which cells were failed in this run, i.e. the failure pattern (run-level key).
  4. cell_id — which cell this row describes, 0…N−1 (key, cell).
  5. V — cell voltage, V (measured).
  6. I — cell current, A (measured).
  7. P_elec — cell electrical power, W; negative when absorbing in reverse (measured).
  8. T_front — converged front-face temperature, °C (measured).
  9. T_rear — converged rear-face temperature, °C (measured).
  10. reverse_flag — was this cell reverse-biased? true/false (flag).
  11. hotspot_flag — did this cell exceed the hot-spot threshold? true/false (flag).
  12. criterion_tripped — which §5 breakdown criterion tripped here, if any (flag).
Rows from one run. A single run records one row per cell, so a run on an N-cell array contributes N rows (e.g. 600 rows for a 600-cell array). That is why the whole sweep has R×N rows: R runs, each adding N rows — one observation per (run, cell). Within those N rows the run-level columns (run_id, mode, failed_ids) repeat identically, which is the redundancy that columnar compression (Parquet/HDF5) crushes to almost nothing.

Section 8 — The Full Pipeline, End to End

Q8-1. Ordering the pipeline for "how many simultaneous random failures can we tolerate before peak temperature crosses the limit?"
Correct execution order (with the owning module/§):
1. C — build the array from the prototype: Circuit.fromPrototype(...) (circuit.py, §1). [once]
2. F — load the substrate and build the orbit environment: getSubstrateData + hapsira Environment (substrate.py / environment_orbit.py, §4). [once]
3. E — choose the failure pattern for this run: the §6 driver picks k random cells (montecarlo.py, §6). [loop body]
4. B — run the electro-thermal coupling loop at the worst orbit point: electrothermal.py (§3) calling the vectorised thermal solve thermal_vectorized.py (§2). [loop body]
5. D — apply the breakdown criteria per cell: temperature OR reverse-power (§5, in montecarlo.py). [loop body]
6. G — record the per-cell results to the long-format store and reset the circuit (results.py §7 + §6's reset). [loop body; then go back to step 3 for the next run]
7. A — aggregate, rank and plot once all runs are done (results.py, §7). [once]
So the order is C → F → E → B → D → G → A, with E–G repeating once per run.

The question asks how damage grows with the number of simultaneous failures, so you fail k random cells for k=1,2,3,… (several random draws per k) and find the smallest k at which peak temperature crosses Tlimit. (Position mode is wrong — it only ever fails one cell; pure random-probability mode answers "spread of outcomes", not "how many we tolerate".)
