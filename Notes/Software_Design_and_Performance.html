Module & package A module is one .py file; a package is a folder of modules with an __init__.py. PowerPy is a package.

1What Software Design Is, and the Principles That Matter

You can already write code that runs. This part of the notes is about something the running code never tells you: whether the program is built well — whether, six months from now, you (or a teammate) can read one piece of it without reading all of it, and change that piece without breaking everything around it. That is what software design is about, and it is a skill quite separate from knowing a language. PowerPy is our worked specimen throughout: a Python tool that simulates a satellite solar array. We will see where its modern layout is designed well, where its older "legacy" files are designed badly, and exactly which principles tell the two apart. Every term and every acronym is defined the first time it appears.

D1.1 What "software design" actually means

Most beginners think of programming as a single activity: you sit down and type instructions until the computer does the thing. That works for a twenty-line script. It falls apart the moment a program grows to thousands of lines that many people must understand and edit over years. At that scale a second activity becomes just as important as typing — and it has a name.

Term Software design — deciding how to split a program into parts, and how those parts talk to each other, both before you write code and while you write it, so that the program stays understandable and changeable as it grows. It is not about making the code run (the computer does not care how you organise it); it is about making the code survivable for the humans who must live with it.

Read that definition again, because two phrases in it do all the work. "Split into parts" — a big program is not one undifferentiated blob; it is a set of pieces, and you get to choose where the cuts go. "How those parts talk" — once cut apart, the pieces must still cooperate, so each piece exposes some way for the others to use it. Good design is mostly about where you put the cuts and how thin you can make the connections across them.

Note The computer is indifferent to design. A 2000-line file with everything jammed together and a tidy set of small modules can compile to the exact same behaviour and run at the exact same speed. Design is purely for the benefit of people — the reader, the tester, the person fixing a bug at 2 a.m. If you ever catch yourself thinking "the machine doesn't care, so why bother," remember: you are not organising the code for the machine. You are organising it for the next human, who is often you.
Key idea Code that works and code that is well designed are two different achievements. Tests and the interpreter check the first. Only design discipline checks the second — and the second is what decides whether the program can still be changed cheaply a year from now. A program is read and modified far more often than it is written; design optimises for that long tail.

D1.2 The principles that matter

"Good design" sounds vague until you break it into named principles you can point at. The rest of this section is a tour of the handful that matter most. Each one gets a plain definition and a concrete PowerPy example, because a principle you cannot apply to real code is just a slogan.

Separation of concerns

Term Separation of concerns — the principle that each part of a program should be responsible for one kind of job, and different kinds of job should live in different parts. A "concern" is simply one area of responsibility: reading input is one concern, doing the physics is another, producing a report is a third. Keep them apart and each can be understood and changed on its own.

PowerPy's modern layout does this genuinely well. Instead of one giant file, it is organised into subpackages, each owning exactly one concern. (A subpackage is just a sub-folder of code that groups related modules together.)

  • schemas/ — defines the shapes of the data (what a cell record looks like, what fields a substrate has). Concern: what the data is.
  • loader/ — reads those data shapes from files on disk. Concern: getting data in.
  • simulation/ — runs the actual physics (the electrical and thermal model). Concern: computing the answer.
  • render/ — turns results into output (plots, reports). Concern: showing the answer.

Notice how cleanly the responsibilities divide. If the file format on disk changes, you edit loader/ and nothing else. If you improve the physics, you edit simulation/ and nothing else. If marketing wants prettier charts, you edit render/ and nothing else. Each concern has its own home, and a change to one rarely disturbs the others.

Cohesion and coupling

Separation of concerns has two measurable sides, and they are the two words you will hear most often when engineers judge a design. They are easy to mix up, so define them carefully — they pull in opposite directions.

Term Cohesion — how focused a single module is: how strongly the things inside it belong together and serve one purpose. A module where every function works toward the same job has high cohesion (good). A module that is a junk drawer of unrelated functions has low cohesion (bad). Cohesion is about what is inside one box.
Term Coupling — how much one module depends on another: how much it must know about another module's internals, and how badly it breaks when that other module changes. Modules that lean on each other heavily are tightly coupled (bad); modules that interact only through a small, stable connection are loosely coupled (good). Coupling is about the wires between boxes.
Key idea — the design rule in five words KEY High cohesion, low coupling. Make each part tightly focused on one job (high cohesion) and make the parts depend on each other as little as possible (low coupling). Almost every other principle in this section is a specific way of moving a design toward those two goals at once. When you are unsure whether a change helps the design, ask: does it raise cohesion or lower coupling? If neither, it probably is not helping.
Note The two are not the same thing, and that is the usual confusion. Cohesion looks inward at one module ("is this box about one thing?"). Coupling looks outward at the connections ("how tangled are the wires between boxes?"). You can have a module that is beautifully focused inside (high cohesion) yet wired into ten others (high coupling) — and that module is still painful to change, because every edit ripples outward. You want both fixed: each box about one thing, and thin wires between boxes.

Single Responsibility

Term Single Responsibility — the principle that a module, class, or function should have one reason to change. Phrase it as a test: "what kinds of future change would force me to edit this piece?" If the honest answer lists several unrelated things, the piece is doing too much and should be split. One responsibility means one reason to change.

This is exactly where PowerPy's legacy code falls down — and it is worth seeing the contrast, because it makes the principle concrete. The older design has a few big files — cell.py, string.py, section.py, panel.py — and inside each, one large class tries to do everything at once:

  • loads its own data from files,
  • does the parameter maths (the equations that turn raw numbers into cell behaviour),
  • generates a netlist (a netlist is a text description of an electrical circuit — which components connect to which — fed to a circuit simulator),
  • runs the simulation, and
  • plots the results.

Count the reasons that one class could need changing: the file format changes; a physics equation is corrected; the netlist syntax is updated for a new simulator; the simulation algorithm is improved; the plotting library is swapped. That is five different reasons to change, all colliding inside one class. The class violates Single Responsibility five times over. The practical pain is real: the file is long and hard to read, you cannot test the maths without also dragging in file-loading and plotting, and a careless edit to one job can silently break another.

Note Spot the relationship between the principles. The legacy monolith has low cohesion (one class mixing five unrelated jobs) and it forces high coupling (anything wanting the maths must also depend on the file-loading and the plotting bundled with it). The modern schemas/ loader/ simulation/ render/ split is the same code reorganised for high cohesion, low coupling. Single Responsibility is, in effect, "high cohesion" applied at the level of one class or function.

Interfaces and encapsulation

Once you split a program into parts, the parts must cooperate without each one needing to know how the others are built inside. The mechanism for that is the interface, and the discipline behind it is encapsulation.

Term Interface — the small, stable set of named operations a part exposes to the outside world: the public "buttons" you are allowed to press on it. Callers use only the interface; they never reach inside. A good interface is small (few buttons) and stable (the buttons don't keep changing), so the code that depends on it stays simple and rarely needs updating.
Term Encapsulation — the practice of hiding a part's internal details behind its interface, so that the inside can be changed freely without disturbing any caller. The part keeps its data and its private workings to itself and exposes only the public operations. Encapsulation is what makes the interface a real boundary rather than a polite suggestion: outsiders cannot depend on the internals because they cannot see them.

A PowerPy example: imagine the substrate (the physical backing a panel is mounted on) exposes one public method, say surface(), that returns the area available for cells. Callers ask substrate.surface() and get a number. How the substrate computes that — from stored dimensions, from a lookup table, from a formula — is hidden inside. Tomorrow you can rewrite the internal calculation completely, and as long as surface() still returns the right number, not a single caller notices or breaks. That is encapsulation paying off: the inside changed, the outside did not care.

Key idea — the test of a good boundary A boundary (a module, class, or interface) is well placed if you can answer yes to both: (1) Can you understand the part from its interface alone, without reading its internals? (2) Can you change the internals without breaking the parts that use it? If either answer is no — you must read the guts to use it, or an internal tweak breaks its users — then the boundary is in the wrong place, and the part is leaking. Use this two-question test every time you draw a line in your code.

DRY and YAGNI

Term DRY (Don't Repeat Yourself) — the principle that each piece of knowledge should live in exactly one place in the code. If the same logic is copied into several spots, the copies will drift: someone fixes a bug or changes a constant in one copy and forgets the others, so the program now behaves inconsistently and no one can tell which copy is "right." This is called copy-paste drift. The cure is to write the logic once and call it from everywhere it is needed.

In PowerPy, suppose the formula that converts a measured current to a temperature-corrected current is pasted into cell.py, string.py, and section.py. A physicist later refines the formula. If you update only two of the three copies, sections now silently use the old, wrong physics — a bug that is maddening to find because the code "looks" correct in each place. DRY says: put that formula in one function and have all three call it. Fix it once, fixed everywhere.

Term YAGNI (You Aren't Gonna Need It) — the principle that you should not build features, options, or generality on the speculation that they might be useful someday. Build what the current task actually needs; add the rest when a real need appears. Speculative code is pure cost: it must be written, read, tested, and maintained, yet usually the imagined future never arrives, or arrives in a shape your guess did not fit.
Note DRY and YAGNI pull in slightly different directions, and the tension is healthy. DRY pushes you to factor out shared logic; YAGNI warns you not to invent an elaborate, configurable "framework" to do so before you actually have two or three real cases that share the logic. Remove duplication you have; do not build machinery for duplication you merely imagine. Two real repetitions justify a shared function; one hypothetical future one does not.

Abstraction

Term Abstraction — programming in terms of what something does, not how it is represented or implemented. You give an idea a name and a small set of meaningful operations, and then work with that idea rather than with its raw underlying data. Abstraction lets you think at the level of the problem ("a substrate has a surface area") instead of the level of the bytes ("a dictionary with a 'dims' key holding a tuple").

Concretely in PowerPy: a raw dict of substrate keys — something like {'w': 0.3, 'h': 0.2, 'mat': 'carbon-fibre'} — forces every caller to know the magic key names and to do the area arithmetic themselves, scattered everywhere. A Substrate object instead offers substrate.surface() and other named operations. Callers say what they want (the surface) and never touch the how (the keys, the multiplication). If the internal representation changes from a dict to something else, the object absorbs the change; with raw dicts, every caller would have to be hunted down and edited. Abstraction is the difference between naming an idea and exposing a pile of keys.

D1.3 Worked example — one function doing three jobs

Principles land best on real code. Here is a small, very common mistake: a single function that loads a file, computes something, and plots the result — three concerns fused into one. (Note: three concerns here, deliberately smaller than the legacy monolith's five jobs above — the lesson is the same at any size.) The toy reads a JSON file (JavaScript Object Notation, a plain-text format for structured data) and saves a PNG (Portable Network Graphics, an image file).

Worked example The bad version — three concerns in one function.
Step 1 — look at what it mixes. Read it and list the jobs: 
def report_cell(path):
    # concern 1: LOAD — read the cell data from disk
    with open(path) as f:
        data = json.load(f)

    # concern 2: COMPUTE — the physics
    pmp = data["vmp"] * data["imp"]
    ff  = pmp / (data["voc"] * data["isc"])

    # concern 3: PLOT — draw and save a chart
    plt.bar(["Pmp", "FF"], [pmp, ff])
    plt.savefig("report.png")
    return pmp, ff
One function, three concerns: reading a file, doing the maths, and drawing a picture. They are welded together.
Step 2 — feel the pain. Suppose you want to test the maths — check that a known cell gives the right Fill Factor. You cannot, not cleanly: calling report_cell forces a real file to exist on disk and pops open a plotting library and writes a PNG, just to get two numbers. Suppose you want the same maths in a web page that shows no chart: you cannot reuse it without also dragging in the file-reading and the plotting. The three concerns cannot be exercised, tested, or reused independently because they are not separate.
Step 3 — split along the concerns. Give each job its own function, each with a small, clear interface: 
def load(path):                 # concern 1: input only
    with open(path) as f:
        return json.load(f)

def compute(data):              # concern 2: pure maths, no I/O
    pmp = data["vmp"] * data["imp"]
    ff  = pmp / (data["voc"] * data["isc"])
    return pmp, ff

def plot(pmp, ff):              # concern 3: output only
    plt.bar(["Pmp", "FF"], [pmp, ff])
    plt.savefig("report.png")

# the caller wires the three thin pieces together:
data       = load("cell.json")
pmp, ff    = compute(data)
plot(pmp, ff)
Check: the split is better on both design goals. Cohesion rose — each function now does exactly one job, so its name fully describes it. Coupling fell — compute() depends on nothing but the data it is handed; it touches no disk and no plotting library. Concretely: compute() is now testable on its own (feed it a dictionary of known numbers, assert the Fill Factor — no files, no charts) and reusable on its own (call it from a web page, a batch job, a test, with or without a plot). That is the entire payoff of separating concerns: each piece can be understood, tested, and reused by itself.

D1.4 Why splitting a working program makes it better

A fair objection from someone new to design: "the original report_cell worked. Splitting it into three functions just made more functions to keep track of. How is more files an improvement?" This is the right question to wrestle with, so let us climb to the answer one rung at a time.

Think it through — "why is splitting better, not just more files?"
Q1. What is a "reason to change" a piece of code?
It is any kind of future event that would force you to edit that piece: a file format changes, a physics equation is corrected, a plotting library is swapped. Each is an independent source of change. A piece with one reason to change is touched rarely and for one clear cause; a piece with five reasons is touched constantly, for unrelated causes.
Q2. What is coupling, in one sentence, and why does it hurt?
Coupling is how much one piece depends on another's internals. It hurts because dependence transmits breakage: if A is tightly coupled to B, then changing B can break A even though you never touched A. The more wires between pieces, the more places a single change can ripple to.
Q3. So what happens when one 2000-line class mixes five jobs (load, maths, netlist, simulate, plot)?
All five reasons to change now collide in one place, so the class is edited constantly and for unrelated reasons. Because the jobs share the same internals, they are tightly coupled to each other: a change to the plotting can disturb the maths, an edit to the loader can break the simulation, and you cannot touch one job without risking the other four. And you cannot exercise any one job in isolation — to run the maths you must also load files and open the plotter. The class is hard to read, hard to change safely, and hard to test.
Q4. How do small, separated parts help with testing and change?
Testing: a small part with one job and few dependencies can be checked on its own — hand compute() some numbers and assert the result, with no files and no plots in the way. You can pin down each part's correctness independently. Change: because each part has one reason to change and thin connections to the rest, an edit stays local — fix the loader without touching the maths, swap the plotter without touching the physics. The blast radius of any change shrinks to one part.
Q5. So — why does splitting a working program into smaller parts make it BETTER, not just more files?
Because "more files" is not the point — fewer reasons to change per part and fewer wires between parts is. Splitting raises cohesion (each part now does one job, so it is easy to understand) and lowers coupling (each part depends on little, so changes stay local and do not ripple). The payoff is concrete: you can read one part without reading the whole, test each part alone, and change a part's insides without breaking its users. The program does the same thing it always did — but it has become cheap to understand and safe to change, which for code that lives for years is worth far more than saving a few function definitions.

D1.5 The opposite failure — over-engineering

By now the moral seems to be "always split, always abstract." Not quite. There is an equal and opposite mistake, and a beginner who has just discovered design principles is especially prone to it.

Pitfall — over-engineering Splitting a simple program into too many tiny abstractions is just as harmful as cramming a big program into one monolith — sometimes worse. If a fifty-line one-off script is shattered into a dozen micro-classes, five interfaces, and a configuration layer "for flexibility," nobody can follow the flow: the logic is smeared so thin across so many files that reading it means hopping between twenty places to trace one action. That is low readability by a different route, and it is the direct violation of YAGNI — you built generality and structure the problem never asked for. Match the structure to the size of the problem. A small script deserves a few clean functions, not an enterprise architecture; a large, long-lived system earns its subpackages and interfaces. The goal is never "maximum splitting" — it is the right amount of structure for the job, no more and no less.
Key idea Design is a balance, not a direction. Too little structure gives a tangled monolith; too much gives a fog of needless abstractions. Both are bad, and both lower the very readability you were trying to protect. Aim for the modest middle: enough separation that each part has one job and thin connections, and not one layer more than the problem genuinely needs. When in doubt, prefer the simpler structure and add more only when a real need shows up — that is YAGNI guarding you from over-engineering.

2 Structuring a Python module and package well

A simulator is not one big program; it is a pile of files that have to find one another. Where you put those files — and what you name them — decides whether a newcomer can read the code, whether you can swap a file format without re-deriving the physics, and (we will see a real PowerPy example) whether an innocent-looking filename can silently break pandas. This section teaches the vocabulary of Python's file organisation from the ground up — module, package, __init__.py, the public API, absolute and relative imports — then uses PowerPy's own layout as the model to copy, and ends on the trap that gives this section its bite: a module named string.py that shadows the standard library. We define every term the moment it appears.

2.1 The two foundational words: module and package

Everything in this section is built on two ideas. Get these exactly right and the rest follows.

Term module — a single Python file. One file cell.py is a module named cell. When you write import cell, Python finds the file cell.py, runs it top to bottom once, and hands you back an object whose attributes are the names defined inside (its functions, classes, variables). "Module" and ".py file" are, to a first approximation, the same thing.
Term package — a folder of modules that Python treats as one importable unit, marked by a file named __init__.py inside it. The folder name becomes the package name. A package can hold sub-folders that are themselves packages — those are subpackages — so a package is a tree of folders and files, not a flat list.
Key idea A module is a file; a package is a folder of modules. The folder earns the title "package" by containing an __init__.py. PowerPy is a package: the top-level folder powerpy/ contains an __init__.py and a set of subpackages — schemas/, loader/, simulation/, render/ — each of which is again a folder of modules. So powerpy.simulation.solver reads literally as "the solver.py module, inside the simulation subpackage, inside the powerpy package": folder, folder, file.
Note — the dotted name is a path The dots in powerpy.simulation.solver mirror the folders on disk exactly the way the slashes do in powerpy/simulation/solver.py. Reading an import statement, you can always recover where the file lives by replacing each dot with a folder boundary and adding .py at the end. This one-to-one mapping between dotted names and disk layout is why a tidy folder structure and tidy imports are the same achievement.

2.2 PowerPy's layout — the model to copy

PowerPy already uses a clean, layered layout. We teach it here not just as "where PowerPy keeps things" but as the pattern you should imitate for any simulator. There are four subpackages, and they form a stack — each one is allowed to use the layers below it, never above.

The four subpackages of powerpy/, listed bottom of the stack to top. Each layer may depend only on the layers below it.
LayerSubpackageIts one jobMay use
1 (bottom)schemas/Describe the data shapes — dataclasses for a cell, a layout, the fluxes. Pure structure, no behaviour.nothing (only the standard library)
2loader/Read JSON and other files into those shapes — parsing, validation, turning text on disk into schemas objects.schemas
3simulation/The physics. Compute what the array does: cell_level, string_level, section_level, panel_level, array_level, plus environment, solver, pipeline.schemas, loader
4 (top)render/Turn results into output for humans — figures, report.schemas, loader, simulation
Term dataclass — a small Python class whose main purpose is to hold named fields of data, written with the @dataclass decorator so Python generates the boilerplate (the constructor, equality, a readable repr) for you. A schemas dataclass for a cell is just a labelled bundle: area, Vmp, Imp, and so on. It carries the shape of the data without doing any physics.
Term layer / layering — organising code into levels where each level is allowed to call only the levels beneath it. The arrows of dependency all point one way, downward. A layered design has no cycles: render reaches down to simulation, which reaches down to loader and schemas — but schemas never reaches up to know about rendering. Lower layers are ignorant of higher ones.
Key idea — why this layering pays for itself Because each layer touches only the ones below, you can replace a layer without disturbing the rest. Suppose the cell data arrives as CSV next year instead of JSON. You rewrite one file in loader/; simulation/ does not change a single line, because it never knew the data came from JSON in the first place — it only ever saw finished schemas objects. The file format is quarantined inside loader. The same insulation works the other way: you can re-style every figure in render/ without any risk of perturbing the physics, because the physics cannot see render at all.
Note — the names tell the story Notice the simulation modules are named for the array hierarchy itself: cell_level, string_level, section_level, panel_level, array_level. A reader who has met the cell → string → section → panel ladder can guess what each file does before opening it. Then environment supplies the conditions (sun, temperature), solver does the numerical work, and pipeline wires the steps together into one run. Good filenames are free documentation.

2.3 __init__.py and the public API

The file __init__.py is the one piece of machinery that turns a plain folder into a package. It does two jobs, one structural and one editorial.

Term __init__.py — the file whose presence marks a folder as a package. It runs automatically the first time anything in the package is imported. It may be completely empty (its mere existence is enough to make the folder importable), or it may contain code — most usefully, a handful of imports that decide what names the package offers to the outside world.
Term public API — Application Programming Interface: the set of names a module or package intends for outsiders to use, as opposed to its private internals. The "public" part means "promised and supported"; everything else is an implementation detail you should be free to change. A package shapes its public API in __init__.py by importing the chosen names up to the top level.

Concretely, if __init__.py contains

from .cell import cell

then a user can write the short, clean from powerpy import cell instead of the long from powerpy.simulation.cell_level import cell. You have chosen to promote cell to the front door and to keep the exact module it lives in as a private detail you may rearrange later. That is what "exposing a clean public API" means: the package presents a tidy, stable set of names and hides the folder structure behind them.

Note — the leading dot The . in from .cell import cell is not decoration; it means "from this package." We unpack that next — it is the difference between a relative and an absolute import, and getting it wrong is a real and common bug.

2.4 Imports: absolute vs relative

There are two ways to name the thing you are importing, and inside a package you will meet both.

Term absolute import — names the target by its full path from the top-level package, e.g. from powerpy.simulation.electric import ng_sim. It reads the same no matter which file it appears in, because it always starts from the root. Absolute imports are explicit and unambiguous.
Term relative import — names the target relative to the current module's own package, using leading dots. One dot means "the package this file is in"; two dots mean "the package one level up." Inside the simulation subpackage, from .electric import ng_sim means "from the electric module sitting beside me." It is shorter and it survives the whole package being renamed, because it never spells out the top-level name.
Key idea A leading dot is a position, not a name. from .cell import cell says "look here, in my own folder." from .electric import ng_sim says the same for a sibling module. There is no such place as a folder literally called "."; the dot is resolved at import time to the package the running file belongs to.
Pitfall — a wrong import that breaks the module A very common bug is writing an absolute import that names a top-level package which does not exist. Inside powerpy, writing
from power.cell import cell
(note: power, not the leading dot, and not even the right package name) makes Python hunt for a top-level package called power. There isn't one, so the import raises ModuleNotFoundError: No module named 'power' and the whole file fails to load. The fix is the relative form that says "from this package": from .cell import cell. The single missing dot is the entire bug. Whenever you see a ModuleNotFoundError for a name that looks almost like your own package, suspect a dropped dot first.

2.5 How Python finds a module — and the search path

To understand the trap in §2.6 you first need to know how import X locates the file. Python does not scan your whole disk. It walks a specific, ordered list of folders and stops at the first match.

Term search path — sys.path — a Python list of folder names that import consults in order, top to bottom, taking the first folder that contains a matching module. It includes the standard library's location and the installed-packages folder (site-packages). You can print it: import sys; print(sys.path).
Term sys.path[0] — the first entry — the slot searched before all others. When you run python script.py, Python inserts the folder containing script.py at sys.path[0]. When you start the interactive interpreter (the REPL), it inserts the current working directory there instead. Either way, "where you are" gets first say in which module wins.
Term current working directory (cwd) — the folder your shell is "in" when you launch Python — what cd set and pwd reports. It is a property of how you ran the program, not of where the program's files live. The same script can be launched from many different cwds, and — crucially — that choice changes sys.path[0].
Key idea — first match wins, and cwd often goes first Two facts combine into the whole trap. (1) import takes the first folder on sys.path that has the module — it does not prefer the standard library, it simply stops at the earliest hit. (2) The folder you launched from is placed at the front of that list. So if your launch folder contains a file with the same name as a standard-library module, your file is found first and the standard library's version never gets a look in.

2.6 The name-shadowing trap (a real PowerPy gotcha)

This is the centrepiece of the section, so we go slowly. PowerPy's legacy code contains a module named string.py. The Python standard library also has a module named string (it holds things like string.ascii_letters). Two different files, same name. Which one wins an import string depends entirely on sys.path order — and that depends on where you ran Python from.

Term shadowing — when a name you have introduced hides an existing name that would otherwise be visible, so references resolve to yours instead of the original. A local file string.py earlier on sys.path shadows the standard library's string: every import string in the whole process now reaches your file. The standard one is still installed; it is simply never found, because the search stops before it.
Worked — how the trap springs
1. You sit inside the package directory. Your shell's cwd is .../powerpy/ (or wherever string.py lives), and you launch python or run a script from there. Python puts that folder at sys.path[0].
2. Something does import string. It need not be your code. Deep inside the library stack, dateutil (the date-parsing library that pandas relies on) contains a plain import string.
3. First match wins — and yours is first. Python checks sys.path[0] (your folder), finds string.py, and binds the name string to your module. The standard library's string, further down the path, is never reached.
4. The wrong module misbehaves. Your string.py does not define string.ascii_letters (or whatever the caller expected). dateutil gets an AttributeError on a name it was sure would exist — and so pandas breaks, on a line that has nothing to do with your code.
The bug is spooky: nothing you wrote imported string wrongly, and the error surfaces in a third-party library. The only "mistake" was a filename plus a launch directory. Change either — rename the file, or run from elsewhere — and the symptom vanishes.
Note — why sys can't be shadowed but string can The trap only works on modules Python finds by searching the path. A few core modules (sys itself, builtins) are compiled into the interpreter and resolved before the path is consulted, so a local sys.py could not hijack them. string, by contrast, is an ordinary path-based module living in the standard-library folder — exactly the kind a same-named local file can get in front of. That is why string.py is dangerous in a way sys.py would not be.
Key idea — two lessons in one First: do not name your modules after standard-library modules. Second: the current working directory affects imports — where you run Python from can change which file an import resolves to. Both lessons point at the same cure, which we reach in §2.8: install the package, and run from outside it, so the cwd no longer sits in front of the standard library.

2.7 Naming and organisation conventions

Beyond avoiding clashes, a few plain habits keep a package readable as it grows.

  • snake_case module names. Lower-case words joined by underscores: cell_level.py, string_level.py, array_level.py. No capitals, no hyphens (a hyphen is illegal in an import name — cell-level.py cannot be imported).
  • One clear responsibility per file. A reader should be able to say in a single sentence what a module is for. If the sentence needs the word "and" three times, the file is doing too much.
  • Keep files small. A 700-line class that mixes five jobs — reads files, runs physics, formats numbers, draws plots, writes a report — is a code smell: a sign the design wants splitting. Break it along its jobs into separate modules, each in the layer where it belongs.
Term code smell — a surface symptom that hints at a deeper design problem. It is not a bug; the code may run perfectly. It is a warning sign — like a giant multi-job file — that the structure will be painful to change later. "Smell" because you notice it before you can prove anything is wrong.
Note — legacy vs. modern, side by side PowerPy carries both patterns, and the contrast is the lesson. The legacy files cell.py, string.py, section.py, panel.py are large, multi-job modules — and one of them, string.py, is the very file behind the shadowing trap. The modern simulation/ subpackage splits the same responsibilities into focused modules (cell_level, string_level, …). For new code, follow the modern pattern: small, single-job modules grouped into layered subpackages. The legacy files are kept working, but they are not the model to imitate.

2.8 Packaging: making it installable

The deepest fix for the cwd trap is not to memorise rules about where to stand when you launch Python — it is to make the package installable, so Python can find it by name from anywhere, with no help from the current directory.

Term pyproject.toml / setup — the configuration file that describes your package to Python's packaging tools: its name, version, dependencies, and which folders are the package. pyproject.toml is the modern form (older projects used a setup.py/setup.cfg). It is what lets pip install your code like any other library.
Term editable install — pip install -e . — installs the package in editable (a.k.a. "development") mode. Instead of copying the files into site-packages, pip drops a link there pointing back at your source folder, and registers the package by name on sys.path. The "-e" is for editable and the "." means "the package described in this directory." Result: import powerpy works from any directory, and edits to your source take effect immediately — no reinstall.
Key idea — installing dissolves the trap Once PowerPy is installed, it lives on sys.path as a named package, reachable as powerpy.simulation.string_level, and you run your programs from a normal working directory — not from inside the package. The package folder is therefore no longer sitting at sys.path[0] in front of the standard library, so a bare import string resolves to the real standard-library string again. The cwd stops mattering because imports now go through the installed name, not through "whatever folder I happen to be standing in." Install the package and the whole class of "it works on my machine, in this folder" bugs disappears.

2.9 Dependencies and data files

A simulator ships data, not just code: PowerPy keeps cell, diode, and substrate specifications as JSON under a data/ folder. How you reach that data matters as much as where you put it.

Pitfall — hard-coded paths break after install The tempting, wrong way is to open the data with a fixed relative path like open("data/cells.json"). That only works if the cwd happens to be the project root — the very assumption we just spent §2.8 escaping. After an editable or real install, the files live wherever pip put the package, and "data/cells.json" points at nothing. The path-based open is the data twin of the import-shadowing bug: both rely on where you launched from.
Term package-resource lookup — importlib.resources API — the standard-library way to open a data file relative to the package it ships with, regardless of where that package is installed. You ask, in effect, "give me cells.json from inside the powerpy data," and Python locates it next to the installed code. No cwd, no hard-coded absolute path. This is what makes the data still load after pip install.
Key idea Code finds its own data through the package, not through the filesystem's current position. data/ rides along inside the package; importlib.resources asks the package where its files are. The rule mirrors imports exactly: don't depend on the current directory — depend on the package.

2.10 Worked example: a tiny package, inside vs outside

Here is a minimal package, small enough to hold in your head, that exhibits everything above — including, deliberately, the trap.

mypkg/
├── __init__.py     # marks mypkg as a package; can expose the public API
├── core.py         # the real work
├── io.py           # reads/writes data   <-- NOTE: 'io' is itself a stdlib name!
└── tests/
    ├── __init__.py
    └── test_core.py

The files connect with imports. Inside __init__.py we promote one name to the front door:

# mypkg/__init__.py
from .core import run        # relative: 'run' from the core module beside me

Inside core.py, the physics asks its sibling to load data:

# mypkg/core.py
from .io import load_table   # relative: the io module in my own package

def run(path):
    data = load_table(path)
    return summarise(data)

And a test, run from outside, uses the clean public name the package chose to expose:

# tests/test_core.py
from mypkg import run        # absolute: works because mypkg is importable by name
assert run("sample.json") is not None
Note — io.py is a live instance of the trap The data module above is named io.py, and the standard library also has an io module (it underlies file objects). Inside the package we are safe, because from .io import load_table has a leading dot — it explicitly means "my io," so there is no ambiguity. The danger is a bare import io executed while standing in the package folder: that could pick up mypkg/io.py instead of the real one. The same medicine applies — prefer a less collision-prone name (data_io.py), and install the package so you are never running from inside it.

Now watch the difference between where you run Python from.

Worked — running from outside vs inside
From outside (the right way). Your cwd is the parent folder, above mypkg/ (or the package is installed). You run python -m mypkg.tests.test_core. sys.path[0] is the parent, not the package interior, so a bare import io anywhere resolves to the standard-library io. Everything behaves.
From inside (the trap). You cd into mypkg/ and run python io.py or just start python there. Now sys.path[0] is the inside of the package. A bare import io — yours, or one buried in a library you imported — finds mypkg/io.py first. The standard io is shadowed, and code that expected the real one misbehaves, often far from the apparent cause.
Identical files, identical code — only the launch directory differs, and that alone flips which io (or string) gets imported. That is the whole shadowing bug in one comparison, and the cure is to stop running from inside: install the package (§2.8) and launch from a neutral directory.

2.11 Ladder — why the launch directory changes which module loads

Ladder — from "what is a module" to "why does WHERE you run Python change which module imports, and how do we stop it mattering?"
Rung 1 — What is a module, concretely, on disk?
A single .py file. import string means "find a file string.py (or the stdlib string), run it once, give me the resulting object." Because a module is just a file Python has to locate, the question of which file becomes live the moment two files share a name.
Rung 2 — How does Python decide which file an import gets?
It walks sys.path, an ordered list of folders, and takes the first one containing a match — it does not privilege the standard library. And the folder you launched from is inserted at the front (sys.path[0]): the script's own folder for python script.py, the cwd for the REPL. So "where you are" gets first vote.
Rung 3 — So what is shadowing, in this machinery?
If your launch folder holds string.py, it sits at sys.path[0], ahead of the standard library. Every import string in the process — including one hidden inside dateutil, hence pandas — finds your file first. Your file shadows the real one: the standard string is still installed, just never reached. The result is a breakage in code you never touched.
Rung 4 — How does installing the package make the launch directory stop mattering?
pip install -e . registers the package by name on sys.path (via site-packages) and lets you run from a neutral directory instead of from inside the package. The package interior is no longer at sys.path[0] in front of the stdlib, so a bare import string resolves to the standard library again. Imports now travel through the installed name (powerpy.…), which is the same wherever you stand — so the cwd no longer changes the outcome. That is the answer to the whole ladder: it is sys.path order, the launch folder rides at the front, and installing replaces "the folder I'm in" with "the package by name."
Pitfall — the two habits that cause this whole family of bugs (1) Naming a module after a standard-library module — string.py, random.py, types.py, json.py, io.py, email.py, code.py. Any of these can shadow the real one and break libraries that import it. Check a candidate name against the standard library before you use it. (2) Relying on the current directory for imports (or data) instead of installing the package. Code that only works when launched from one magic folder is fragile and "works on my machine"-shaped. Make the package installable (pyproject.toml + pip install -e .), reach data through importlib.resources, and run from a neutral directory — then neither the filename clash nor the cwd can bite.

3Writing Functions and Classes That Are Clear and Correct

3   Writing functions and classes that are clear and correct

The previous sections were about what PowerPy computes — IV curves, thermal balance, the coupling loop. This section is about how to write the code that computes it so that you, and the next person, can trust it. A simulator is only as believable as its quietest function. A subtle sign error or a mis-indexed return value will not crash; it will hand you a plausible-looking number that is wrong, and you will plot it, believe it, and ship it. The whole craft here is making code that is hard to fool yourself with: small pieces that do one thing, named so plainly that the name is the documentation, with inputs checked at the door and failures that shout instead of whisper.

The one idea. Code is read far more often than it is written, and in a scientific codebase it is trusted more often than it is read. So optimise for trust: write small, single-purpose, well-named functions that take their inputs as arguments and return their results as values (rather than secretly mutating hidden state), document them with a docstring and type hints, bundle related data in a dataclass instead of a bare tuple or dict, validate inputs early, and fail loudly when something is wrong. Every one of those choices makes a class of bug either impossible or obvious. This section defines each term from scratch and shows the bug it kills.

We will build the vocabulary first (a function that does one thing; a pure function; naming conventions; docstrings; type hints; dataclasses; exceptions), then walk two full before/after refactors, then climb a Socratic ladder to the question that ties it all together: why is a small pure function with a dataclass result easier to trust than a big method that mutates self and returns a tuple?

3.1   A good function does ONE thing

The single most useful rule for functions is also the simplest: a function should do one thing, and do it well. If you cannot describe what a function does without using the word "and", it is probably doing two things and wants to be two functions.

Why does this matter so much? Three reasons, and they compound:

  • You can name it. A function that does one thing has an obvious name. A function that "validates the config and loads the cell data and computes the short-circuit current and formats it for the report" has no good name — so it ends up called something vague like process() or do_stuff(), and the vagueness of the name tells you the function is overloaded.
  • You can test it. One thing means one behaviour to check. You feed it inputs, you assert on the output. A four-job function needs you to set up four worlds at once just to test the third job.
  • You can reuse it. The short-circuit-current calculation buried inside that mega-function cannot be called from anywhere else, because calling it drags along the validation, the file loading, and the formatting you did not want.
Single Responsibility, in one sentence. A function (or a class) should have one reason to change. If the report format changes and the physics changes and the file format changes, and all three force edits to the same function, that function is carrying three responsibilities that should have been three pieces. This is the "Single Responsibility Principle" (SRP) applied at the smallest scale.

3.2   Return values, not hidden mutations: the pure function

There are two ways a function can deliver its result. It can return a value — you call isc = compute_isc(cell, temperature) and the answer comes back in your hand. Or it can mutate hidden state — you call cell.update_isc() and the answer is silently written into cell.isc somewhere inside the object, where you have to know to look for it. The first style is almost always easier to reason about, and the reason is captured by one precise concept.

Pure function. A function whose output depends only on its inputs, and which has no side effects — it does not read or write any state outside itself (no globals, no mutating its arguments in place, no writing files, no touching self). Call a pure function twice with the same arguments and you get the same answer, every time, regardless of what else the program has done. compute_isc(cell, t) that takes a cell and a temperature and returns a number, touching nothing else, is pure. cell.recompute() that reaches into self and overwrites several attributes is not.
Side effect. Any change a function makes to the world beyond returning a value: mutating one of its arguments, writing to a global or to self, printing, logging, reading a clock, hitting the disk or network. Side effects are not evil — a program that only computed pure values and never wrote a file would be useless. The point is to know where they are and keep them out of the functions whose correctness you most need to trust.
Small + pure = easy to test and reason about. A small pure function is the easiest thing in all of software to be sure about. To test it you need no setup, no mocks, no fixtures: you call it with inputs and assert on the output. To reason about it you need to read only that function — nothing it does can depend on hidden state, so nothing outside it can change its behaviour. That is the gold standard we push the physics toward: make the core calculations pure, and confine the messy side-effecting parts (file I/O, the ngspice call) to a thin, clearly-marked shell around them.

Concretely, prefer the left over the right:

# GOOD — pure: output depends only on inputs, returns a value
def compute_isc(cell, temperature):
    """Return the short-circuit current of `cell` at `temperature` (A)."""
    return cell.isc_ref * (1 + cell.alpha_isc * (temperature - cell.t_ref))

isc = compute_isc(cell, 40.0)        # answer in hand; cell untouched

# AVOID — side-effecting: mutates hidden state, returns nothing
def update_isc(cell, temperature):
    cell.isc = cell.isc_ref * (1 + cell.alpha_isc * (temperature - cell.t_ref))
    # ...and now the rest of the code must KNOW that cell.isc changed in here

The right-hand version is not wrong, but it is harder to trust: to know what update_isc did you must know it reached into cell; to test it you must build a cell, call the method, then go fishing for cell.isc; and if two parts of the program both mutate cell, the order in which they ran now matters. The pure version has none of those worries.

3.3   Few parameters, and a verb-or-noun name

Few parameters. A function with two or three parameters is easy to call correctly. A function with seven is a minefield: you will eventually pass them in the wrong order, and if several are the same type (four floats, say) the language will not catch you. When a function genuinely needs many related values, that is a signal to bundle them — which is exactly what a dataclass is for (§3.6).

A name that says what it is. Two tiny rules carry most of the weight:

  • Verbs for actions. A function that does something or computes something gets a verb phrase: compute_isc, build_models, solve_circuit, load_config. Reading the call site then reads like a sentence: "compute isc".
  • Nouns for values. A function or property that simply is a value gets a noun: cell_count, bus_voltage, operating_point. You are naming the thing it gives you, not an action.
The name is the first line of documentation. A reader meets the name before they read the body. compute_isc(cell, t) tells them, before they look inside, that this returns a short-circuit current for a cell at a temperature. proc2(c, t) tells them nothing and forces them to read the body to learn what they could have read in the name. Spend the name well; it is read the most and costs the least.

3.4   Naming conventions: PEP 8

Beyond "verb or noun", Python has a shared, community-wide style guide so that everyone's code looks the same and you spend zero attention decoding formatting. It is called PEP 8.

PEP. Python Enhancement Proposal — a numbered design document for the Python language and community. PEP 8 is the one that matters here: it is Python's official style guide, the agreed convention for how Python code should be formatted and named. Following it is not about taste; it is about being instantly readable to every other Python programmer.

The naming rules you must internalise:

  • Functions and variables: snake_case — lowercase words joined by underscores: compute_isc, bus_voltage, cell_count.
  • Classes: CapWords (also called PascalCase) — each word capitalised, no underscores: Environment, Substrate, SolarCell.
  • Constants: UPPER_CASE with underscores: BOLTZMANN, T_REF, MAX_ITERATIONS.

And the spirit behind them: descriptive, consistent, no cryptic abbreviations. temperature beats tmp; short_circuit_current or the well-known isc beats x3; regressor beats rgsr. The cost of a longer name is a few keystrokes once; the cost of a cryptic one is paid by every reader, forever.

Pitfall — inconsistent or cryptic names. The worst naming sin is not ugliness, it is inconsistency: calling the same quantity temp in one function, T in the next, and theta in a third forces the reader to prove to themselves that the three are the same thing. Pick one name per concept and use it everywhere. Equally, a clever abbreviation that saved you two seconds ("vmp", fine, it is standard; "vmpp_t_adj_fin", not fine) costs every future reader a decode.

3.5   Docstrings: say what a function does

Docstring. A string literal placed as the first statement inside a function, class, or module, written in triple quotes, that documents it: a short description of what it does, what its parameters mean, and what it returns. Unlike a comment, a docstring is a real object Python keeps at runtime (help(compute_isc) prints it), and tools (IDEs, doc generators) read it. PowerPy's existing code already has good docstrings — that is a strength worth preserving as the code grows.
def compute_isc(cell, temperature):
    """Short-circuit current of a cell at a given temperature.

    Parameters
    ----------
    cell : SolarCell
        The cell whose reference parameters (isc_ref, alpha_isc, t_ref) are used.
    temperature : float
        Cell temperature in degrees Celsius.

    Returns
    -------
    float
        Short-circuit current in amperes (A).
    """
    return cell.isc_ref * (1 + cell.alpha_isc * (temperature - cell.t_ref))

Notice the discipline: one summary line, then the parameters with their units, then the return value with its unit. In a scientific codebase, units in the docstring are not decoration — a function that quietly expects kelvin while you feed it celsius is a textbook source of silent error, and the docstring is where that contract is stated.

3.6   Type hints: document expected types and catch mistakes

Type hint. An optional annotation on a function's parameters and return value that states the type each is expected to be — for example def thermal(p_sun: float, area: float) -> dict: says "p_sun is a float, area is a float, and this returns a dict". Hints are not enforced by Python at runtime (the code runs the same with or without them), but they document the contract and let tools — type checkers like mypy, and your editor's autocomplete — catch a whole class of mistakes before you ever run the code, such as passing a string where a float was meant.
# Without hints — the contract lives only in your head
def thermal(p_sun, area):
    ...

# With hints — the contract is on the page and tools can check it
def thermal(p_sun: float, area: float) -> dict:
    ...
Hints are optional, but valuable here. For a small script you can skip them. For a scientific codebase full of floats, arrays, and dicts being threaded through many functions, type hints earn their keep: they tell the reader (and the checker) that thermal wants two floats and yields a dict, so passing it a NumPy array by mistake, or unpacking its result as a tuple, gets flagged early instead of blowing up three functions downstream. They are documentation that the machine can verify.

For the array-heavy parts of PowerPy, hints like numpy.ndarray on a parameter immediately tell a reader "this is a vector over all cells, not a single value" — a distinction that is otherwise invisible and easy to get wrong.

3.7   Dataclasses: bundle related fields with names

Functions often need to carry several related values together — a temperature, an irradiance, a sun angle. You can do that with a bare tuple or a dict, but both are traps. A tuple makes you remember positions (was voltage element 2 or element 3?); a dict makes you remember string keys ("vmp" vs "v_mp" vs "Vmp") and mis-typing one fails silently or at runtime. The fix is a dataclass.

Dataclass. A class declared with the @dataclass decorator whose job is to bundle a set of named fields together, with the boilerplate (constructor, equality, a readable repr) generated for you. Instead of passing around an anonymous tuple or a stringly-keyed dict, you get a named type with named attributes you reach by . — env.temperature, env.irradiance — which the editor autocompletes and the type checker verifies.
from dataclasses import dataclass

@dataclass
class Environment:
    temperature: float      # ambient temperature, °C
    irradiance: float       # incident solar flux, W/m^2
    sun_angle: float        # angle of incidence, degrees

env = Environment(temperature=28.0, irradiance=1361.0, sun_angle=0.0)
# Reach fields by NAME, with autocomplete and type-checking:
q_in = env.irradiance * area

PowerPy's modern Environment dataclass and the planned Substrate are good examples of this style. Contrast them with the legacy habit they replace: a function that returned a 6-tuple

# LEGACY — a 6-tuple: positional, undocumented, easy to mis-index
return (model, config, isc, imp, vmp, voc)

# ...called like this, somewhere far away:
result = build_cell(...)
v_mp = result[4]      # is vmp really index 4? count again... is it imp? voc?

or a dict of string keys ({"isc": ..., "imp": ..., "vmp": ..., "voc": ...}) where a typo'd key — result["Vmp"] — fails. Both push onto every reader the job of remembering an order or a spelling. The dataclass moves that knowledge into the type, where the tools enforce it:

@dataclass
class CellResult:
    model: object

result = build_cell(...)
v_mp = result.vmp        # unambiguous; a typo `result.vmpp` is caught

3.8   Error handling: validate early, fail loudly

The opposite of trustworthy code is code that, when something is wrong, keeps going anyway and hands you a wrong answer with a straight face. The defence is a posture: check inputs at the door, and when they are bad, stop with a clear message rather than limping onward.

Exception. Python's mechanism for signalling that something has gone wrong: you raise an exception (e.g. raise ValueError("temperature must be in °C, got kelvin-sized value")), which immediately stops normal execution and propagates up until some caller catches it with try/except — or, if nobody does, halts the program with a traceback pointing at the problem. An exception with a clear message is a loud, locatable failure: it tells you what went wrong and where.

Two habits follow:

  • Validate inputs early. Check preconditions at the top of the function and raise a clear exception if they are violated — before you compute on garbage. A negative area, an irradiance of NaN, an empty cell list: catch these at the door, where the error message can still say which input was bad.
  • Fail loudly, not silently. A wrong answer that looks right is far more dangerous than a crash. A crash points at the bug; a silently-wrong number gets plotted and believed. So prefer a loud exception over a quiet fallback that papers over the problem.
def thermal(p_sun: float, area: float) -> dict:
    """Thermal balance for one element. Raises ValueError on bad input."""
    if area <= 0:
        raise ValueError(f"area must be positive, got {area}")
    if p_sun < 0:
        raise ValueError(f"p_sun (incident power) must be >= 0, got {p_sun}")
    ...
Pitfall — the bare except: pass that hides real errors. The single most dangerous error-handling pattern is catching everything and doing nothing: 
try:
    result = run_ngspice(netlist)
except:          # bare: catches EVERYTHING, even bugs and Ctrl-C
    pass         # ...and silently swallows it — the failure vanishes
PowerPy's ngspice wrapper does exactly this, and it is a trap: a real failure (a malformed netlist, a missing binary, a parse error) is caught and discarded, so the loop sails on with stale or empty results and "converges" to nonsense. The cure is to catch specific exceptions and log them, so failures are visible and locatable: 
try:
    result = run_ngspice(netlist)
except NgspiceError as exc:
    logger.error("ngspice failed on netlist %s: %s", name, exc)
    raise                     # re-raise: do not hide it
A bare except also catches KeyboardInterrupt and SystemExit, so it can even stop you killing the program with Ctrl-C. Catch the specific thing you can handle; let everything else surface.

3.9   Comments vs. self-documenting code

Good names reduce the need for comments. A comment that merely restates what the code plainly says is noise that can drift out of date and start lying. The rule of thumb: comment the WHY, not the WHAT.

# BAD — the comment just restates the code (and will rot)
i = i + 1          # add one to i
omega = 0.5        # set omega to 0.5

# GOOD — the code says WHAT; the comment says WHY (which code can't)
omega = 0.5        # under-relax: omega=1 oscillates on a strongly heating fault
Pitfall — over-commenting obvious code. A comment on every line, each paraphrasing its line, does not aid the reader — it doubles the text to read and creates two things to keep in sync. When the code changes and the comment does not, the comment becomes a lie, which is worse than no comment. Prefer a clear name (under_relaxation_factor over omega # the under-relaxation factor); reserve comments for the things the code genuinely cannot say — the reason a value was chosen, a non-obvious constraint, a reference to a paper or a gotcha.

3.10   Worked example (a) — refactor a fat function EXAM-HOT

Worked example — splitting a function that validates + loads + computes + formats. Here is a function in the legacy style: it does four jobs, takes many parameters, mutates nothing cleanly, and returns a 6-tuple. We will turn it into three small typed single-purpose functions feeding a @dataclass result.
Before — one function, four jobs, a 6-tuple out. 
def process_cell(path, t, area, alpha, vref, fmt):
    # job 1: validation
    if area <= 0 or t is None:
        return None                      # silent failure: caller gets None
    # job 2: load
    cfg = read_config(path)
    model = cfg["model"]
    # job 3: compute the four IV points
    isc = cfg["isc_ref"] * (1 + alpha * (t - 25))
    imp = isc * 0.95
    vmp = vref * (1 - 0.003 * (t - 25))
    voc = vmp * 1.18
    # job 4: format for the report
    line = f"{model}: Isc={isc:.2f} Imp={imp:.2f} Vmp={vmp:.2f} Voc={voc:.2f}"
    print(line)
    return (model, cfg, isc, imp, vmp, voc)   # 6-tuple: mis-index waiting to happen
What is wrong with it. Six parameters in an order you must get right; validation that fails silently by returning None (the caller will AttributeError three lines later, far from the cause); loading, physics, and report-formatting welded together so none can be tested or reused alone; and a 6-tuple return whose positions (result[4]?) are an invitation to mis-index. It has four reasons to change.
After — a dataclass result, and three pure, typed, single-purpose functions. 
from dataclasses import dataclass

@dataclass
class CellResult:
    model: str
    isc: float
    imp: float
    vmp: float
    voc: float

def load_cell_config(path: str) -> dict:
    """Read and return the raw cell configuration dict from `path`."""
    return read_config(path)

def compute_iv_points(config: dict, temperature: float,
                      alpha: float, v_ref: float) -> CellResult:
    """Compute the four IV points at `temperature`. Pure: no I/O, returns a value."""
    if temperature is None:
        raise ValueError("temperature must be provided")
    isc = config["isc_ref"] * (1 + alpha * (temperature - 25))
    imp = isc * 0.95
    vmp = v_ref * (1 - 0.003 * (temperature - 25))
    voc = vmp * 1.18
    return CellResult(model=config["model"], isc=isc, imp=imp, vmp=vmp, voc=voc)

def format_cell_result(r: CellResult) -> str:
    """Format a CellResult as one report line. Pure: returns the string."""
    return f"{r.model}: Isc={r.isc:.2f} Imp={r.imp:.2f} Vmp={r.vmp:.2f} Voc={r.voc:.2f}"
Why the after is better. Each function does one thing and is named for it. compute_iv_points is pure — same inputs, same output, no file read, no print — so you can test it with a dict literal and an assert, no disk needed. Validation now raises with a message instead of returning None, so a bad call fails loudly at the door. The result is a CellResult you read by name (r.vmp, never result[4]), so mis-indexing is impossible and a typo like r.vmpp is caught. Loading and formatting are separated out, so swapping the report format or the config source touches one small function, not the physics. ✓

3.11   Worked example (b) — spot and fix two classic bugs EXAM-HOT

Two bugs that genuinely appear in legacy scientific code. Both are the kind that do not announce themselves — one makes a method uncallable, the other corrupts results across calls.

Bug 1 — a duplicated parameter in the signature.
The bug. 
class Cell:
    def setTemperature(self, self, temperature):   # `self` appears TWICE
        self.temperature = temperature
A copy-paste typo has put self in the parameter list twice. Python rejects a duplicate argument name in a function signature, so this raises SyntaxError: duplicate argument 'self' and the method is uncallable — in fact the whole module will not even import.
The fix. Remove the duplicate. A method takes self exactly once, as its first parameter. 
class Cell:
    def set_temperature(self, temperature):        # one `self`; snake_case name
        self.temperature = temperature
(While here, the PEP 8 name is set_temperature, not setTemperature — Python uses snake_case for methods, not the Java-style camelCase.)
Check. One self, then the real parameters. The module imports; cell.set_temperature(40) works. ✓
Bug 2 — a mutable default argument.
The bug. 
def accumulate(value, cache={}):     # default is a MUTABLE dict
    cache[value] = value ** 2
    return cache
This is one of Python's most famous traps. The default value {} is created once, when the function is defined — not fresh on each call. So every call that does not pass its own cache shares the same dict, and it accumulates across calls: 
accumulate(2)   # {2: 4}
accumulate(3)   # {2: 4, 3: 9}   <-- the 2 from last time is STILL there!
Two unrelated calls quietly contaminate each other. In a simulator this surfaces as results from one run leaking into the next.
The fix — default to None, create inside. 
def accumulate(value, cache=None):
    if cache is None:
        cache = {}                   # a FRESH dict on every call that omits it
    cache[value] = value ** 2
    return cache
Now each call with no cache gets its own empty dict, so calls are independent.
Check. accumulate(2) → {2: 4}; accumulate(3) → {3: 9}, with no leftover 2. The same fix applies to any mutable default — list [], set, dict: default to None and build it inside. ✓

3.12   Why a small pure function beats a big mutating method

Q·a (foundation). What is a side effect, and why does a method that mutates self have one?
A side effect is any change a function makes beyond returning a value — writing a global, touching the disk, or mutating its arguments or self. A method like cell.recompute() that overwrites self.isc, self.vmp, … has a side effect: its effect is the hidden mutation of the object, not a returned value. To know what it did you must know which attributes it touched, and after it runs the object is in a new, possibly surprising state.
Q·b (build up). What can go wrong specifically with returning a 6-tuple like (model, config, isc, imp, vmp, voc)?
Position becomes load-bearing and invisible. The caller must remember that vmp is element 4 and voc is element 5 — and nothing checks them. Swap two fields of the same type and the bug is silent: result[3] still returns a float, just the wrong one. Add a seventh field at the front and every index downstream shifts by one. There is no autocomplete, no type check, no name — only a count you must get right every single time.
Q·c (why). Why is a small pure function so much easier to test than that big mutating method?
Because a pure function's output depends only on its inputs and it touches nothing else, the entire test is "call with inputs, assert on output" — no fixtures, no mocks, no building an object into the right state first, no fishing for mutated attributes afterward, and no worry that a previous test left state behind. Same input, same output, every time. The mutating method needs you to construct a cell, call it, then inspect several attributes, and its result can depend on whatever happened to the object earlier — so the test is bigger, slower, and more fragile.
Q·d (deep — the target). So why is a small pure function with a dataclass result easier to trust than a big method that mutates self and returns a tuple? And what do a type hint or a dataclass catch early?
Put the rungs together. The pure function has no side effects (Q·a), so reading it tells you everything it does and testing it is trivial (Q·c) — nothing hidden can change its answer. Its dataclass result replaces the position-counting and silent mis-indexing of the tuple (Q·b) with named fields the reader and the tools both understand. And the type hint / dataclass catch mistakes early: a hint like -> CellResult lets the checker flag a caller that tries to unpack the result as a tuple or pass a string where a float was meant — before the code runs — and a dataclass turns a typo'd field (r.vmpp) into an AttributeError at the access site instead of a wrong number three functions later. The big mutating method offers none of this: its effect is hidden in self, its tuple is a mis-index trap, and its errors surface far from their cause. We trust the small pure function because there is simply less that can go wrong, and what does go wrong is caught early and located precisely.

4Making It Fast: Performance and Time-Efficiency

4   Making it fast: performance and time-efficiency

The previous sections were about writing code that is clear and correct. This one is about making correct code fast enough — and the very first thing to learn about performance is that almost everyone gets it wrong by starting in the wrong place. They guess where the program is slow, rewrite that part to be clever, and discover the program is exactly as slow as before because the real bottleneck was somewhere they never looked. Performance work is not about being clever; it is about being empirical. You measure, you find the one place that actually costs the time, you fix that, and you measure again. Everything in this section hangs off that discipline.

The one idea. Measure before you optimise. A program's runtime is almost never spread evenly — it nearly always hides in a small hot spot, a handful of lines that run millions of times or call something expensive. Your job is to find that hot spot by measurement (profiling), then attack it with the biggest lever available: usually a better algorithm (so the work grows more slowly with problem size) or vectorisation (so a whole array is processed in one compiled operation instead of a slow Python loop). Cleverness applied anywhere else is wasted effort that only makes the code harder to read. This section defines every term from scratch, works the numbers with units, and shows where PowerPy itself is fast and where it is slow.

We build the vocabulary in the order you would actually use it: first profiling (how to find the hot spot), then Big-O (why the algorithm dominates for large problems), then vectorisation (the single biggest win in scientific Python), then a cluster of "stop doing redundant work" techniques (caching, hoisting, copying cost, batching external calls), then data structures, then parallelism, and finally the discipline of when not to optimise at all. Two fully worked numeric examples and a Socratic ladder tie it together.

4.1   The golden rule: measure first, with a profiler

The temptation, when something feels slow, is to stare at the code, form a theory ("it must be the matrix multiply"), and start optimising on the strength of that theory. This is how most performance effort is wasted. Human intuition about where time goes is famously bad: the bottleneck is routinely a line you would never have suspected — a string being reparsed in a loop, an object being deep-copied on every iteration, an engine being spun up and torn down thousands of times. The cure is to stop guessing and measure.

Profiling. Measuring where a program actually spends its time (and sometimes its memory) while it runs — which functions are called, how often, and how many seconds accumulate in each. A profiler instruments your code and hands you a ranked table: "this function took 71% of the total." That table, not your intuition, tells you what to optimise. Profiling turns "I think it's slow here" into "it is slow here, by measurement."

Python gives you two tiers of tool, for two different questions.

  • Timing one snippet — "how long does this small piece take?" Use time.perf_counter(), a high-resolution clock, around the code; or the timeit module, which runs the snippet many times and reports an average (so a single noisy measurement does not fool you). Good for comparing two implementations of the same small thing.
  • Profiling a whole run — "across this entire simulation, where does the time go?" Use cProfile, Python's built-in profiler. You run the whole program under it and it produces a per-function breakdown — call counts and cumulative time — so the hot spot rises to the top of the table on its own.
# Tier 1 — time a single snippet with perf_counter
import time
start = time.perf_counter()
result = solve_thermal(cells)            # the piece under suspicion
elapsed = time.perf_counter() - start    # seconds, wall-clock
print(f"solve_thermal took {elapsed*1e3:.1f} ms")

# Tier 1 — or let timeit average away the noise (many repetitions)
import timeit
t = timeit.timeit("solve_thermal(cells)", globals=globals(), number=100)
print(f"mean {t/100*1e3:.2f} ms over 100 runs")

# Tier 2 — profile a WHOLE run; the hot function floats to the top
import cProfile
cProfile.run("run_monte_carlo(n=1000)", sort="cumulative")
"Don't guess, profile." This is the oldest slogan in performance work, and it survives because it keeps being true. Run cProfile first, read the top line of the table, and you will usually find that one function — often one you did not suspect — accounts for the bulk of the runtime. Optimise that. A 2× speed-up of a function that is 5% of the runtime buys you 2.5% overall; a 2× speed-up of the function that is 80% of the runtime buys you 40%. The profiler tells you which function is the 80% one. Without it you are optimising blind.

4.2   Algorithmic complexity: Big-O notation

Once the profiler has pointed you at the hot spot, the first and biggest question is not "can I shave a few microseconds off each iteration?" but "is this the right algorithm?" — because for a large problem, a better algorithm beats every micro-optimisation combined, by margins no amount of clever code can close. The tool for reasoning about this is Big-O notation.

Big-O notation. A way of describing how an algorithm's running time (or memory) grows as the size of its input grows. We call the input size N (number of cells, number of Monte-Carlo runs, number of voltage points). Big-O captures the shape of that growth, ignoring constant factors and small terms, because for large N the shape is what dominates. Reading "O(...)": you say "order of".

The handful of growth rates you must recognise on sight, from gentle to catastrophic:

  • O(1) — constant. Time does not depend on N at all. Looking up one value in a dict, reading array[0]. Double the input, same time.
  • O(N) — linear. Time grows in direct proportion to N. One pass over a list. Double the input, double the time. A position sweep over N cells is O(N).
  • O(N²) — quadratic. Time grows as the square of N — typically a loop nested inside another loop, each over N. Double the input, quadruple the time. Tolerable for N in the hundreds, painful in the thousands, hopeless in the millions.
  • O(2^N) — exponential. Time doubles every time you add one to N. This is the wall: at N = 60 you already need more operations than there are atoms you could ever count. Any O(2^N) approach is usable only for tiny N.
Why the algorithm dominates. Constant factors — how fast each individual step is — matter, but they are a fixed multiplier. The growth rate is a multiplier that gets worse with N. So for large enough N, an O(N) algorithm in slow Python beats an O(N²) algorithm in hand-tuned C, and an O(2^N) algorithm loses to everything. Pick the right complexity class first; tune the constant factor second. Getting this order backwards — polishing the constant on a fundamentally bad algorithm — is the most expensive mistake in the section.

PowerPy's example. Suppose you wanted to know the probability that a solar array still meets its power budget given that some cells have failed. The brute-force idea is to enumerate every possible combination of working-and-failed cells. With N cells, each either working or failed, there are 2^N combinations — for a realistic array of several hundred cells that is a number with a hundred-plus digits, utterly impossible to enumerate. This is exactly why Chapter 2's reliability analysis uses Monte-Carlo sampling instead: rather than visit all 2^N cases, it draws a few thousand random failure patterns and estimates the probability from the sample. Monte-Carlo trades an impossible O(2^N) exact computation for a tractable, tunable O(number-of-samples) estimate. By contrast, sweeping a fault through each cell position one at a time is only O(N) — N independent runs — which is why a position sweep is cheap while exhaustive enumeration is not.

Pitfall — polishing the constant on an exponential algorithm. Making each step of an O(2^N) enumeration twice as fast does not rescue it: at N = 300 it is still 2^300 steps, and halving an impossible number leaves an impossible number. The only fix for a bad complexity class is a different algorithm (here, sampling instead of enumeration). Recognise the class before you optimise, or you will tune something that can never finish.

4.3   Vectorisation: the big win in scientific Python

This is the single most important performance technique for numeric Python, and it is the one that turns PowerPy's thermal model from slow to fast. The idea is to stop looping over array elements in Python and instead express the whole computation as operations on entire arrays, which a compiled library (numpy) executes in one shot.

Vectorisation. Replacing an explicit Python for loop that processes array elements one at a time with a single array operation that processes the whole array at once in compiled C code (via numpy). Instead of for i in range(n): out[i] = a[i] + b[i], you write out = a + b — and that one line loops over all elements down in C, not in the Python interpreter.

Why is that faster, when it does the same arithmetic? Because of where the loop runs and what the loop costs per step. In a Python loop, every single iteration pays a tax the interpreter cannot avoid: it has to look up what type each object is, unwrap ("unbox") the float out of its Python object, do the add, wrap the result back into a new Python object, and advance the loop machinery. That overhead is often larger than the arithmetic itself, and you pay it N times. A vectorised numpy operation pays the per-element overhead essentially once for the whole array: it knows up front that the array is a contiguous block of, say, 8-byte floats all of the same type, so it runs a tight compiled loop with no per-element type checks or boxing. The arithmetic is identical; the bookkeeping around it nearly vanishes.

The mechanism in one sentence. A Python loop pays the interpreter's per-iteration overhead (type check + unbox + rebox + loop step) once per element; a vectorised numpy call pays that overhead once for the entire array and does the real arithmetic in compiled code — so the more elements you have, the bigger the win.
Case study — PowerPy's thermal solver (Chapter 2 §2)

Here is the win on real PowerPy code. A solar array has many cells (call it N), and the thermal model must find each cell's equilibrium temperature — the temperature where heat in equals heat out, which is the root of a nonlinear energy-balance equation.

Case study — before and after.
Legacy — one solver call per cell, in a Python loop. The old code calls scipy.fsolve (a general-purpose root finder) once for every single cell, inside a Python for loop. Each call spins up the solver afresh, iterates to a root for that one cell, and tears down — then the loop moves to the next cell and does it all again. For N cells that is N independent solver spin-ups and N trips round the slow Python loop.
New — one vectorised Newton step over ALL cells. The new solver treats every cell's temperature as one entry in a numpy array and applies Newton's method to the whole array at once: each iteration computes the residual and its derivative for all N cells as array operations, and updates all N temperatures in a single vectorised step. A handful of iterations converge every cell together. The entire solve is roughly ~6 array operations — no per-cell loop, no per-cell solver spin-up.
The shape of the change. The legacy version turns one physical problem into N solver start-ups plus N Python iterations; the new version turns it into about six array operations that run in compiled C regardless of how many cells there are — so the cost stops scaling with the number of cells. ✓
Why this is the textbook vectorisation case. The per-cell calculations are independent — cell 12's equilibrium temperature does not depend on cell 13's — so there is nothing forcing them to happen one after another. That independence is exactly the condition that lets you stack all the cells into an array and solve them in one vectorised sweep. (Hold onto this: when the per-element work is not independent, vectorisation across that dimension is not available — we return to it in §4.10 and the final pitfall.)
An order-of-magnitude timing sketch

Let us put rough numbers on it. These are illustrative order-of-magnitude estimates to show the shape of the win, not measured benchmarks.

Suppose one fsolve call costs about 1 ms. For an array of N = 600 cells:
  legacy thermal work per run = 600 cells × 1 ms/cell = 600 ms ≈ 0.6 s per run
  vectorised thermal work per run ≈ 1–2 ms total (a few array operations, independent of cell count)
where "per run" means one full thermal solve of the whole 600-cell array; 1 ms/fsolve and ~2 ms/vectorised-solve are stand-in figures chosen to illustrate the order of magnitude.

That is a speed-up in the ~100× class for a single thermal solve (≈600 ms collapsing to a couple of ms). And it compounds: the reliability study in Chapter 2 runs the whole thing inside a Monte-Carlo sweep of, say, 1000 runs. The thermal work alone then goes from 1000 × 0.6 s = 600 s (ten minutes) down to 1000 × 2 ms ≈ 2 s. Same physics, same answer — the wait drops from "go get a coffee" to "already done." (We compute the exact ratio in Worked Example (b).)

4.4   Avoid redundant work (1): caching and memoisation

The fastest computation is the one you never do. A large class of speed-ups comes not from doing work faster but from not repeating work you have already done. The headline technique is caching.

Memoisation (caching). Storing the result of a computation the first time you do it, keyed by its inputs, and returning the stored result the next time the same inputs come in — instead of recomputing. "Caching" is the general idea; "memoisation" is the specific case of caching a function's return value keyed by its arguments. The trade is memory for time: you spend a little space holding past results to avoid spending time redoing them.

Python makes function memoisation almost free with a decorator from the standard library:

from functools import lru_cache

@lru_cache(maxsize=None)
def cell_iv_curve(area, isc_ref, temperature):
    # expensive: builds and solves the cell's subcircuit
    ...
    return curve

# First call with these args: computes and stores the curve.
# Any later call with the SAME (area, isc_ref, temperature): returns
# the stored curve instantly, no recomputation.

lru_cache ("Least Recently Used cache") remembers recent calls and serves repeats from memory. It only works when the function is pure in the §3 sense — same inputs must always give the same output — because that is exactly what makes a stored result safe to reuse.

PowerPy already caches by content. PowerPy hashes a cell's parameters into an md5 digest (a short fixed-length fingerprint of the argument values) and uses that as the cell's name. The effect: two cells with identical parameters hash to the same name, so the corresponding subcircuit is recognised as already built and is not constructed a second time. That is caching — keyed by the content of the parameters rather than by call order. In an array where many cells share the same specification, this quietly avoids rebuilding the same subcircuit hundreds of times.

4.5   Avoid redundant work (2): hoisting, copying cost, and batching

Hoist invariant work out of loops

If something inside a loop produces the same value every iteration, computing it inside the loop wastes that work N times over. Hoisting means lifting that loop-invariant computation out of the loop, computing it once, and reusing it.

# SLOW — recomputes a constant factor on every one of N iterations
for cell in cells:
    factor = boltzmann * t_ref / charge      # identical every pass!
    cell.adjust(factor)

# FAST — hoist the invariant out; compute it ONCE
factor = boltzmann * t_ref / charge          # once, before the loop
for cell in cells:
    cell.adjust(factor)

The arithmetic of factor happens once instead of N times. The rule generalises: anything inside a loop that does not depend on the loop variable should usually live outside it.

Copying cost

Copying data is not free, and copying big data repeatedly can quietly dominate a runtime. A deep copy duplicates an object and everything it refers to, all the way down — so a deep copy of a large nested configuration allocates and fills a whole second structure.

PowerPy's deep-copy habit. PowerPy frequently deep-copies objects by serialising them to JSON and parsing them back — a convenient way to get a fully independent copy. For a small configuration that is perfectly fine; the cost is negligible. But the same trick becomes expensive if it is done per cell, per iteration inside a hot loop: serialising and re-parsing a structure thousands of times adds up fast. The guideline: copy only what actually changes. If a loop mutates one field of a big object, copy that field — not the entire object — each pass.
Expensive external calls — batch and amortise

The most expensive thing PowerPy does per call is talk to ngspice, an external circuit simulator. Each invocation of the wrapper ng_sim creates a fresh ngspice engine instance and destroys it again when done. Spinning an engine up and tearing it down has a fixed cost that has nothing to do with the actual circuit — and if you pay that cost per cell, per voltage point, it utterly dominates.

Batching (and amortising). Batching means grouping many small operations into one larger operation so a fixed per-operation overhead is paid once instead of once-per-item. Amortising means spreading that fixed cost across all the items it now serves, so the cost per item becomes small. Pay a setup cost once and use it for 600 things, and the amortised setup cost per thing is 1/600th of what it was.

Applied to ngspice: instead of one .op (operating-point) call per voltage per cell — thousands of engine spin-ups — solve a whole circuit or a whole IV sweep in a single ngspice run, and reuse the engine across points where you can. The engine start-up is then paid once and amortised over the entire sweep, instead of being re-paid thousands of times.

Mini worked figure — amortising engine start-up.
Say an ngspice engine takes ~50 ms to spin up and tear down, and the actual solve of one operating point is ~1 ms.
per-point approach (engine per .op): (50 + 1) ms × 100 points = 5100 ms
batched approach (one engine, 100 points in it): 50 ms + (100 × 1 ms) = 150 ms
where 50 ms is the fixed engine setup, 1 ms is one operating-point solve, 100 is the number of voltage points in the sweep.
The fixed 50 ms is paid 100 times in the first case (5000 ms of pure overhead) and once in the second. Batching cuts the sweep from ~5.1 s to ~0.15 s — roughly 34× — entirely by amortising a fixed cost. ✓

4.6   Data structures and memory

The container you choose for numeric data affects both speed and memory. The key contrast is a Python list of floats versus a numpy array of floats.

  • A Python list of floats is a list of pointers to individual float objects scattered in memory; each float carries object overhead, and iterating touches them one boxed object at a time. Flexible, but heavy and cache-unfriendly for numbers.
  • A numpy array of floats is one contiguous, typed block of raw 8-byte values — no per-element object overhead, laid out so the CPU can stream through it, and the substrate that makes vectorisation (§4.3) possible.

So for numeric work, prefer numpy arrays: they are smaller in memory and faster to compute on. Memory matters in its own right for the big result tables a Monte-Carlo sweep produces — thousands of runs × many quantities per run. Two levers keep that under control: compact dtypes (store values in the smallest type that suffices — e.g. float32 instead of float64 halves the table when full double precision is not needed) and columnar storage (store each quantity as its own contiguous column rather than as rows of mixed-type records, which is both more compact and faster to scan one quantity at a time). This links directly to Chapter 2 §7's handling of Monte-Carlo result tables.

4.7   Parallelism (briefly)

Some work can be split across several CPU cores and done at the same time. Monte-Carlo is the ideal case for it.

Embarrassingly parallel. A workload made of pieces that are completely independent — none needs any result from another — so they can run simultaneously on different cores with essentially no coordination. The 1000 runs of a Monte-Carlo sweep are embarrassingly parallel: each run draws its own random sample and computes its own result, sharing nothing, so you can hand run 1 to one core, run 2 to another, and so on. With C cores you approach a C× ("near-linear") speed-up.

In Python you reach for joblib (a thin, friendly wrapper, popular in scientific code) or the standard-library multiprocessing to spread independent runs across cores. But there is a Python-specific catch you must understand to choose the right tool.

GIL — Global Interpreter Lock. A lock inside CPython (the standard Python interpreter) that allows only one thread to execute Python bytecode at a time, even on a multi-core machine. It exists to keep the interpreter's internals safe, but its consequence for performance is blunt: multiple Python threads cannot run Python computation truly in parallel — they take turns holding the lock. So threads do not speed up CPU-bound Python work.
Processes, not threads, for CPU-bound work. Because the GIL serialises threads, the way to get real parallelism for computation is to use multiple processes — each a separate Python interpreter with its own GIL — so they genuinely run at once on separate cores. joblib and multiprocessing default to processes for exactly this reason. (Threads still help for I/O-bound work — waiting on disk or network — because a thread waiting on I/O releases the GIL. But a Monte-Carlo sweep is CPU-bound, so it wants processes.)

4.8   When NOT to optimise

Everything above is about how to make code fast. This subsection is about when not to bother — which is most of the time.

Premature optimisation. Making code more complicated in the name of speed before you have measured that the speed is actually needed there. It is the habit of rewriting clear code into clever code on a hunch — and it almost always trades real, permanent readability for imaginary or trivial performance, because the "optimised" spot was usually not the bottleneck at all.
Optimise the proven hot spot, and nothing else. Clear code has a real, ongoing value: it is easier to trust, debug, and change (§3). Speed has value only where it is measurably lacking. So the rule closes the loop with §4.1: profile first, find the one function that genuinely costs the time, optimise that one — accepting some added complexity there because it buys a measured win — and leave the other 95% of the code clear and simple. Optimising code that is not hot makes the program harder to read for no benefit you can measure. The hot spot earns its complexity; nothing else does.

4.9   Worked example (a) — Big-O and parallel wall-clock EXAM-HOT

Worked example (a) — a 600-position fault sweep. You want to sweep a single-cell fault through every position in a 600-cell array: for each position, mark that one cell failed and run the full simulation. Each full run takes 0.5 s. Question: what is the wall-clock time on 1 core, and on 8 cores?
Identify the complexity. One run per position, 600 positions, runs independent of one another → the sweep is O(N) with N = 600. (Contrast the impossible O(2^N) of enumerating all failure combinations in §4.2 — here we change one cell at a time, not every subset.)
Single-core wall-clock. The runs happen one after another:
T₁ = N × (time per run) = 600 runs × 0.5 s/run = 300 s (= 5 minutes)
where N = 600 positions, 0.5 s = duration of one full simulation run, and "× 1" because one core does them sequentially.
Eight-core wall-clock. The 600 runs are embarrassingly parallel (each position is independent), so 8 cores share the load near-linearly:
T₈ ≈ T₁ / cores = 300 s / 8 = 37.5 s ≈ 38 s
where 300 s is the single-core total and 8 is the number of cores; the ÷8 assumes ideal near-linear scaling.
Check and caveat. 600 × 0.5 s = 300 s on one core; spread over 8 independent workers, ≈ 38 s — an ~8× reduction, which is the near-linear speed-up an embarrassingly-parallel workload allows. The 38 s is an idealised floor: real scaling is slightly worse because of process start-up and result-gathering overhead, so expect a little above 38 s, not below. Because the runs are CPU-bound, this uses 8 processes (not threads — the GIL would serialise threads). ✓

4.11   Socratic ladder — why does vectorising the loop make it ~100× faster?

Q·a (foundation). When you run a plain Python for loop for i in range(n): out[i] = a[i] + b[i], what does the interpreter actually do on every single iteration, beyond the addition itself?
On each pass it does a pile of bookkeeping around the one add: it advances the loop and bounds-checks i; it indexes a and b, which means following pointers to two Python float objects; it checks the type of each object and unboxes the raw double out of it; it performs the add; it boxes the result into a brand-new Python float object; and it stores that back into out. The arithmetic is one machine instruction; everything around it — the type checks, the boxing and unboxing, the loop machinery — is many, and it repeats every iteration.
Q·b (name it). That surrounding bookkeeping has a name — what is it, and how does its cost compare to the arithmetic?
It is interpreter (per-iteration) overhead: the fixed cost the Python interpreter pays to execute each step of a loop in a dynamically-typed language where every value is a boxed object of unknown type until checked. For simple numeric work this overhead is typically larger than the arithmetic it surrounds — often by an order of magnitude — so a Python numeric loop spends most of its time not doing the maths, but managing objects and the loop.
Q·c (the shift). When you replace the loop with one numpy operation out = a + b, what work gets pushed out of the interpreter and into compiled code, and why does that kill the overhead?
numpy knows in advance that a and b are contiguous, typed arrays — say, blocks of 8-byte doubles all of the same type. So it can run a single tight loop in compiled C that strides straight through the raw memory adding doubles, with no per-element type check and no boxing/unboxing: the values are already raw doubles, not Python objects. The per-iteration overhead that Python paid N times is paid essentially once, at the boundary, for the whole array. Same additions; the interpreter bookkeeping around each one is gone.
Q·d (why it grows with N). Why does the speed-up get bigger as the array gets larger, rather than staying a fixed percentage?
Because the Python loop's cost is roughly N × (overhead + arithmetic) while the vectorised cost is roughly (fixed setup) + N × (arithmetic). The expensive overhead term scales with N in the loop but is paid once in the vectorised version. So as N grows, the loop keeps re-paying the overhead while numpy does not, and the ratio between them widens. At N = 600 cells (the case study) the loop has paid the per-element tax 600 times; the vectorised solve, essentially zero times.
Q·e (the boundary — the target). Put it together: why does rewriting a per-cell Python loop as one numpy array operation make it ~100× faster when it does the same arithmetic? And what kind of computation would not vectorise this way?
The speed-up: the arithmetic is identical, but the Python loop wraps every element in interpreter overhead — type checks, boxing/unboxing, loop machinery (Q·a–b) — and pays it N times, whereas the numpy operation pushes the loop into compiled C over a typed contiguous block and pays that overhead essentially once (Q·c). Since the dominant cost was the overhead and it no longer scales with N, the larger the array the larger the win (Q·d) — for PowerPy's 600 independent cells, a ~100×-class collapse from ~600 ms to a couple of ms. What would NOT vectorise: a computation with a genuine per-element sequential dependency — where computing element i requires the result of element i−1. You cannot do all elements "at once" if each one needs the previous one's answer first. PowerPy shows both faces of this: the thermal solve vectorises across cells because the cells are independent, but the Newton iterations within the solve are inherently sequential (each step uses the previous step's estimate), as is the coupling fixed-point loop between the electrical and thermal models. You vectorise the independent dimension (cells) and accept that the dependent one (iteration steps) stays a loop. Vectorisation removes the interpreter tax wherever the work is independent; where the work is truly sequential, that tax cannot be removed by vectorising and you need a different approach.

4.12   Pitfalls

Pitfall 1 — optimising before profiling. If you optimise on a hunch instead of a measurement, you will almost certainly speed up the wrong thing: you pour effort into a function that turns out to be 3% of the runtime while the real 80% hot spot sits untouched, and the program is no faster — but now it is also harder to read. The fix is the golden rule (§4.1): run cProfile first, read the top of the table, and optimise only the function that the measurement proves is hot. Don't guess, profile.
Pitfall 2 — "vectorising" something with a genuine sequential dependency. Vectorisation only works when the per-element computations are independent. If element i truly depends on element i−1 — an iterative solver step that needs the previous estimate, a running cumulative total, a fixed-point loop that feeds its own output back as input — you cannot express it as one array operation that does everyone "at once," because the inputs for later elements do not exist yet. Forcing it produces wrong answers (silently using stale values) or simply cannot be written. The honest move is to recognise the dependency and either keep that part as a loop, or reach for a method designed for sequential structure (e.g. numpy.cumsum for cumulative sums, or a vectorised iteration that steps all elements forward together — as PowerPy's Newton solver does across cells while still iterating sequentially in time).

5Testing, Reproducibility, and Trusting Your Results

5   Testing, reproducibility, and trusting your results

Everything so far has been about writing code that is clear and fast. This section is about the question that outranks both: is it right? A simulator that crashes is annoying; a simulator that quietly returns the wrong number is dangerous, because nothing warns you. You run it, the output looks plausible — a panel temperature of 71 °C, a bus current of 4.2 A — you drop it in a report, and you have just shipped a wrong answer with full confidence. The defence is not "read the code more carefully" (you will still fool yourself). The defence is to make the machine prove the code works, on cases whose answers you already know, automatically, every time the code changes.

The one idea. A scientific result you have not tested is a result you only hope is correct. Testing turns "I think it works" into "I can show it works, repeatably." You do this by feeding each piece of code an input whose correct output you know independently — from a trusted reference, a hand calculation, or a physical bound — and asserting that the code reproduces it. Pin those checks in a tests/ folder, run them with pytest on every change, and fix the random seed so a stochastic run repeats exactly. The payoff is enormous: a silent numerical bug, the worst kind, becomes a loud, located test failure instead of a plausible lie in your plot.

We will build the vocabulary first — the four kinds of test, then oracle, fixture, deterministic, continuous integration — then make each concrete for PowerPy (a thermal parity test against the legacy fsolve, a hand-worked energy-balance oracle, a 2×2 circuit you can check by hand before trusting 600 cells, and cheap physical-sanity bounds). Then a fully narrated pytest example, a Socratic ladder, the classic pitfalls, and woven practice.

5.1   Why a scientific code needs tests at all

In ordinary software a bug usually announces itself: the button does nothing, the page 500s, the program throws. In numerical software the most damaging bugs do the opposite — they succeed. A sign flipped in a heat-balance term, a Jacobian transposed, an index off by one, kelvin fed where celsius was meant: none of these crash. Each produces a number, and the number is in the right ballpark, so it looks like an answer. You cannot tell a correct 65 °C from a buggy 65 °C by looking at it.

That is the whole reason testing matters more here than almost anywhere. A test is an independent statement of what the answer should be, written down once and checked forever after. It is the difference between trusting your code because it ran, and trusting it because you can demonstrate, on command, that it reproduces answers you obtained some other way.

The silent-bug threat model. Ask of every result: "if this were wrong by 10%, would anything have told me?" If the honest answer is "no" — no crash, no warning, no check — then you are not verifying that number, you are hoping about it. A test is how you convert a hope into a verification.

5.2   The four kinds of test

"Testing" is not one activity. There are four distinct kinds, each catching a different class of error. Learn the name and the job of each.

Unit test. A test that checks one small piece of code — usually a single function — against a known expected output. You give it inputs, you assert the result equals (or is close to) the value you worked out independently. It is the smallest, fastest, most numerous kind of test. In Python you write them with pytest: a plain function named test_<something> containing an assert. 
# a unit test: one function, one known answer
def test_compute_isc_at_reference():
    cell = make_reference_cell()       # isc_ref = 0.50 A at 25 °C
    assert compute_isc(cell, 25.0) == 0.50    # at t_ref, isc == isc_ref exactly
Integration test. A test that checks several pieces working together, end to end, rather than in isolation. A unit test proves build_circuit and solve_circuit each work alone; an integration test proves that building a small circuit and then solving it produces the right node voltages — that the pieces connect correctly. It catches the bugs that live in the seams between functions, which no single unit test can see.
Regression test. A test that pins the current, known-correct output so that a future change cannot silently alter it. You capture today's right answer as the expected value; if a later edit changes it, the test fails and forces you to notice. The name says it: it guards against a regression — a change that makes previously-correct behaviour wrong. ("I refactored the Jacobian and the converged temperature moved by 4 °C" should be a red test, not a surprise in next month's plot.)
Parity test. A test that checks a new implementation agrees with a trusted existing one on the same input, to within a tolerance. When you rewrite a slow-but-correct routine as a fast one, the old routine becomes your reference: feed both the identical input and assert the outputs match. Parity testing is exactly how you gain confidence that an optimisation (Section 4's vectorised thermal solve) did not change the answer, only the speed.
The testing pyramid. A healthy test suite is shaped like a pyramid: a broad base of many fast unit tests (cheap, run constantly), a thinner middle of integration tests, and a few slow end-to-end tests at the apex. You want most of your safety net to be fast unit tests because they run in milliseconds and you run them all the time. Reserve the heavy end-to-end runs for the handful of checks that genuinely need the whole system.

5.3   The oracle: where the "expected answer" comes from

Every test asserts that the code's output equals an expected value. That raises the obvious question: where does the expected value come from, and why do you trust it? That trusted source has a name.

Oracle. A trusted source of the correct answer for a given input — the thing your code's output is checked against. An oracle can be (a) a reference implementation you already trust (the legacy fsolve solver), (b) a hand calculation you did on paper (a single-cell energy balance worked out by hand), (c) an analytic closed-form solution for a special case, or (d) a known physical bound the answer must respect. The art of testing a numerical code is largely the art of finding cheap, trustworthy oracles.

The four kinds of test and the oracle interlock: a parity test uses a reference implementation as its oracle; a unit test often uses a hand or analytic oracle; a sanity check uses a physical-bound oracle. Keep the word in mind — when someone asks "how do you know that's the right answer?", the answer is always "because my oracle says so, and here is why I trust the oracle."

5.4   Practical tests for PowerPy

Abstractions land only when made concrete, so here are the exact tests that would guard PowerPy's thermal and electrical core — the code from Chapter 2.

(a) Parity: the vectorised solver vs the legacy fsolve. Section 4 replaced the per-cell scipy.optimize.fsolve loop with a single vectorised Newton step over the whole array. That is a performance change, and a performance change must not move the answer. The parity test makes that explicit: take the worked-example inputs from Chapter 2, run the old per-cell solver and the new vectorised solver on the same single cell, and assert they agree. Both must give a front-face temperature of 65.26 °C (the converged value from worked example 2.1). The assertion idea:

t_ref = legacy_fsolve_single_cell(cell)     # trusted reference (the oracle)
t_new = vectorised_solver(cell)             # the new implementation
assert abs(t_new - t_ref) < 1e-3            # agree to a millikelvin

If that passes, you have shown the optimisation preserved the physics to a millikelvin. If it fails, you caught a regression the instant you introduced it — not three plots later.

(b) Hand-worked oracle, when the reference itself is unavailable. Parity assumes you have a trusted reference. Sometimes you do not — the legacy code is broken, missing, or itself under suspicion. Then you fall back to an oracle you can compute by hand: the single-cell energy balance from Chapter 2. For one cell at steady state the front and rear nodes must each have net-zero heat flow; solving that 2-node balance by hand gives front = 65.26 °C, rear = 64.60 °C. That paper answer is now your oracle, and the new solver is checked against it:

# oracle = the hand-computed Chapter-2 single-cell balance
assert math.isclose(t_front_new, 65.26, abs_tol=0.01)
assert math.isclose(t_rear_new,  64.60, abs_tol=0.01)

This is strictly stronger than parity in one way: a parity test only proves the new code matches the old code — if the old code was also wrong, they agree on a wrong answer. A hand oracle proves the code matches physics.

(c) Small-circuit sanity: a 2×2 before a 600-cell array. Before you trust ngspice on a 600-cell array whose internals you can never check by hand, trust it on a 2×2 cell circuit you can. With four cells you can reason about every node voltage and branch current on paper — series cells add voltage, parallel strings add current — and then read the per-cell V and I back from ngspice and confirm each matches your hand reasoning. Only once the 2×2 reads back correctly do you scale up. The small case is where bugs in wiring, indexing, and reading results are cheap to find; the 600-cell case is where they are impossible to find.

(d) Physical-sanity checks. The cheapest tests of all assert that a result lands inside a physically plausible band — no exact oracle needed, just physics. A sunlit panel temperature should sit somewhere in, say, 0–150 °C; a cell current in forward bias should be non-negative; output power should never exceed the incident power. These are one-line asserts that cost nothing and catch gross errors instantly — a temperature of −800 °C or 4000 °C, a negative current, a panel that outputs more than it absorbs all scream a bug.

assert 0.0 <= t_panel <= 150.0,  f"panel temp {t_panel} °C is unphysical"
assert current >= 0.0,           f"forward-bias current {current} A is negative"
assert p_out <= p_in,            f"output {p_out} W exceeds input {p_in} W"
Sanity checks vs exact checks. A physical-sanity bound is weak (a buggy 64 °C still passes "0–150 °C") but cheap and broad — it guards every result for free. An oracle check is strong (it pins the exact value) but narrow — it only covers the one case you computed. Use both: sanity bounds everywhere as a coarse net, exact oracles on the few cases you can pin precisely.

5.5   Floating point: never compare with ==

There is a trap that bites every numerical-testing beginner exactly once: comparing two floating-point numbers with ==. It does not work, and the reason is fundamental, not a quirk you can wish away.

Floating-point rounding. Computers store real numbers in a finite number of bits, so most values are kept only approximately. Numbers like 0.1 have no exact binary representation, and every arithmetic operation rounds its result to the nearest representable value. Tiny rounding errors therefore accumulate, so two calculations that are mathematically equal can differ in their last bits. The classic shock: 0.1 + 0.2 == 0.3 is False in Python, because 0.1 + 0.2 stores as 0.30000000000000004.

The consequence for testing: assert t_new == 65.26 is almost guaranteed to fail even when the answer is correct, because the solver's last bits will differ from the literal. The fix is to compare within a tolerance — assert the two values are close enough, not bit-identical.

  • One scalar: math.isclose(a, b, abs_tol=1e-3) — true when a and b differ by less than the tolerance. (Use abs_tol for an absolute window near zero; rel_tol for a relative one on large values.)
  • A whole array (PowerPy's natural case — temperatures over all N cells): numpy.allclose(arr_new, arr_ref, atol=1e-3) — true when every element is within tolerance. This is the vectorised cousin of isclose, and exactly what you want for checking a 600-element temperature vector in one line.
import math, numpy as np

math.isclose(0.1 + 0.2, 0.3)                 # True  — within default tolerance
(0.1 + 0.2) == 0.3                           # False — the == trap

np.allclose(t_new_array, t_ref_array, atol=1e-3)   # all N cells agree?  one bool
Pitfall — comparing floats with ==. assert result == 65.26 or if voltage == 0.3: will fail (or branch wrong) on values that are correct to twelve decimal places, because rounding makes the last bits differ. Never use == (or !=) on computed floats. Always compare with a tolerance: math.isclose for a scalar, numpy.allclose for an array. The only floats you may safely == are ones you never did arithmetic on (a literal you stored and read back unchanged).

5.6   Reproducibility: fix the seed, record the run

Chapter 2's Monte-Carlo hot-spot study draws random numbers. That creates a problem for testing and for science alike: if every run draws different randoms, every run gives a different answer, and you can neither test it (the expected value moves) nor reproduce a result you reported last week. The fix rests on one concept.

Deterministic. A computation is deterministic if the same inputs always produce exactly the same outputs — no run-to-run variation. Ordinary arithmetic is deterministic; anything that draws "random" numbers is not, unless you pin the randomness. A pseudo-random generator is actually a deterministic sequence controlled by a starting value called the seed: fix the seed and the "random" draws are identical every run, which makes the whole Monte-Carlo computation deterministic and therefore reproducible and testable.

import numpy as np
rng = np.random.default_rng(seed=12345)   # fix the seed
samples = rng.normal(size=1000)           # identical sequence on every run

With the seed fixed, a Monte-Carlo run repeats exactly, so you can regression-test its output and anyone can reproduce your figure. But a seed alone is not enough for science: you must also record the seed and the parameters alongside the results. A result file that does not say which seed and which inputs produced it is not reproducible, however deterministic the code was — the determinism is wasted if no one knows the starting conditions. (This ties directly to Chapter 2 §6–§7, where the run's seed and parameter set are logged with the output.)

Reproducible = deterministic code + recorded conditions. Fixing the seed makes the code deterministic (same inputs → same outputs). Recording the seed and parameters with the results makes the run reproducible (someone else, later, can recreate the exact inputs). You need both. A determinism you don't record, and a record without determinism, each fail to let a future reader reproduce your number.

5.7   How tests are organised and run

Tests are not scattered through the code; they live in a conventional place and are run by a tool.

  • A tests/ folder. By convention, test files live in a top-level tests/ directory, named test_*.py, mirroring the modules they cover (test_thermal.py tests the thermal solver). Keeping them separate from the source keeps the shipped code clean while keeping the safety net close.
  • Small, fast tests. Each test should check one thing and run quickly. Fast tests get run often (you will run a 200 ms suite after every edit; you will avoid a 10 minute one), and a test that runs often is a test that actually catches bugs.
  • pytest to run them. pytest is the standard Python test runner: point it at the folder and it discovers every test_* function, runs them, and reports passes and failures with the offending values. You write plain functions with assert; pytest does the rest.
Fixture. A reusable piece of setup shared by many tests — a pre-built input object the tests need. In pytest you write a function decorated with @pytest.fixture that builds, say, a reference solar cell, and any test that lists the fixture's name as a parameter receives that ready-made object. Fixtures keep tests DRY (Don't Repeat Yourself — you build the reference cell once, not in every test) and make each test read as just "given this input, assert this output". 
import pytest

@pytest.fixture
def reference_cell():
    """A single cell with the Chapter-2 worked-example parameters."""
    return make_cell(isc_ref=0.50, alpha_isc=0.0005, t_ref=25.0, area=4e-4)

def test_isc_at_reference(reference_cell):       # fixture injected by name
    assert compute_isc(reference_cell, 25.0) == 0.50
Continuous Integration (CI). Continuous Integration — the practice of running your test suite automatically on every change (every push or merge), on a server, so broken code is caught the moment it is introduced rather than discovered later. You configure it once; thereafter every change is greeted by the full suite, and a red result blocks the change. CI is what turns "we have tests" into "the tests are always run" — the difference between a safety net you own and one you actually use.

5.8   Assertions and defensive checks in the code itself

Tests live outside the code and run on demand. But you can also place checks inside the code that fire during every real run, guarding against bad inputs and broken invariants. The goal is the same as in Section 3: make a failure loud and located rather than a silent NaN that propagates.

The sharpest PowerPy example comes straight from Chapter 2's vectorised solver. The global thermal Jacobian there is block-diagonal: N independent 2×2 blocks, one per cell. Each Newton step inverts those 2×2 blocks, and a 2×2 inverse divides by the determinant det = ad − bc. If a block's determinant is (near) zero — a degenerate cell, a bad parameter — that division produces inf or NaN, which then silently poisons every downstream temperature. A defensive check catches it at the source, where the message can still name the cause:

# inside the Newton step, before dividing by the 2×2 determinant
det = a * d - b * c
if np.any(np.abs(det) < 1e-12):
    bad = np.where(np.abs(det) < 1e-12)[0]
    raise ValueError(f"singular 2x2 Jacobian block(s) at cells {bad}: det≈0")
delta = solve_2x2(a, b, c, d, residual)     # safe: no block is singular

Without the guard, a singular block yields a NaN temperature that flows on through the electro-thermal loop and surfaces — if at all — as a baffling "the array won't converge" three layers away. With the guard, you get a precise message naming the exact cells, at the exact line. Validate inputs and invariants early, so failures are loud and located, not silent and far away.

5.9   Worked example — a narrated pytest parity + sanity test EXAM-HOT

Worked example — one pytest file that parity-checks the thermal solve and asserts a physical bound. We build the shared input as a fixture, call both solvers, and assert (1) they agree to a tolerance and (2) the result is physically plausible. Every line is narrated below.
The test file (tests/test_thermal_parity.py): 
import math
import pytest
from powerpy.thermal import legacy_fsolve_single_cell, vectorised_solver
from powerpy.cells import make_cell

@pytest.fixture

    """The single cell from Chapter-2 worked example 2.1."""
    return make_cell(isc_ref=0.50, alpha_isc=0.0005, t_ref=25.0, area=4e-4)

def test_vectorised_matches_legacy(worked_example_cell):
    t_ref = legacy_fsolve_single_cell(worked_example_cell)   # trusted oracle
    t_new = vectorised_solver(worked_example_cell)           # new implementation
    assert math.isclose(t_new, t_ref, abs_tol=1e-3)          # parity to a mK

def test_panel_temperature_is_physical(worked_example_cell):
    t_new = vectorised_solver(worked_example_cell)
    assert 0.0 <= t_new <= 150.0      # a sunlit panel must land in a sane band
Line by line.
  • import math, pytest — math.isclose for the tolerance comparison; pytest for the fixture decorator and to run the file.
  • the imports from powerpy pull in the two solvers under test and the cell builder — the things whose correctness we are checking.
  • @pytest.fixture / def worked_example_cell() — builds the shared input once: a cell carrying the exact parameters of Chapter 2's worked example. Any test that names worked_example_cell as a parameter is handed this object, so we never rebuild it.
  • def test_vectorised_matches_legacy(worked_example_cell) — pytest sees the test_ prefix and runs it; it sees the parameter name matches the fixture and injects the cell.
  • t_ref = legacy_fsolve_single_cell(...) — run the trusted legacy solver; its output is our oracle (≈ 65.26 °C).
  • t_new = vectorised_solver(...) — run the new vectorised solver on the identical input.
  • assert math.isclose(t_new, t_ref, abs_tol=1e-3) — the heart of the parity test: the two must agree to within a millikelvin. Note isclose, not == — float rounding would break ==.
  • the second test, test_panel_temperature_is_physical, is the cheap sanity net: whatever the exact value, a sunlit panel must lie in 0–150 °C, so a gross bug (a negative or four-figure temperature) fails loudly even if no oracle were available.
Running it. pytest tests/test_thermal_parity.py discovers both test_ functions, injects the fixture, runs them, and prints 2 passed — or, on a regression, shows the two values it compared and the line that failed. Two small, fast tests now stand permanently between a future edit and a silent change to the converged temperature. ✓

5.10   Socratic ladder — trusting a solver with no perfect reference

Q·a (foundation). What is a unit test, in one sentence, and what does it need to exist?
A unit test checks one small piece of code — typically a single function — against a known expected output: call the function with chosen inputs and assert the result matches the value you obtained independently. It needs exactly one thing to exist: an expected answer you trust. No expected answer, no test — which is why the next rung matters.
Q·b (the source of truth). That "expected answer" — what is it called, and what can supply it when there is no trusted reference solver to copy from?
It is the oracle: a trusted source of the correct answer. When no reference implementation exists, the oracle can instead be a hand calculation, an analytic solution for a special case, or a physical bound the answer must respect. The reference solver is only one possible oracle; losing it does not leave you blind, it just sends you to the other three.
Q·c (a hand-computable case). For PowerPy's 2-node thermal balance, what concrete case can you actually compute by hand to serve as that oracle?
A single cell at steady state. With two nodes (front, rear) and the requirement that each has net-zero heat flow, the balance is a small system you can solve on paper — and Chapter 2 does, getting front = 65.26 °C, rear = 64.60 °C. That hand answer becomes the oracle: assert math.isclose(t_front_new, 65.26, abs_tol=0.01). It checks the new solver against physics, not against possibly-buggy old code.
Q·d (cheap broad coverage). The hand oracle covers one case. What do physical-sanity bounds add that an exact oracle does not?
Breadth, for free. A bound like 0 ≤ T ≤ 150 °C, or current ≥ 0 in forward bias, or power ≤ input, holds for every result, not just the one you hand-computed. It is weak (a wrong-but-plausible 64 °C slips through) but it costs one line and catches every gross error — a −800 °C temperature, a negative current, a panel emitting more than it absorbs — across the whole simulation. Coarse net everywhere, sharp oracle on a few points.
Q·e (scaling confidence). Why build a 2×2-cell circuit and check it by hand before trusting a 600-cell array?
Because the 2×2 is hand-checkable and the 600-cell is not. With four cells you can reason out every node voltage and branch current on paper and confirm the per-cell V/I read back from ngspice match — catching wiring, indexing, and result-reading bugs while they are cheap to find. Those same bugs in a 600-cell array are invisible (no one can hand-check 600 cells). Get the small case provably right first, then scale, so the only new thing in the big run is size, not correctness.
Q·f (the target). So: if we have no perfectly-trusted reference implementation, how do we still know the new solver is correct?
Stack the rungs. We do not need a reference implementation because the oracle can be a hand calculation (Q·b): the single-cell 2-node balance gives 65.26/64.60 °C by hand (Q·c), and a unit test (Q·a) pins the solver to it with math.isclose. Around that exact check we throw physical-sanity bounds (Q·d) that cheaply guard every other result, and we earn confidence in the wiring on a 2×2 circuit we can check by hand (Q·e) before scaling to 600 cells. Correctness without a trusted twin comes from anchoring to physics you can compute, bounding everything you cannot, and proving the small case before trusting the large one. If a reference later appears, a parity test (abs(t_new − t_ref) < 1e-3) is added on top — but it was never the only road to trust.

5.11   Pitfalls

Pitfall 1 — comparing floats with ==. (Restated because it is the one everyone hits.) assert t_new == 65.26 fails on a correct answer because rounding makes the last bits differ. Use a tolerance: math.isclose(t_new, 65.26, abs_tol=1e-2) for a scalar, numpy.allclose(...) for an array. Reserve == for integers and for floats you never did arithmetic on.
Pitfall 2 — a test that never actually asserts (it passes vacuously). A test with no reachable assert always "passes" — and proves nothing. The classic forms: 
def test_solver():
    t = vectorised_solver(cell)      # computes a value...
    # ...and never checks it. Passes no matter how wrong t is.

def test_close():
    math.isclose(t_new, t_ref)       # returns True/False but it is NOT asserted!
The first never checks its result; the second computes a boolean and throws it away (you must write assert math.isclose(...)). A green suite full of vacuous tests is worse than no suite — it is false confidence. Every test must contain at least one assert that can actually fail.
Pitfall 3 — trusting a big simulation you never checked on a small hand-solvable case. Running the 600-cell array and eyeballing "looks about right" is not verification — it is the exact situation where a silent bug hides, because no one can hand-check 600 cells. If you never confirmed the 2×2 circuit, or never pinned the single-cell temperature to the hand oracle, then the big number rests on nothing. Always anchor the large run to a small case whose answer you know.

Section recap. Test a scientific code because its worst bugs are silent — a wrong number that looks plausible. Four kinds of test: unit (one function vs a known output, via pytest + assert), integration (pieces working together end-to-end), regression (pin today's correct output so a future change can't silently move it), parity (new impl vs a trusted one to a tolerance). The expected value comes from an oracle — a trusted reference implementation, a hand calculation, an analytic case, or a physical bound. Organise tests in tests/, keep them small and fast, run with pytest, share setup via fixtures, and run them automatically on every change (Continuous Integration, CI). 6A Concrete Improvement Roadmap for PowerPy

6   A concrete improvement roadmap for PowerPy

The previous five sections were about principles: how to cut a program into parts, how to write a function you can trust, how to test physics, how to find and fix the hot spots. This final section spends that capital. It turns those principles into a single, ordered, actionable plan for this codebase — PowerPy, the satellite solar-array simulator we have been dissecting throughout — and, just as importantly, it argues for the order. A roadmap is not a wish-list. A wish-list says "here are ten good things". A roadmap says "here are ten good things, and here is which one you do first, and why doing them out of order will hurt you". The hard, counter-intuitive lesson at the centre of this section is that the most exciting work — the vectorised solver, the parallel Monte-Carlo, the new physics modules — is the work you do last, and starting with it is the single most common way to waste weeks on a scientific codebase.

The one idea. Make it run and be tested first; then refactor the structure under the protection of those tests; then optimise the hot spots you have actually measured; then build the new features. Each stage protects the next. Refactoring without tests is editing blind — you cannot tell whether you broke the physics. Optimising without profiling is guessing — you speed up code that was never the bottleneck. Building features on a package that does not even import is building on sand. The roadmap below is exactly this order, made concrete for PowerPy, with each item framed as problem → fix → why it matters → rough effort.

6.1   The current state of PowerPy, stated plainly

Before planning repairs, look honestly at what is there. PowerPy is not uniformly good or uniformly bad — it is a two-layer codebase, and the two layers are at very different levels of health. Recognising this split is the whole basis of the plan, because the two layers need opposite things.

The good, modern layer. A set of subpackages — schemas/, loader/, simulation/, render/ — built in the style this course recommends: small modules, @dataclass data shapes, a clear dependency direction (schemas at the bottom, render at the top), and an explicit pipeline that wires the stages together. This layer is a strength. The plan's job for it is mostly to extend it: it is the target shape everything else should grow toward.

The fragile, legacy layer. A set of large, top-level, multi-job classes in flat modules — cell.py, string.py, section.py, panel.py. Several of these files were OCR-reconstructed (recovered from a scan or a PDF rather than from clean source), and that process left real defects behind: a duplicated-self method signature, unbalanced parentheses, stray characters — the kind of damage that stops the package importing at all. This layer is where most of the urgent, unglamorous work lives.

Keep the two layers straight. The legacy top-level string.py (a flat module holding a big String class) is a different thing from the modern simulation/string_level module. When this section says "the legacy monolith classes", it means the flat cell.py / string.py / section.py / panel.py files — not their well-built counterparts inside simulation/. The plan is, in one phrase, to grow the legacy layer toward the modern one — but only after we have made it run and pinned its behaviour with tests.

6.3   Priority 1 — correctness and "make it run"

This tier has one theme: the package must import cleanly. Until it does, every other item is blocked, because you cannot test, refactor, profile, or extend code that will not load. There is no judgement call here about whether P1 comes first — it is a precondition, not a preference.

Fix the OCR/syntax defects. The OCR-reconstructed legacy files contain mechanical damage: a method written def setTemperature(self, self, temperature) (the duplicated self is a SyntaxError), parentheses that do not balance, stray characters dropped mid-line. Each is a hard error that stops import. Work through them until python -c "import powerpy" runs without a traceback. Why it matters: nothing downstream can be trusted — or even attempted — until the code runs. Effort: small-to-medium; mechanical once located, and the interpreter points you at each one in turn.

Rename string.py. Python resolves the legacy top-level string.py before the standard library's string when code runs from inside the package, so any library that imports stdlib string — pandas does, indirectly — gets PowerPy's module instead and breaks in a way whose error message blames pandas. Rename the file (e.g. to cellstring.py) and fix the imports that referred to it. Why it matters: it removes a silent, confusing breakage that can swallow an afternoon, because the symptom appears nowhere near the cause. Effort: small.

Make the package installable. Right now imports work only because you happen to be in the right directory. Add a pyproject.toml declaring the package, then pip install -e . for an editable install (your edits take effect immediately, but the package is importable from anywhere). Why it matters: imports stop depending on the current directory, and the test runner, your editor, and CI can all find the package the same way. Effort: small.

6.4   Priority 2 — reliability

Now the code runs. The next question is: does it run correctly? — and, crucially, can we keep it correct while we change it? That is what tests buy. This tier exists to lay down a safety net before the refactoring of P3, because refactoring is precisely the activity that silently breaks things.

Add a tests/ suite. Start with the three kinds Section 5 argued for: a thermal parity test (the new vectorised path, once it exists, must agree with the trusted loop within tolerance), a 2×2-circuit sanity test (a tiny hand-checkable circuit whose answer you know), and physical-bound asserts (no negative power, efficiency in [0, 1], a cell no hotter than physically possible). Why it matters: it locks in correctness before refactoring — a test that passes now will tell you the instant a later edit changes the answer. Effort: medium.

Kill the bare except: pass. The ngspice wrapper currently catches everything and does nothing — a malformed netlist, a missing binary, a parse error all vanish, and the simulation sails on with stale or empty results. Replace it with a try/except that names the specific exception it can handle, logs it, and re-raises what it cannot. Why it matters: it stops hiding failures — a hidden failure is worse than a crash, because the crash points at the bug and the hidden one produces a plausible wrong number you will plot and believe. Effort: small.

6.5   Priority 3 — structure and clarity

The code runs and is pinned by tests. Only now is it safe to change its shape. This tier moves the legacy layer toward the modern one — and every edit here is protected by the P2 tests, so if a split accidentally alters the physics, a test goes red immediately.

Split the monolith classes by concern. Each legacy class does several jobs at once: reading data, doing parameter maths, generating the netlist, running the simulation, plotting. Separate these into distinct modules in the modern style — one for loading, one for the parameter maths, one for netlist generation, one for simulation, one for plotting. Why it matters: each concern becomes testable and changeable in isolation; a change to the report format no longer risks the physics. Effort: large — this is the biggest structural item, which is exactly why it comes after the safety net and not before.

Replace bare tuple/dict returns with dataclasses. The 6-tuple (model, config, isc, imp, vmp, voc) forces every caller to remember that vmp is element 4 — a mis-index waiting to happen, silent because every element is a float. Replace such returns with a @dataclass whose fields have names, and add type hints to the public functions. Why it matters: fewer mis-index bugs, and the signature becomes self-documenting — the reader and the type-checker both see what comes back. Effort: medium.

6.6   Priority 4 — performance

Clean, tested, well-structured code that is too slow is a real problem — but it is a fourth-order problem, addressed only once the first three tiers are done. The governing rule of this tier is one word: measure. You do not optimise what you guess is slow; you optimise what a profiler shows is slow.

Profile first. Before touching any of the three changes below, run a profiler on a representative workload and read off where the time actually goes. Why it matters: intuition about performance is notoriously wrong; profiling replaces a guess with a measurement, so your effort lands on the real bottleneck. Effort: small — and it pays for itself by stopping you optimising code that was never the problem.

Vectorise the thermal solve (Chapter 2 §2). The cell-by-cell Python loop is the prime suspect for Monte-Carlo cost, because it runs once per cell per sample, thousands of times over. Recast it as NumPy array operations over all cells at once. Why it matters: it is the single biggest speed win for Monte-Carlo. Effort: medium — and note the P2 thermal parity test is what proves the vectorised version still gives the right answer.

Batch ngspice. Today the simulator issues many separate .op calls, paying engine start-up and parse cost each time. Issue one run per circuit / IV sweep and reuse the engine instead of relaunching it. Why it matters: it removes a large, repeated fixed overhead. Effort: medium.

Parallelise Monte-Carlo across cores. Monte-Carlo samples are independent — an "embarrassingly parallel" workload — yet they run one at a time. Distribute them across CPU cores with joblib. Why it matters: near-linear speed-up for almost no algorithmic change. Effort: small-to-medium.

6.7   Priority 5 — the new feature work

Last, and only last, comes the genuinely new capability: the Chapter 2 physics modules. They go last not because they are unimportant but because they should be built on solid ground — a package that imports, has tests, is cleanly structured, and runs fast. Add them in the modern layout, each as a small focused module with its own test file: circuit.py, substrate.py, thermal_vectorized.py, electrothermal.py, environment_orbit.py, montecarlo.py, results.py. Why it matters: new features built on a stable, tested base stay correct as they grow, instead of inheriting and compounding the old fragility. Effort: large — but de-risked by everything that came before.

6.8   How to sequence it — and why not start with the fun part

The sequencing rule. Run & test → refactor → optimise → extend. Concretely: P1–P2 make it run and tested; then P3 refactors the structure under the protection of those tests; then P4 optimises the hot spots you have profiled and proven; then P5 builds new features on the now-solid base. The justification is a chain of dependencies, not a matter of taste: refactoring without tests is risky (you cannot tell whether a structural change altered the physics), and optimising without profiling is guesswork (you may speed up code that was never the bottleneck while the real one stays slow). Each tier is the safety equipment for the next.

Why is it so tempting to do this backwards? Because the bottom of the pyramid is boring and the top is exciting. Fixing a stray bracket and renaming a file feels like janitorial work; writing a vectorised electro-thermal solver feels like real engineering. But the excitement is exactly the trap: the glamorous work is the most dependent on the unglamorous foundation, and building it first means building it on code you cannot trust, cannot test, and have not measured.

Definition of done — a checklist for a healthy module. When is a PowerPy module finished? When all of these are true:
  • Single clear responsibility — one reason to change.
  • Type-hinted public functions — the contract is on the page.
  • Docstrings — what it does, parameters with units, what it returns.
  • A test file — its behaviour is pinned, not hoped about.
  • No stdlib-name clashes — nothing named string, json, random, … that shadows the standard library.
  • No bare excepts — failures are specific and logged, never swallowed.
  • Data via package resources, not hard-coded paths — it loads its data through the package, not from C:\Users\me\....
This is the bar P5's new modules must clear, and the bar P3 is dragging the legacy modules up to.

6.9   Worked example — one legacy smell, end to end EXAM-HOT

Worked example — the string.py shadowing smell: problem → fix → the test that proves it. We take one P1 item and walk it the whole way, ending — and this is the part that ties P1 to P2 — in the concrete test you would add so the bug can never silently return.
Problem. PowerPy has a top-level module string.py (the legacy String class). When any code runs from inside the package directory, Python finds this string on the import path before the standard library's string. Most libraries import stdlib string somewhere in their machinery — pandas does — so import pandas ends up loading PowerPy's module by mistake and fails. The error names pandas, so you hunt in entirely the wrong place. 
powerpy/
  string.py        # legacy String class — SHADOWS stdlib `string`
  cell.py
  ...

# from inside powerpy/:
>>> import pandas
ImportError: ... (pandas tried to use stdlib `string`, got PowerPy's instead)
Fix. Rename the module so it no longer collides with a standard-library name, and update every import that referred to it. 
# rename the file:
powerpy/string.py   ->   powerpy/cellstring.py

# update imports throughout the package:
- from powerpy.string import String
+ from powerpy.cellstring import String
Now nothing in the package is named string, so Python's stdlib string resolves normally and import pandas works from anywhere.
The test you would add (this is the P2 step). A fix is not done until a test guards it. Add a tiny test that fails on the old code and passes on the fixed code — it asserts both that the shadow is gone and that the symptom is cured: 
# tests/test_no_stdlib_shadow.py
import importlib

def test_stdlib_string_not_shadowed():
    """The stdlib `string` must resolve to the standard library, not PowerPy."""
    import string
    assert "site-packages" in string.__file__ or "lib" in string.__file__.lower()
    # i.e. it is NOT powerpy/string.py

def test_pandas_imports_cleanly():
    """The original symptom: importing pandas must not break."""
    pandas = importlib.import_module("pandas")
    assert pandas is not None
Check. Run the tests before the rename: test_stdlib_string_not_shadowed fails (stdlib string points at powerpy/string.py) and test_pandas_imports_cleanly fails (the ImportError). Run them after the rename: both pass. The bug is now regression-proofed — if anyone ever re-adds a top-level string.py, the suite goes red and names the real problem instead of blaming pandas. ✓ That is the P1→P2 motion in miniature: fix the defect, then pin it with a test so it cannot silently return.

6.10   In what order do you fix a fragile scientific codebase?

Q·a (foundation). Before you can trust any result PowerPy prints, what must be true of the code?
At minimum it must run — the package must import without a SyntaxError and without a name clash that breaks a dependency — and its core answers must be checked against something you know. A number from code that barely imports, or that you have never compared to a hand-calculation or a physical bound, is not a result; it is an output. So the first two things to establish are "it runs" (P1) and "its answers are pinned by tests" (P2). Everything else is premature until these hold.
Q·b (build up). Why refactor under tests — why not just carefully restructure the monolith classes and re-check by eye?
Because a refactor is a structural change with a promise attached: "I changed the shape but not the behaviour." Re-checking by eye cannot verify that promise across a large class with many code paths — you will miss the one branch where the split quietly changed a sign or an index. A test suite verifies the promise mechanically and instantly: make the structural change, run the tests, and if any answer moved, a test goes red and points at it. That is why P2 (tests) precedes P3 (refactor): the tests are the equipment that makes refactoring safe rather than a gamble.
Q·c (why). Why profile before optimising — surely the slow cell-by-cell loop is obviously the bottleneck?
It probably is — but "probably" is the problem. Performance intuition is wrong often enough that committing days of work to a guess is reckless. The loop might be dwarfed by, say, the repeated ngspice start-ups, or by file I/O. A profiler turns the guess into a measurement: it shows where the time actually goes, so you spend your effort where it pays. Optimising without profiling can mean speeding up code that was never the bottleneck — you do real work and the program is no faster. So P4 starts with "profile first", and only then vectorises, batches, and parallelises.
Q·d (deep — the target). So in what order should we fix a fragile scientific codebase, and why is the exciting performance/feature work last?
Put the rungs together. First make it run and be tested (Q·a) — without that, no result is trustworthy and nothing can be verified. Then refactor under those tests (Q·b) — the tests are what make restructuring safe rather than blind. Then profile and optimise (Q·c) — measure before you accelerate, so effort lands on the real bottleneck. Then, on that stable, tested, clean, fast base, build the new features (P5), each with its own tests, in the modern layout. The exciting work is last precisely because it is the most dependent: a vectorised solver you cannot test is unverifiable, an optimisation you have not profiled may be pointless, and a feature added to code that does not import cannot even run. You fix from the foundation up because each tier is the safety equipment for the one above it — and the glamorous tiers sit at the top.
Pitfall — starting with the fun part. The most common and most expensive mistake on a fragile scientific codebase is to open it and immediately reach for the interesting work: vectorise the solver, parallelise the Monte-Carlo, add the new orbit module. On code that does not yet import, or has no tests, this is building on sand. You vectorise a thermal solve with no parity test, so you cannot tell whether your fast version still gives the right answer. You parallelise before fixing the bare except that hides failures, so now failures are hidden and spread across cores. You add a feature to a package that crashes on import, so you cannot even run it. Every one of these efforts is wasted or worse, because the foundation it needed was never laid. Do P1–P2 first. Always.

Section 1 — What Software Design Is, and the Principles That Matter

D1-1. A function summarise_panel(path) that (i) opens and parses a CSV file, (ii) sums section powers and finds the weakest section, and (iii) prints a formatted table.

(a) The three concerns. They are exactly the same trio as the worked example, in different clothing: concern 1 — input/loading (open and parse the CSV into data); concern 2 — computation (sum the per-section powers, find the minimum); concern 3 — output/presentation (format and print the table to the screen). Reading a file, doing the maths, and showing the result are three different kinds of job fused into one function.

(b) The split. One focused function per concern, each with a small interface:
  • load(path) — takes a file path, returns the parsed data (e.g. a list of section records). Input only; no maths, no printing.
  • compute(data) — takes that data, returns the results (e.g. total_power and the weakest section). Pure computation; no file access, no printing.
  • show(total_power, weakest) — takes the computed results and prints the formatted table. Output only; no maths.
The caller wires them: data = load(path); total, weakest = compute(data); show(total, weakest).

(c) A concrete benefit + a new test. Cohesion rose (each function now does one job, fully described by its name) and coupling fell (compute() depends only on the data handed to it — no disk, no screen). The thing you can now test that you could not before: compute() in isolation — hand it a hand-built list of known section powers and assert that the total and the identified weakest section are correct, with no CSV file on disk and nothing printed. Previously, testing the arithmetic meant creating a real file and capturing console output just to check two numbers.
D1-2. Classifying the two designs by cohesion and coupling.

Design A — one SolarThing class doing five jobs (load, physics, netlist, simulate, plot), with other classes reaching directly into its internal variables.
  • Cohesion: LOW. A single class mixing five unrelated jobs is unfocused — it has five reasons to change, so it is the opposite of "one part, one job."
  • Coupling: HIGH. Other classes reach directly into its internal variables, so they depend on exactly how SolarThing is built inside. Change an internal variable and those classes break — the internals are not hidden, so the dependence is tight. (No encapsulation: there is no real boundary to protect the insides.)
Design B — four focused modules (schemas/ loader/ simulation/ render/), each doing one job, communicating only through small public methods.
  • Cohesion: HIGH. Each module does exactly one job, so each has a single reason to change and is easy to understand on its own.
  • Coupling: LOW. Modules talk only through small public methods and never touch one another's internals, so a change inside one module stays local and does not ripple to the others (encapsulation is respected).
Which is better, and why (one line). Design B. It achieves the design rule "high cohesion, low coupling": each part is understandable alone, testable alone, and changeable inside without breaking the others — whereas Design A is a low-cohesion, high-coupling monolith where every change is risky and ripples outward.
D1-3. A 40-line script (load one file, print one number) rewritten into 9 classes, 4 manager layers, 3 interfaces, and a plug-in system for file formats that will only ever have one format.

(a) Principle violated, and the name of the mistake. This violates YAGNI — building generality and structure on speculation ("in case we need other formats later") that the problem never actually asked for. The mistake is called over-engineering: imposing far more abstraction and machinery than the size of the problem warrants.

(b) More or less readable, and why. Less readable. The logic of a trivial 40-line task is now smeared across more than a dozen files, so following a single action means hopping between many classes, layers, and interfaces to trace what should be a couple of straightforward steps. Excessive splitting lowers readability just as surely as a giant monolith does — the flow is buried under structure that earns nothing. A few plain functions would let a reader see the whole job at a glance.

(c) The guideline that would have prevented it. YAGNI — "You Aren't Gonna Need It." Build only what the current task genuinely needs and add structure when a real need appears, rather than on the guess that it might be useful one day. Following it, the engineer would have kept a few clean functions and added the plug-in system only if and when a second file format actually arrived.

Section 2 — Structuring a Python Module and Package Well

D2-1. A flat pile — cell.py, read_json.py, physics.py, plotting.py, report.py, shapes.py — reorganised into a layered package like PowerPy's.

(a) Layered grouping. Sort each file by the one job it does, then stack the groups so each layer uses only the ones below it:
  • schemas/ (layer 1, bottom — data shapes): shapes.py. These are the dataclasses; they describe data and do nothing else, so they sit at the bottom and depend on nobody.
  • loader/ (layer 2 — read files into shapes): read_json.py. Turns JSON on disk into schemas objects; depends only on schemas.
  • simulation/ (layer 3 — the physics): physics.py, and cell.py if it carries cell-level behaviour. Computes results from loaded data; depends on schemas and loader. (If cell.py is purely a data shape rather than behaviour, it belongs in schemas/ instead — judge it by whether it holds data or runs physics.)
  • render/ (layer 4, top — output for humans): plotting.py, report.py. Turns results into figures and a report; may use everything below.
(b) Dependency arrows. All arrows point one way, downward — higher layers depend on lower, never the reverse:
render → simulation → loader → schemas
Read it as: render may import from simulation, loader, schemas; simulation from loader and schemas; loader from schemas; and schemas imports none of them. No arrow ever points up, so there are no cycles — that is exactly what lets you swap read_json.py for a CSV loader without touching the physics.

(c) The latent stdlib-clash risk. cell.py and the others are safe, but watch the pattern: the risk file in this pile is one whose name matches a standard-library module. Here that is none of the literal names given — unless you were tempted to rename read_json.py to json.py (the obvious "it handles JSON" name). json is a standard-library module, so a local json.py run from inside the folder would shadow it and break any import json elsewhere. Lesson for the grouping: keep the descriptive read_json.py (or json_loader.py) precisely because the tidy short name json.py collides with the standard library.
Check: every file landed in exactly one layer, the arrows form a straight downward chain with no cycle, and the one name that would have clashed with the stdlib was kept distinct. ✓
D2-2. A helper named json.py is added inside the package; afterwards import json + json.loads fails with AttributeError, but only when run from the project folder.

(a) What breaks, in terms of sys.path order. Running Python from inside the project folder puts that folder at sys.path[0] — the first place searched. import json walks sys.path top to bottom and takes the first match, which is now the local json.py, not the standard library's json (which sits further down the list). The local file does not define loads, so json.loads(...) raises AttributeError: module 'json' has no attribute 'loads'. It is launch-directory-dependent because sys.path[0] is the launch directory (the script's folder, or the cwd for the REPL): run from outside the folder and the local json.py is no longer at the front, so the real json is found first and the error disappears. The filename hasn't changed — only its position on the search path has.

(b) Why the stdlib still "has" the real json. Shadowing is about which file the name resolves to in this process, not about deleting anything. The standard library's json is still installed on disk and still further down sys.path; it is simply never reached, because the search stops at the first match (the local file). Nothing was overwritten — the real module is intact and would load fine the moment the local one is no longer in front of it.

(c) Two independent fixes.
• Filename fix: rename the helper to something that does not collide with a standard-library module — e.g. json_utils.py or data_io.py. With no local json.py, import json can only find the standard one, wherever you run from.
• Run/install fix: make the package installable (pyproject.toml + pip install -e .) and run from a neutral directory instead of inside the package. Then the package interior is no longer at sys.path[0] ahead of the stdlib, so even a stray same-named file would not sit in front of the real json at launch.
Check: both fixes attack the same mechanism (a local file winning the search) from the two independent angles the question asked for — remove the collision, or remove the package interior from the front of the path. Either alone resolves it. ✓
D2-3. Imports inside the simulation subpackage: ng_sim from a sibling electric.py, and cell from the schemas subpackage one level up.

(a) Relative import for ng_sim. electric sits beside the current file in the same package, so one leading dot ("my own package"):
from .electric import ng_sim

(b) Absolute import for cell. Name it by its full path from the top-level package:
from powerpy.schemas.cell import cell
(The equivalent relative form, since schemas is one level up from simulation, would use two dots: from ..schemas.cell import cell — one dot for "my package", a second to climb to the parent powerpy, then down into schemas.)

(c) The from power.cell import cell error. The fix is a single character: add the leading dot (turn the broken absolute reference to a non-existent top-level power package into a relative import). If cell really lives beside the file, the corrected line is
from .cell import cell.
Why it failed: from power.cell … told Python to find a top-level package named power; none exists on sys.path, so it raised ModuleNotFoundError: No module named 'power'. The dot says "look in this package" instead of hunting for a stranger named power — the whole bug was the missing dot.
Check: (a) uses one dot for a sibling, (b) spells the full path from the root for an absolute import, and (c) repairs an absent-package error with the single dot that redirects the search from "top-level power" to "here." ✓

Section 3 — Writing Functions and Classes That Are Clear and Correct

D3-1. The bug: a mutable default argument. The default log=[] is created once, when the function is defined — not freshly on each call. Every call that omits log therefore shares the same list, so readings accumulate across unrelated calls: 
append_reading(1)   # [1]
append_reading(2)   # [1, 2]   <-- the 1 from the previous call is still there
Why it misbehaves (one line). The default list is evaluated once at definition time, so all calls that don't pass their own list mutate one shared object.
The fix — default to None, create the list inside: 
def append_reading(x, log=None):
    if log is None:
        log = []            # a fresh list on every call that omits it
    log.append(x)
    return log
Now append_reading(1) → [1] and append_reading(2) → [2]: the two calls are independent. The same pattern fixes any mutable default (list, dict, set) — never use a mutable literal as a default; default to None and build it in the body.
D3-2. Rewrite returning a dataclass instead of a 4-tuple. 
from dataclasses import dataclass

@dataclass
class IVPoints:
    isc: float
    imp: float
    vmp: float
    voc: float

def iv_points(cell, t: float) -> IVPoints:
    """Return the four IV points of `cell` at temperature `t` (°C)."""
    isc = cell.isc_ref * (1 + 0.0005 * (t - 25))
    imp = isc * 0.95
    vmp = cell.vmp_ref * (1 - 0.003 * (t - 25))
    voc = vmp * 1.18
    return IVPoints(isc=isc, imp=imp, vmp=vmp, voc=voc)
A call site reading a field by name: 
pts = iv_points(cell, 40.0)
peak_power = pts.vmp * pts.imp        # by NAME — no counting positions
One bug the dataclass version makes impossible. Mis-indexing. With the tuple, a caller writing result[3] intending vmp would silently get voc instead (both are floats, so nothing complains) — a wrong number with no error. With the dataclass you write pts.vmp; there is no position to miscount, and a typo such as pts.vmpp raises AttributeError at the access site rather than flowing on as a wrong value. (Bonus: reordering or adding a field can't shift any caller's index, and the editor autocompletes the field names.)
D3-3. The two problems.
  1. Hard error — duplicated self in the signature. def calcT(self, self, p) lists self twice, which is a SyntaxError: duplicate argument 'self'. The method is uncallable and the module won't even import.
  2. Bad practice — bare except: pass. The try/except catches everything and silently discards it. If self.k is zero (division by zero), missing, or wrong-typed, the failure vanishes and self.T is left stale or never set — a silent wrong answer. It even swallows KeyboardInterrupt.
Clean rewrite (PEP 8 name, docstring, type hints, early validation, loud failure): 
class Panel:
    def compute_temperature(self, power: float) -> float:
        """Set and return the panel temperature from dissipated `power` (W).

        Parameters
        ----------
        power : float
            Dissipated power, in watts.

        Returns
        -------
        float
            The computed temperature (°C), also stored on self.temperature.

        Raises
        ------
        ValueError
            If the thermal conductance `self.k` is not positive.
        """
        if self.k <= 0:
            raise ValueError(f"thermal conductance k must be > 0, got {self.k}")
        self.temperature = power / self.k
        return self.temperature
What changed and why. One self (fixes the syntax error); the PEP 8 name compute_temperature with a descriptive power parameter instead of calcT/p; a docstring stating units, return, and what it raises; type hints (power: float -> float); the bad input (k ≤ 0) is validated early and raises a clear ValueError instead of being swallowed — so a real problem fails loudly and locatably rather than silently corrupting self.T. The method also now returns the value (not purely a hidden mutation), so callers can use it directly.
Note: this method still has a side effect (it writes self.temperature); if you wanted it fully pure you would compute and return power / self.k without storing it, letting the caller decide whether to save it.

Section 4 — Making It Fast: Performance and Time-Efficiency

D4-1. Which loops vectorise, and why.
  • (i) vectorises. Each output element out[k] depends only on its own input v[k] and a constant offset — there is no dependency between elements, so every element can be computed independently and therefore all at once. The one-line numpy replacement: 
    out = v * 2.0 + offset      # whole array, one compiled operation
    This pays the interpreter's per-element overhead once for the entire array instead of n times.
  • (ii) does not vectorise (as written). It is a running cumulative sum: total[k] = total[k-1] + x[k], so element k requires the result of element k−1. The blocking property is a genuine per-element sequential dependency — you cannot compute all elements "at once" because each one's input does not exist until the previous one is done. A plain out = ... array add cannot express it. (You can still avoid the Python loop with a purpose-built primitive: total = numpy.cumsum(x), which performs the prefix-sum in compiled code — but that is a special cumulative operation, not ordinary element-wise vectorisation.)
The lesson. Vectorisation works exactly when the per-element computations are independent; a true dependence on a neighbouring element blocks ordinary vectorisation and needs either a loop or a dedicated sequential primitive.
D4-2. (a) Legacy thermal time for the whole study. One fsolve per cell, 800 cells, 500 runs: 
legacy per run = 800 cells × 1.5 ms/cell = 1200 ms = 1.2 s/run
legacy total   = 500 runs × 1.2 s/run    = 600 s   (= 10 minutes)
(b) Vectorised total and speed-up. 3 ms per run for the whole array, 500 runs: 
vectorised total = 500 runs × 3 ms/run = 1500 ms = 1.5 s
speed-up         = 600 s / 1.5 s       ≈ 400×
Answer. The legacy approach costs ≈600 s of thermal work across the study; vectorising drops it to ≈1.5 s, a ≈400× speed-up (equivalently, per run: 1.2 s ÷ 3 ms ≈ 400×). The saving is ≈598.5 s, just under ten minutes. As in the notes, treat these as order-of-magnitude figures — the per-call costs are illustrative stand-ins — but the structural point holds for any reasonable inputs: vectorising removes the factor of N (here 800) from the per-run cost, and that factor multiplies across all 500 runs.
D4-3. Choose processes, not threads. The work is CPU-bound (heavy numpy/ngspice computation, no I/O waiting), and CPython's GIL (Global Interpreter Lock) permits only one thread to execute Python bytecode at a time. So Python threads doing computation cannot run truly in parallel — they take turns holding the lock — and you would get little or no speed-up from threads on this workload; they would essentially run one-at-a-time plus overhead. Processes each have their own interpreter and their own GIL, so they genuinely run simultaneously on separate cores. The 2000 runs are embarrassingly parallel (each is independent), so processes across the 8 cores give near-linear scaling. Use joblib or multiprocessing (both default to processes).
Best-case wall-clock. 
single-core total = 2000 runs × 0.4 s/run = 800 s
on 8 cores (ideal) ≈ 800 s / 8           = 100 s
So roughly 100 s in the ideal near-linear case — treat it as a floor; real time is a little higher because of process start-up and gathering results. (Threads, by contrast, would stay near the 800 s single-core figure for this CPU-bound work.)
D4-4. Critique. This is textbook premature optimisation: the colleague added complexity (12 tidy lines → 40 lines of bit-twiddling) to a function the profiler proves is only 0.4% of runtime, so even making it infinitely fast would cut total runtime by at most 0.4% — an unmeasurable gain bought at a permanent cost in readability. They optimised on a hunch instead of the measurement, and so sped up the wrong thing. What should have been optimised: the Monte-Carlo thermal solve, which the profiler shows is 86% of runtime — that is the hot spot, and the place where vectorising the per-cell fsolve loop or parallelising the independent runs would actually move the total. The rule (§4.1, §4.8): profile first, optimise only the proven hot spot, and leave clear code clear.

Section 5 — Testing, Reproducibility, and Trusting Your Results

D5-1. The assertion (agree to one millikelvin): 
import math
assert math.isclose(t_new, t_ref, abs_tol=1e-3)
Equivalently assert abs(t_new - t_ref) < 1e-3 — both say "the two front-temperatures differ by less than a millikelvin (1e-3 °C)". abs_tol (an absolute window) is the right choice here because we want a fixed millikelvin band, not a band that scales with the value.
Why not == (one line). Floating-point rounding makes two correct computations differ in their last bits, so t_new == t_ref would report failure on an answer that is right to twelve decimals — computed floats must be compared within a tolerance, never with ==.
D5-2. Two physical-sanity checks (any two of the following; messages make the failure self-explanatory): 
assert 0.0 <= t_panel <= 150.0, f"panel temp {t_panel} °C outside plausible 0–150 band"
assert p_out <= p_in,           f"output {p_out} W exceeds incident {p_in} W"
(A third valid one, if a current is in hand: assert current >= 0.0, f"forward-bias current {current} A is negative".)
Which bug each catches.
  • The temperature-band check catches a gross magnitude error in the thermal solve — a sign flip, a unit mix-up (kelvin vs °C), or a non-converged blow-up that lands the panel at, say, −800 °C or 4000 °C.
  • The power check catches a conservation-of-energy violation — a bug that has the panel output more electrical power than the sunlight delivers, which is physically impossible and signals a wrong efficiency, double-counting, or a sign error in the electrical model.
Both are weak (a plausible-but-wrong value still passes) but cost one line each and guard every result, so they are a cheap coarse net beneath the exact oracle checks.
D5-3. Why fixing the seed matters when comparing two layouts. A Monte-Carlo run draws random numbers; without a fixed seed those draws differ every run, so the computation is not deterministic — run the same layout twice and you get two different converged temperatures. If layout A and layout B are each simulated with a different unfixed seed, then any difference you observe between them is a mix of two causes you cannot separate: the genuine effect of the layout, and the noise of two different random draws. You might "conclude" A runs cooler when in truth it was just handed a milder random sample — the comparison is confounded.
What fixing the seed buys you. Give both layouts the same fixed seed and the random draws are identical, so the only thing that differs between the two runs is the layout itself. Now the computation is deterministic (same inputs → same outputs, so A repeats exactly and so does B) and the experiment is controlled: any temperature difference is attributable to the layout, not to luck. It is also reproducible — recording that seed and the parameters lets anyone recreate both runs exactly and confirm your conclusion. (This is the same discipline as Chapter 2 §6–§7: fix the seed, and log it with the results.)
D5-4. Why it gives false confidence. The test computes a boolean and then throws it away. math.isclose(t, 65.26, abs_tol=0.01) returns True or False, but its return value is never asserted — nothing acts on it. The function therefore reaches its end without raising, so pytest records it as passed no matter what t is. It is a vacuous pass: green, but proving nothing. (The same fault appears whenever a test calls a check but forgets the assert in front of it.)
The fix — assert the boolean: 
def test_thermal():
    t = vectorised_solver(reference_cell)
    assert math.isclose(t, 65.26, abs_tol=0.01)
Now the comparison's result is actually checked: if t drifts outside 65.26 ± 0.01 °C the assert raises and the test goes red. The rule: every test must contain at least one assert whose condition can genuinely fail — a check you compute but do not assert is no check at all.

D6-1. The right order: A → C → B. The rule is "make it run, then test, then optimise" — applied to these three exact issues.
  1. A — the unbalanced parenthesis (fix first). It is a SyntaxError, so the file will not import. Nothing else is even possible — you cannot run the slow loop, cannot write a test against the module, cannot profile it — until the code loads. A is a precondition for the other two, so it goes first regardless of how minor it looks. (Priority 1.)
  2. C — the missing thermal test (second). Before you dare change the slow loop, you need a way to tell whether your change altered the answer. A thermal test (e.g. a parity check, or a physical-bound assert) pins the current correct behaviour, so that when you rewrite the loop you can prove the result is unchanged. Writing the test before the optimisation is what makes the optimisation safe rather than a gamble. (Priority 2.)
  3. B — the slow cell-by-cell loop (last). Now, and only now, optimise: vectorise the loop. With the file importing (A) and the behaviour pinned by a test (C), you can change the loop aggressively and the test tells you instantly if you broke the physics. Optimising first would have meant editing code that might not even run, with no way to confirm the fast version is still correct. (Priority 4 — and ideally you profile first to confirm this loop really is the bottleneck.)
One-line justification each: A first because a SyntaxError blocks everything; C next because you must lock in the right answer before changing the code that produces it; B last because optimisation is only safe once the code runs and a test guards its correctness.
D6-2. Definition of done for montecarlo.py (the §6.8 checklist made specific to this module):
  • Single clear responsibility: run N independent samples of the array simulation with parameters drawn from their distributions, and collect the results — nothing else. It does not define the physics (it calls the simulation modules), does not parse input files (the loader does), and does not draw plots (render does). Its one reason to change is "how we sample / aggregate runs".
  • Type-hinted public functions: e.g. def run_monte_carlo(config: SimConfig, n_samples: int, seed: int | None = None) -> MonteCarloResult: — the inputs and the returned result type are on the page.
  • Docstrings: the public function states what it does, what n_samples and seed mean, what distribution each sampled parameter follows, and what the returned object contains (with units).
  • A test file (tests/test_montecarlo.py) that asserts: with a fixed seed the results are reproducible (same seed → same output); the output contains exactly n_samples entries; every sampled result obeys the physical bounds (no negative power, efficiency in [0, 1]); and a degenerate run with zero variance returns the deterministic answer.
  • No stdlib-name clashes: the module is montecarlo.py (not random.py, statistics.py, or anything that shadows the standard library).
  • No bare excepts: if one sample fails, it is caught specifically, logged with its sample index and parameters, and either re-raised or recorded as a failed sample — never silently dropped, which would bias the statistics.
  • Data via package resources, not hard-coded paths: the parameter distributions and any reference data come in through the config object (itself loaded via the package), not from a hard-coded C:\... path baked into the module.
Returns a typed result, not a bare structure: it hands back a @dataclass MonteCarloResult (e.g. samples, mean_power, p05, p95, n_failed) so callers read fields by name — consistent with the rest of the modern layer.
D6-3. What is wrong with the ordering. The plan is upside-down: it does the exciting, most-dependent work first (vectorising, a new feature) and leaves the foundation — making the code import and adding tests — for "if there's time" on the last day. That is exactly the pitfall of building on sand. On Day 1 you would be vectorising a thermal solve in a module that (per Day 3) does not even import yet, and with no test to prove the fast version still gives the right answer. On Day 2 you would add a new feature on top of that same broken, untested base, compounding the fragility. And tests treated as an optional afterthought usually never get written at all.

The week, reordered:
  • Day 1 — fix the import error and make the package installable. Nothing can run until it imports. (Priority 1.)
  • Day 2 — add the tests, especially the thermal parity / physical-bound tests, to pin the current correct behaviour. (Priority 2.)
  • Day 3 — vectorise the thermal solve (ideally profile first to confirm it is the bottleneck), checking the Day-2 tests stay green to prove the answer is unchanged. (Priority 4.) Then, on this now-stable base, add environment_orbit.py with its own tests. (Priority 5.)
In short: run → test → optimise → extend. The teammate's plan runs that sequence backwards, so each day's work rests on a foundation that has not been built yet.
