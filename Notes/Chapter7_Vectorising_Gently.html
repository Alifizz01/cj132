1What a vector is

A vector is just a row of numbers, treated as one thing. That's the whole definition.

And vectorising means doing the same operation to the whole row at once, instead of one number at a time. The simplest example — add two rows:

[10, 20, 30]  +  [1, 2, 3]  =  [11, 22, 33]

The computer did all three adds (10+1, 20+2, 30+3) in one step, not three separate ones. Doing a whole row in one go — that is all "vector" means.

The picture One-at-a-time is a single cashier ringing up items one by one. Vectorised is running the whole basket down the belt in one pass. Same items, same total — just not in a slow queue.

2The cells are just a row

Now the bridge — it's the same thing. Each solar cell has a temperature. Line them up in a row, and you can correct them all in one line.

T  = [ 65,  39,  187 ]   ← three cells' temperatures
dT = [ +2,  +1,  −5 ]   ← each cell's correction
─────────────────────────
T  = [ 67,  40,  182 ]   ← all three updated in ONE line

That is identical to [10,20,30] + [1,2,3] — only now the rows are the cells. If there were a thousand cells, the same single line T = T + dT would fix all thousand.

So far

Vectorising the thermal solver = keep every cell's temperature in a row, and nudge the whole row each step — instead of fixing cell 1, then cell 2, then cell 3.

The only question left: where does that dT row (the corrections) come from?

3Where the corrections come from

It's the Newton step you already know — nothing new. For a single unknown:

dT = − f ÷ f′

correction = − (wrongness) ÷ (slope)

The vectorised solver just does that on the whole row. Each cell has its own wrongness f and slope f′, lined up:

f  = [ +11.07,  −2.0,  +5.0 ]   ← each cell's imbalance
f′ = [ −0.033,  −0.10,  −0.05 ]   ← each cell's slope
─────────────────────────────
dT = −f ÷ f′ = [ +335,  −20,  +100 ]

(Check the first one: −(11.07 ÷ −0.033) = +335 — the exact step we computed by hand for the cold reverse-bias cell.) Then T = T + dT applies them. So one Newton round is just two row-lines:

dT = −f ÷ f′   ← Newton step, all cells at once
T  = T + dT   ← update, all cells at once

4Why doing the whole row is faster

It's not the same total work. The arithmetic is the same — but the one-at-a-time loop wraps a mountain of bookkeeping around every single number, and vectorising throws that away.

The big reason — overhead

In a loop, for every number the language quietly asks "what type is this? make an object to hold the result, fetch the next item, …" — and that bookkeeping dwarfs the actual add. You pay it 1000 times. A row operation hands the whole row to fast pre-compiled code that pays the bookkeeping once and just does the adds. That alone is most of the speed-up (often 10–100×).

The cashier again The slow cashier, per item, walks to the shelf, looks up the price, walks back — then adds it. Almost all the time is walking, not adding. The fast lane runs the whole basket in one pass: the walking is done once, not per item.
The bonus reason — a little real "at once"

On top of that, the chip has special instructions (called SIMD) that genuinely add a small batch — say 8 numbers — in a single tick, and big jobs can spread across several CPU cores. So some of it really is simultaneous — just ~8 at a time, not 1000. The main win is still the overhead removal above.

In one line

Faster because the slow part was never the maths — it was the per-number bookkeeping, and vectorising pays it once for the whole row (plus the chip crunches a few at a literal once).

5The one wrinkle: a cell has two temperatures

Everything above used one temperature per cell. A real cell has two — a sunlit front (T₁) and a shaded rear (T₂) — joined by heat conducting through the cell. That's the only complication, and it's where the letters a, b, c, d come from.

Two temperatures means two wrongnesses:

f₁ = front's imbalance   |   f₂ = rear's imbalance

And here's the new bit: with one unknown, the slope f′ was one number. But now nudging the front changes both faces (heat flows front→rear), and nudging the rear changes both too. So one slope becomes four:

how it changes f₁ (front)how it changes f₂ (rear)
nudge T₁ac
nudge T₂bd
Why four, in plain words The two faces are tied together by a heat "pipe." So you can't move one without the other feeling it. The four numbers just record "if I nudge this face, how do both faces react." The two cross-numbers (b and c) are exactly that pipe.

6The four slopes and the det recipe

With one unknown the step was simply dT = −f ÷ f′ — "divide by the slope." With two tangled unknowns you can't divide by a single slope; you undo the little 2×2 table of four slopes. A helper number called det makes that possible:

det = a·d − b·c

dT₁ = −(d·f₁ − b·f₂) ÷ det   ← front correction
dT₂ = −(−c·f₁ + a·f₂) ÷ det   ← rear correction

It looks bigger, but it's the same move as before — "wrongness ÷ slope" — just written for two linked unknowns. Notice each correction uses both f₁ and f₂: that's the two faces borrowing from each other because they're tied together. det is simply the two-unknown version of dividing by f′.

The shape of one Newton step now

two imbalances (f₁, f₂) → four slopes (a, b, c, d) → one helper (det) → two corrections (dT₁, dT₂) → update both temperatures. Same spirit as the one-unknown step, one dimension richer.

7One real cell, all the way through

The idle sunlit cell, starting from a guess of 28 °C. Every number here is real (checked against the live solver). Heat in from sun = 3.978 W; rear gets no sun; the conduction link C·A = 3.0; start T₁ = T₂ = 301.15 K (28 °C).

1 · the two imbalances
f₁ = +2.719 W   ← front: surplus (sun in, not glowing enough yet → too cold)
f₂ = −1.245 W   ← rear: shortfall (no sun, but still glowing)
2 · the four slopes
a = −3.017    b = 3.0
c = 3.0       d = −3.017

b = c = 3.0 — that's exactly the conduction link C·A, the "pipe" tying the faces. a and d are each face's own response plus that link.

3 · det
det = a·d − b·c = (−3.017)(−3.017) − (3.0)(3.0) = 9.100 − 9.000 = 0.100
4 · the corrections (plug in)
dT₁ = −(d·f₁ − b·f₂)/det = −[(−3.017)(2.719) − (3.0)(−1.245)]/0.100
    = −[−8.201 + 3.736]/0.100 = −(−4.465)/0.100 = +44.6

dT₂ = −(−c·f₁ + a·f₂)/det = −[−(3.0)(2.719) + (−3.017)(−1.245)]/0.100
    = −[−8.156 + 3.757]/0.100 = −(−4.400)/0.100 = +44.0
5 · update
new T₁ = 301.15 + 44.6 = 345.8 K = 72.6 °C
new T₂ = 301.15 + 44.0 = 345.1 K = 72.0 °C

From 28 °C, one step jumped both faces to ~72 °C — a little past the answer. A few more identical steps settle it to 65.26 °C front / 64.60 °C rear (the verified oracle numbers).

8One cell vs a thousand — the payoff

Everything on the last page was a single number for one cell. For a fleet of cells, each of those becomes a row, and every line runs on the whole row in one shot:

det = a*d − b*c        ← every cell's det, one line
dT1 = −(d*f1 − b*f2)/det  ← every cell's front correction, one line
T1  = T1 + dT1       ← update all fronts, one line

That is exactly the block of code that looked scary at first — now you can read it. It's just the two-unknown Newton step (the det recipe), applied to whole rows of cells at the same time.

The whole thing, end to end

One cell: 2 linked temperatures → 2 imbalances + a 2×2 of 4 slopes → solve with the det recipe → get dT₁, dT₂ → update.

A thousand cells: the same lines, but every symbol is a row — all thousand solved together, no per-cell loop. That's the vectorised two-node Newton solver.

9What changes with lateral heat (a peek)

One last thing, so you know the edge of this. Right now the cells are independent — each cell's 2×2 stands alone. Stack them and you get separate little boxes down a diagonal, nothing linking them:

If you turn on lateral heat (heat spreading sideways to neighbours — Chapter 4), the boxes get joined. Then there's no tiny det formula anymore; you need a proper sparse matrix solver. The row idea survives (still no per-cell loop), but the cheap 2×2 trick is replaced by one big solve.

The clean summary

The vector (all cells in one row, no loop) always survives. The cheap part — the 2×2-on-rows — works only because the cells are independent. Link them sideways and you trade it for one sparse solve.
