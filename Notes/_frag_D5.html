5   Testing, reproducibility, and trusting your results

Everything so far has been about writing code that is clear and fast. This section is about the question that outranks both: is it right? A simulator that crashes is annoying; a simulator that quietly returns the wrong number is dangerous, because nothing warns you. You run it, the output looks plausible — a panel temperature of 71 °C, a bus current of 4.2 A — you drop it in a report, and you have just shipped a wrong answer with full confidence. The defence is not "read the code more carefully" (you will still fool yourself). The defence is to make the machine prove the code works, on cases whose answers you already know, automatically, every time the code changes.

The one idea. A scientific result you have not tested is a result you only hope is correct. Testing turns "I think it works" into "I can show it works, repeatably." You do this by feeding each piece of code an input whose correct output you know independently — from a trusted reference, a hand calculation, or a physical bound — and asserting that the code reproduces it. Pin those checks in a tests/ folder, run them with pytest on every change, and fix the random seed so a stochastic run repeats exactly. The payoff is enormous: a silent numerical bug, the worst kind, becomes a loud, located test failure instead of a plausible lie in your plot.

We will build the vocabulary first — the four kinds of test, then oracle, fixture, deterministic, continuous integration — then make each concrete for PowerPy (a thermal parity test against the legacy fsolve, a hand-worked energy-balance oracle, a 2×2 circuit you can check by hand before trusting 600 cells, and cheap physical-sanity bounds). Then a fully narrated pytest example, a Socratic ladder, the classic pitfalls, and woven practice.

5.1   Why a scientific code needs tests at all

In ordinary software a bug usually announces itself: the button does nothing, the page 500s, the program throws. In numerical software the most damaging bugs do the opposite — they succeed. A sign flipped in a heat-balance term, a Jacobian transposed, an index off by one, kelvin fed where celsius was meant: none of these crash. Each produces a number, and the number is in the right ballpark, so it looks like an answer. You cannot tell a correct 65 °C from a buggy 65 °C by looking at it.

That is the whole reason testing matters more here than almost anywhere. A test is an independent statement of what the answer should be, written down once and checked forever after. It is the difference between trusting your code because it ran, and trusting it because you can demonstrate, on command, that it reproduces answers you obtained some other way.

The silent-bug threat model. Ask of every result: "if this were wrong by 10%, would anything have told me?" If the honest answer is "no" — no crash, no warning, no check — then you are not verifying that number, you are hoping about it. A test is how you convert a hope into a verification.

5.2   The four kinds of test

"Testing" is not one activity. There are four distinct kinds, each catching a different class of error. Learn the name and the job of each.

Unit test. A test that checks one small piece of code — usually a single function — against a known expected output. You give it inputs, you assert the result equals (or is close to) the value you worked out independently. It is the smallest, fastest, most numerous kind of test. In Python you write them with pytest: a plain function named test_<something> containing an assert. 
# a unit test: one function, one known answer
def test_compute_isc_at_reference():
    cell = make_reference_cell()       # isc_ref = 0.50 A at 25 °C
    assert compute_isc(cell, 25.0) == 0.50    # at t_ref, isc == isc_ref exactly
Integration test. A test that checks several pieces working together, end to end, rather than in isolation. A unit test proves build_circuit and solve_circuit each work alone; an integration test proves that building a small circuit and then solving it produces the right node voltages — that the pieces connect correctly. It catches the bugs that live in the seams between functions, which no single unit test can see.
Regression test. A test that pins the current, known-correct output so that a future change cannot silently alter it. You capture today's right answer as the expected value; if a later edit changes it, the test fails and forces you to notice. The name says it: it guards against a regression — a change that makes previously-correct behaviour wrong. ("I refactored the Jacobian and the converged temperature moved by 4 °C" should be a red test, not a surprise in next month's plot.)
Parity test. A test that checks a new implementation agrees with a trusted existing one on the same input, to within a tolerance. When you rewrite a slow-but-correct routine as a fast one, the old routine becomes your reference: feed both the identical input and assert the outputs match. Parity testing is exactly how you gain confidence that an optimisation (Section 4's vectorised thermal solve) did not change the answer, only the speed.
The testing pyramid. A healthy test suite is shaped like a pyramid: a broad base of many fast unit tests (cheap, run constantly), a thinner middle of integration tests, and a few slow end-to-end tests at the apex. You want most of your safety net to be fast unit tests because they run in milliseconds and you run them all the time. Reserve the heavy end-to-end runs for the handful of checks that genuinely need the whole system.

5.3   The oracle: where the "expected answer" comes from

Every test asserts that the code's output equals an expected value. That raises the obvious question: where does the expected value come from, and why do you trust it? That trusted source has a name.

Oracle. A trusted source of the correct answer for a given input — the thing your code's output is checked against. An oracle can be (a) a reference implementation you already trust (the legacy fsolve solver), (b) a hand calculation you did on paper (a single-cell energy balance worked out by hand), (c) an analytic closed-form solution for a special case, or (d) a known physical bound the answer must respect. The art of testing a numerical code is largely the art of finding cheap, trustworthy oracles.

The four kinds of test and the oracle interlock: a parity test uses a reference implementation as its oracle; a unit test often uses a hand or analytic oracle; a sanity check uses a physical-bound oracle. Keep the word in mind — when someone asks "how do you know that's the right answer?", the answer is always "because my oracle says so, and here is why I trust the oracle."

5.4   Practical tests for PowerPy

Abstractions land only when made concrete, so here are the exact tests that would guard PowerPy's thermal and electrical core — the code from Chapter 2.

(a) Parity: the vectorised solver vs the legacy fsolve. Section 4 replaced the per-cell scipy.optimize.fsolve loop with a single vectorised Newton step over the whole array. That is a performance change, and a performance change must not move the answer. The parity test makes that explicit: take the worked-example inputs from Chapter 2, run the old per-cell solver and the new vectorised solver on the same single cell, and assert they agree. Both must give a front-face temperature of 65.26 °C (the converged value from worked example 2.1). The assertion idea:

t_ref = legacy_fsolve_single_cell(cell)     # trusted reference (the oracle)
t_new = vectorised_solver(cell)             # the new implementation
assert abs(t_new - t_ref) < 1e-3            # agree to a millikelvin

If that passes, you have shown the optimisation preserved the physics to a millikelvin. If it fails, you caught a regression the instant you introduced it — not three plots later.

(b) Hand-worked oracle, when the reference itself is unavailable. Parity assumes you have a trusted reference. Sometimes you do not — the legacy code is broken, missing, or itself under suspicion. Then you fall back to an oracle you can compute by hand: the single-cell energy balance from Chapter 2. For one cell at steady state the front and rear nodes must each have net-zero heat flow; solving that 2-node balance by hand gives front = 65.26 °C, rear = 64.60 °C. That paper answer is now your oracle, and the new solver is checked against it:

# oracle = the hand-computed Chapter-2 single-cell balance
assert math.isclose(t_front_new, 65.26, abs_tol=0.01)
assert math.isclose(t_rear_new,  64.60, abs_tol=0.01)

This is strictly stronger than parity in one way: a parity test only proves the new code matches the old code — if the old code was also wrong, they agree on a wrong answer. A hand oracle proves the code matches physics.

(c) Small-circuit sanity: a 2×2 before a 600-cell array. Before you trust ngspice on a 600-cell array whose internals you can never check by hand, trust it on a 2×2 cell circuit you can. With four cells you can reason about every node voltage and branch current on paper — series cells add voltage, parallel strings add current — and then read the per-cell V and I back from ngspice and confirm each matches your hand reasoning. Only once the 2×2 reads back correctly do you scale up. The small case is where bugs in wiring, indexing, and reading results are cheap to find; the 600-cell case is where they are impossible to find.

(d) Physical-sanity checks. The cheapest tests of all assert that a result lands inside a physically plausible band — no exact oracle needed, just physics. A sunlit panel temperature should sit somewhere in, say, 0–150 °C; a cell current in forward bias should be non-negative; output power should never exceed the incident power. These are one-line asserts that cost nothing and catch gross errors instantly — a temperature of −800 °C or 4000 °C, a negative current, a panel that outputs more than it absorbs all scream a bug.

assert 0.0 <= t_panel <= 150.0,  f"panel temp {t_panel} °C is unphysical"
assert current >= 0.0,           f"forward-bias current {current} A is negative"
assert p_out <= p_in,            f"output {p_out} W exceeds input {p_in} W"
Sanity checks vs exact checks. A physical-sanity bound is weak (a buggy 64 °C still passes "0–150 °C") but cheap and broad — it guards every result for free. An oracle check is strong (it pins the exact value) but narrow — it only covers the one case you computed. Use both: sanity bounds everywhere as a coarse net, exact oracles on the few cases you can pin precisely.

5.5   Floating point: never compare with ==

There is a trap that bites every numerical-testing beginner exactly once: comparing two floating-point numbers with ==. It does not work, and the reason is fundamental, not a quirk you can wish away.

Floating-point rounding. Computers store real numbers in a finite number of bits, so most values are kept only approximately. Numbers like 0.1 have no exact binary representation, and every arithmetic operation rounds its result to the nearest representable value. Tiny rounding errors therefore accumulate, so two calculations that are mathematically equal can differ in their last bits. The classic shock: 0.1 + 0.2 == 0.3 is False in Python, because 0.1 + 0.2 stores as 0.30000000000000004.

The consequence for testing: assert t_new == 65.26 is almost guaranteed to fail even when the answer is correct, because the solver's last bits will differ from the literal. The fix is to compare within a tolerance — assert the two values are close enough, not bit-identical.

  • One scalar: math.isclose(a, b, abs_tol=1e-3) — true when a and b differ by less than the tolerance. (Use abs_tol for an absolute window near zero; rel_tol for a relative one on large values.)
  • A whole array (PowerPy's natural case — temperatures over all N cells): numpy.allclose(arr_new, arr_ref, atol=1e-3) — true when every element is within tolerance. This is the vectorised cousin of isclose, and exactly what you want for checking a 600-element temperature vector in one line.
import math, numpy as np

math.isclose(0.1 + 0.2, 0.3)                 # True  — within default tolerance
(0.1 + 0.2) == 0.3                           # False — the == trap

np.allclose(t_new_array, t_ref_array, atol=1e-3)   # all N cells agree?  one bool
Pitfall — comparing floats with ==. assert result == 65.26 or if voltage == 0.3: will fail (or branch wrong) on values that are correct to twelve decimal places, because rounding makes the last bits differ. Never use == (or !=) on computed floats. Always compare with a tolerance: math.isclose for a scalar, numpy.allclose for an array. The only floats you may safely == are ones you never did arithmetic on (a literal you stored and read back unchanged).

5.6   Reproducibility: fix the seed, record the run

Chapter 2's Monte-Carlo hot-spot study draws random numbers. That creates a problem for testing and for science alike: if every run draws different randoms, every run gives a different answer, and you can neither test it (the expected value moves) nor reproduce a result you reported last week. The fix rests on one concept.

Deterministic. A computation is deterministic if the same inputs always produce exactly the same outputs — no run-to-run variation. Ordinary arithmetic is deterministic; anything that draws "random" numbers is not, unless you pin the randomness. A pseudo-random generator is actually a deterministic sequence controlled by a starting value called the seed: fix the seed and the "random" draws are identical every run, which makes the whole Monte-Carlo computation deterministic and therefore reproducible and testable.

import numpy as np
rng = np.random.default_rng(seed=12345)   # fix the seed
samples = rng.normal(size=1000)           # identical sequence on every run

With the seed fixed, a Monte-Carlo run repeats exactly, so you can regression-test its output and anyone can reproduce your figure. But a seed alone is not enough for science: you must also record the seed and the parameters alongside the results. A result file that does not say which seed and which inputs produced it is not reproducible, however deterministic the code was — the determinism is wasted if no one knows the starting conditions. (This ties directly to Chapter 2 §6–§7, where the run's seed and parameter set are logged with the output.)

Reproducible = deterministic code + recorded conditions. Fixing the seed makes the code deterministic (same inputs → same outputs). Recording the seed and parameters with the results makes the run reproducible (someone else, later, can recreate the exact inputs). You need both. A determinism you don't record, and a record without determinism, each fail to let a future reader reproduce your number.

5.7   How tests are organised and run

Tests are not scattered through the code; they live in a conventional place and are run by a tool.

  • A tests/ folder. By convention, test files live in a top-level tests/ directory, named test_*.py, mirroring the modules they cover (test_thermal.py tests the thermal solver). Keeping them separate from the source keeps the shipped code clean while keeping the safety net close.
  • Small, fast tests. Each test should check one thing and run quickly. Fast tests get run often (you will run a 200 ms suite after every edit; you will avoid a 10 minute one), and a test that runs often is a test that actually catches bugs.
  • pytest to run them. pytest is the standard Python test runner: point it at the folder and it discovers every test_* function, runs them, and reports passes and failures with the offending values. You write plain functions with assert; pytest does the rest.
Fixture. A reusable piece of setup shared by many tests — a pre-built input object the tests need. In pytest you write a function decorated with @pytest.fixture that builds, say, a reference solar cell, and any test that lists the fixture's name as a parameter receives that ready-made object. Fixtures keep tests DRY (Don't Repeat Yourself — you build the reference cell once, not in every test) and make each test read as just "given this input, assert this output". 
import pytest

@pytest.fixture
def reference_cell():
    """A single cell with the Chapter-2 worked-example parameters."""
    return make_cell(isc_ref=0.50, alpha_isc=0.0005, t_ref=25.0, area=4e-4)

def test_isc_at_reference(reference_cell):       # fixture injected by name
    assert compute_isc(reference_cell, 25.0) == 0.50
Continuous Integration (CI). Continuous Integration — the practice of running your test suite automatically on every change (every push or merge), on a server, so broken code is caught the moment it is introduced rather than discovered later. You configure it once; thereafter every change is greeted by the full suite, and a red result blocks the change. CI is what turns "we have tests" into "the tests are always run" — the difference between a safety net you own and one you actually use.

5.8   Assertions and defensive checks in the code itself

Tests live outside the code and run on demand. But you can also place checks inside the code that fire during every real run, guarding against bad inputs and broken invariants. The goal is the same as in Section 3: make a failure loud and located rather than a silent NaN that propagates.

The sharpest PowerPy example comes straight from Chapter 2's vectorised solver. The global thermal Jacobian there is block-diagonal: N independent 2×2 blocks, one per cell. Each Newton step inverts those 2×2 blocks, and a 2×2 inverse divides by the determinant det = ad − bc. If a block's determinant is (near) zero — a degenerate cell, a bad parameter — that division produces inf or NaN, which then silently poisons every downstream temperature. A defensive check catches it at the source, where the message can still name the cause:

# inside the Newton step, before dividing by the 2×2 determinant
det = a * d - b * c
if np.any(np.abs(det) < 1e-12):
    bad = np.where(np.abs(det) < 1e-12)[0]
    raise ValueError(f"singular 2x2 Jacobian block(s) at cells {bad}: det≈0")
delta = solve_2x2(a, b, c, d, residual)     # safe: no block is singular

Without the guard, a singular block yields a NaN temperature that flows on through the electro-thermal loop and surfaces — if at all — as a baffling "the array won't converge" three layers away. With the guard, you get a precise message naming the exact cells, at the exact line. Validate inputs and invariants early, so failures are loud and located, not silent and far away.

5.9   Worked example — a narrated pytest parity + sanity test EXAM-HOT

Worked example — one pytest file that parity-checks the thermal solve and asserts a physical bound. We build the shared input as a fixture, call both solvers, and assert (1) they agree to a tolerance and (2) the result is physically plausible. Every line is narrated below.
The test file (tests/test_thermal_parity.py): 
import math
import pytest
from powerpy.thermal import legacy_fsolve_single_cell, vectorised_solver
from powerpy.cells import make_cell

@pytest.fixture
def worked_example_cell():
    """The single cell from Chapter-2 worked example 2.1."""
    return make_cell(isc_ref=0.50, alpha_isc=0.0005, t_ref=25.0, area=4e-4)

def test_vectorised_matches_legacy(worked_example_cell):
    t_ref = legacy_fsolve_single_cell(worked_example_cell)   # trusted oracle
    t_new = vectorised_solver(worked_example_cell)           # new implementation
    assert math.isclose(t_new, t_ref, abs_tol=1e-3)          # parity to a mK

def test_panel_temperature_is_physical(worked_example_cell):
    t_new = vectorised_solver(worked_example_cell)
    assert 0.0 <= t_new <= 150.0      # a sunlit panel must land in a sane band
Line by line.
  • import math, pytest — math.isclose for the tolerance comparison; pytest for the fixture decorator and to run the file.
  • the imports from powerpy pull in the two solvers under test and the cell builder — the things whose correctness we are checking.
  • @pytest.fixture / def worked_example_cell() — builds the shared input once: a cell carrying the exact parameters of Chapter 2's worked example. Any test that names worked_example_cell as a parameter is handed this object, so we never rebuild it.
  • def test_vectorised_matches_legacy(worked_example_cell) — pytest sees the test_ prefix and runs it; it sees the parameter name matches the fixture and injects the cell.
  • t_ref = legacy_fsolve_single_cell(...) — run the trusted legacy solver; its output is our oracle (≈ 65.26 °C).
  • t_new = vectorised_solver(...) — run the new vectorised solver on the identical input.
  • assert math.isclose(t_new, t_ref, abs_tol=1e-3) — the heart of the parity test: the two must agree to within a millikelvin. Note isclose, not == — float rounding would break ==.
  • the second test, test_panel_temperature_is_physical, is the cheap sanity net: whatever the exact value, a sunlit panel must lie in 0–150 °C, so a gross bug (a negative or four-figure temperature) fails loudly even if no oracle were available.
Running it. pytest tests/test_thermal_parity.py discovers both test_ functions, injects the fixture, runs them, and prints 2 passed — or, on a regression, shows the two values it compared and the line that failed. Two small, fast tests now stand permanently between a future edit and a silent change to the converged temperature. ✓

5.10   Socratic ladder — trusting a solver with no perfect reference

Q·a (foundation). What is a unit test, in one sentence, and what does it need to exist?
A unit test checks one small piece of code — typically a single function — against a known expected output: call the function with chosen inputs and assert the result matches the value you obtained independently. It needs exactly one thing to exist: an expected answer you trust. No expected answer, no test — which is why the next rung matters.
Q·b (the source of truth). That "expected answer" — what is it called, and what can supply it when there is no trusted reference solver to copy from?
It is the oracle: a trusted source of the correct answer. When no reference implementation exists, the oracle can instead be a hand calculation, an analytic solution for a special case, or a physical bound the answer must respect. The reference solver is only one possible oracle; losing it does not leave you blind, it just sends you to the other three.
Q·c (a hand-computable case). For PowerPy's 2-node thermal balance, what concrete case can you actually compute by hand to serve as that oracle?
A single cell at steady state. With two nodes (front, rear) and the requirement that each has net-zero heat flow, the balance is a small system you can solve on paper — and Chapter 2 does, getting front = 65.26 °C, rear = 64.60 °C. That hand answer becomes the oracle: assert math.isclose(t_front_new, 65.26, abs_tol=0.01). It checks the new solver against physics, not against possibly-buggy old code.
Q·d (cheap broad coverage). The hand oracle covers one case. What do physical-sanity bounds add that an exact oracle does not?
Breadth, for free. A bound like 0 ≤ T ≤ 150 °C, or current ≥ 0 in forward bias, or power ≤ input, holds for every result, not just the one you hand-computed. It is weak (a wrong-but-plausible 64 °C slips through) but it costs one line and catches every gross error — a −800 °C temperature, a negative current, a panel emitting more than it absorbs — across the whole simulation. Coarse net everywhere, sharp oracle on a few points.
Q·e (scaling confidence). Why build a 2×2-cell circuit and check it by hand before trusting a 600-cell array?
Because the 2×2 is hand-checkable and the 600-cell is not. With four cells you can reason out every node voltage and branch current on paper and confirm the per-cell V/I read back from ngspice match — catching wiring, indexing, and result-reading bugs while they are cheap to find. Those same bugs in a 600-cell array are invisible (no one can hand-check 600 cells). Get the small case provably right first, then scale, so the only new thing in the big run is size, not correctness.
Q·f (the target). So: if we have no perfectly-trusted reference implementation, how do we still know the new solver is correct?
Stack the rungs. We do not need a reference implementation because the oracle can be a hand calculation (Q·b): the single-cell 2-node balance gives 65.26/64.60 °C by hand (Q·c), and a unit test (Q·a) pins the solver to it with math.isclose. Around that exact check we throw physical-sanity bounds (Q·d) that cheaply guard every other result, and we earn confidence in the wiring on a 2×2 circuit we can check by hand (Q·e) before scaling to 600 cells. Correctness without a trusted twin comes from anchoring to physics you can compute, bounding everything you cannot, and proving the small case before trusting the large one. If a reference later appears, a parity test (abs(t_new − t_ref) < 1e-3) is added on top — but it was never the only road to trust.

5.11   Pitfalls

Pitfall 1 — comparing floats with ==. (Restated because it is the one everyone hits.) assert t_new == 65.26 fails on a correct answer because rounding makes the last bits differ. Use a tolerance: math.isclose(t_new, 65.26, abs_tol=1e-2) for a scalar, numpy.allclose(...) for an array. Reserve == for integers and for floats you never did arithmetic on.
Pitfall 2 — a test that never actually asserts (it passes vacuously). A test with no reachable assert always "passes" — and proves nothing. The classic forms: 
def test_solver():
    t = vectorised_solver(cell)      # computes a value...
    # ...and never checks it. Passes no matter how wrong t is.

def test_close():
    math.isclose(t_new, t_ref)       # returns True/False but it is NOT asserted!
The first never checks its result; the second computes a boolean and throws it away (you must write assert math.isclose(...)). A green suite full of vacuous tests is worse than no suite — it is false confidence. Every test must contain at least one assert that can actually fail.
Pitfall 3 — trusting a big simulation you never checked on a small hand-solvable case. Running the 600-cell array and eyeballing "looks about right" is not verification — it is the exact situation where a silent bug hides, because no one can hand-check 600 cells. If you never confirmed the 2×2 circuit, or never pinned the single-cell temperature to the hand oracle, then the big number rests on nothing. Always anchor the large run to a small case whose answer you know.

Section recap. Test a scientific code because its worst bugs are silent — a wrong number that looks plausible. Four kinds of test: unit (one function vs a known output, via pytest + assert), integration (pieces working together end-to-end), regression (pin today's correct output so a future change can't silently move it), parity (new impl vs a trusted one to a tolerance). The expected value comes from an oracle — a trusted reference implementation, a hand calculation, an analytic case, or a physical bound. For PowerPy: parity of the vectorised solver vs legacy scipy.fsolve (both → front ≈ 65.26 °C, assert abs(t_new−t_ref)<1e-3); a hand-worked single-cell energy-balance oracle (65.26/64.60 °C) when the reference is unavailable; a 2×2 circuit checked by hand before a 600-cell array; physical-sanity bounds (0–150 °C, current ≥ 0, power ≤ input). Never compare computed floats with == (rounding) — use math.isclose (scalar) or numpy.allclose (array). For reproducibility, fix the seed so a Monte-Carlo run is deterministic, and record seed + parameters with the results so the run is reproducible. Organise tests in tests/, keep them small and fast, run with pytest, share setup via fixtures, and run them automatically on every change (Continuous Integration, CI). Put defensive checks in the code too — e.g. guard det ≠ 0 before inverting each 2×2 Jacobian block — so failures are loud and located, not silent NaNs. Avoid the three traps: == on floats; a test that never asserts (passes vacuously); and trusting a big simulation you never checked on a small hand-solvable case.
