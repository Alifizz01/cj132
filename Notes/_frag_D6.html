6   A concrete improvement roadmap for PowerPy

The previous five sections were about principles: how to cut a program into parts, how to write a function you can trust, how to test physics, how to find and fix the hot spots. This final section spends that capital. It turns those principles into a single, ordered, actionable plan for this codebase — PowerPy, the satellite solar-array simulator we have been dissecting throughout — and, just as importantly, it argues for the order. A roadmap is not a wish-list. A wish-list says "here are ten good things". A roadmap says "here are ten good things, and here is which one you do first, and why doing them out of order will hurt you". The hard, counter-intuitive lesson at the centre of this section is that the most exciting work — the vectorised solver, the parallel Monte-Carlo, the new physics modules — is the work you do last, and starting with it is the single most common way to waste weeks on a scientific codebase.

The one idea. Make it run and be tested first; then refactor the structure under the protection of those tests; then optimise the hot spots you have actually measured; then build the new features. Each stage protects the next. Refactoring without tests is editing blind — you cannot tell whether you broke the physics. Optimising without profiling is guessing — you speed up code that was never the bottleneck. Building features on a package that does not even import is building on sand. The roadmap below is exactly this order, made concrete for PowerPy, with each item framed as problem → fix → why it matters → rough effort.

6.1   The current state of PowerPy, stated plainly

Before planning repairs, look honestly at what is there. PowerPy is not uniformly good or uniformly bad — it is a two-layer codebase, and the two layers are at very different levels of health. Recognising this split is the whole basis of the plan, because the two layers need opposite things.

The good, modern layer. A set of subpackages — schemas/, loader/, simulation/, render/ — built in the style this course recommends: small modules, @dataclass data shapes, a clear dependency direction (schemas at the bottom, render at the top), and an explicit pipeline that wires the stages together. This layer is a strength. The plan's job for it is mostly to extend it: it is the target shape everything else should grow toward.

The fragile, legacy layer. A set of large, top-level, multi-job classes in flat modules — cell.py, string.py, section.py, panel.py. Several of these files were OCR-reconstructed (recovered from a scan or a PDF rather than from clean source), and that process left real defects behind: a duplicated-self method signature, unbalanced parentheses, stray characters — the kind of damage that stops the package importing at all. This layer is where most of the urgent, unglamorous work lives.

Keep the two layers straight. The legacy top-level string.py (a flat module holding a big String class) is a different thing from the modern simulation/string_level module. When this section says "the legacy monolith classes", it means the flat cell.py / string.py / section.py / panel.py files — not their well-built counterparts inside simulation/. The plan is, in one phrase, to grow the legacy layer toward the modern one — but only after we have made it run and pinned its behaviour with tests.

6.3   Priority 1 — correctness and "make it run"

This tier has one theme: the package must import cleanly. Until it does, every other item is blocked, because you cannot test, refactor, profile, or extend code that will not load. There is no judgement call here about whether P1 comes first — it is a precondition, not a preference.

Fix the OCR/syntax defects. The OCR-reconstructed legacy files contain mechanical damage: a method written def setTemperature(self, self, temperature) (the duplicated self is a SyntaxError), parentheses that do not balance, stray characters dropped mid-line. Each is a hard error that stops import. Work through them until python -c "import powerpy" runs without a traceback. Why it matters: nothing downstream can be trusted — or even attempted — until the code runs. Effort: small-to-medium; mechanical once located, and the interpreter points you at each one in turn.

Rename string.py. Python resolves the legacy top-level string.py before the standard library's string when code runs from inside the package, so any library that imports stdlib string — pandas does, indirectly — gets PowerPy's module instead and breaks in a way whose error message blames pandas. Rename the file (e.g. to cellstring.py) and fix the imports that referred to it. Why it matters: it removes a silent, confusing breakage that can swallow an afternoon, because the symptom appears nowhere near the cause. Effort: small.

Make the package installable. Right now imports work only because you happen to be in the right directory. Add a pyproject.toml declaring the package, then pip install -e . for an editable install (your edits take effect immediately, but the package is importable from anywhere). Why it matters: imports stop depending on the current directory, and the test runner, your editor, and CI can all find the package the same way. Effort: small.

6.4   Priority 2 — reliability

Now the code runs. The next question is: does it run correctly? — and, crucially, can we keep it correct while we change it? That is what tests buy. This tier exists to lay down a safety net before the refactoring of P3, because refactoring is precisely the activity that silently breaks things.

Add a tests/ suite. Start with the three kinds Section 5 argued for: a thermal parity test (the new vectorised path, once it exists, must agree with the trusted loop within tolerance), a 2×2-circuit sanity test (a tiny hand-checkable circuit whose answer you know), and physical-bound asserts (no negative power, efficiency in [0, 1], a cell no hotter than physically possible). Why it matters: it locks in correctness before refactoring — a test that passes now will tell you the instant a later edit changes the answer. Effort: medium.

Kill the bare except: pass. The ngspice wrapper currently catches everything and does nothing — a malformed netlist, a missing binary, a parse error all vanish, and the simulation sails on with stale or empty results. Replace it with a try/except that names the specific exception it can handle, logs it, and re-raises what it cannot. Why it matters: it stops hiding failures — a hidden failure is worse than a crash, because the crash points at the bug and the hidden one produces a plausible wrong number you will plot and believe. Effort: small.

6.5   Priority 3 — structure and clarity

The code runs and is pinned by tests. Only now is it safe to change its shape. This tier moves the legacy layer toward the modern one — and every edit here is protected by the P2 tests, so if a split accidentally alters the physics, a test goes red immediately.

Split the monolith classes by concern. Each legacy class does several jobs at once: reading data, doing parameter maths, generating the netlist, running the simulation, plotting. Separate these into distinct modules in the modern style — one for loading, one for the parameter maths, one for netlist generation, one for simulation, one for plotting. Why it matters: each concern becomes testable and changeable in isolation; a change to the report format no longer risks the physics. Effort: large — this is the biggest structural item, which is exactly why it comes after the safety net and not before.

Replace bare tuple/dict returns with dataclasses. The 6-tuple (model, config, isc, imp, vmp, voc) forces every caller to remember that vmp is element 4 — a mis-index waiting to happen, silent because every element is a float. Replace such returns with a @dataclass whose fields have names, and add type hints to the public functions. Why it matters: fewer mis-index bugs, and the signature becomes self-documenting — the reader and the type-checker both see what comes back. Effort: medium.

6.6   Priority 4 — performance

Clean, tested, well-structured code that is too slow is a real problem — but it is a fourth-order problem, addressed only once the first three tiers are done. The governing rule of this tier is one word: measure. You do not optimise what you guess is slow; you optimise what a profiler shows is slow.

Profile first. Before touching any of the three changes below, run a profiler on a representative workload and read off where the time actually goes. Why it matters: intuition about performance is notoriously wrong; profiling replaces a guess with a measurement, so your effort lands on the real bottleneck. Effort: small — and it pays for itself by stopping you optimising code that was never the problem.

Vectorise the thermal solve (Chapter 2 §2). The cell-by-cell Python loop is the prime suspect for Monte-Carlo cost, because it runs once per cell per sample, thousands of times over. Recast it as NumPy array operations over all cells at once. Why it matters: it is the single biggest speed win for Monte-Carlo. Effort: medium — and note the P2 thermal parity test is what proves the vectorised version still gives the right answer.

Batch ngspice. Today the simulator issues many separate .op calls, paying engine start-up and parse cost each time. Issue one run per circuit / IV sweep and reuse the engine instead of relaunching it. Why it matters: it removes a large, repeated fixed overhead. Effort: medium.

Parallelise Monte-Carlo across cores. Monte-Carlo samples are independent — an "embarrassingly parallel" workload — yet they run one at a time. Distribute them across CPU cores with joblib. Why it matters: near-linear speed-up for almost no algorithmic change. Effort: small-to-medium.

6.7   Priority 5 — the new feature work

Last, and only last, comes the genuinely new capability: the Chapter 2 physics modules. They go last not because they are unimportant but because they should be built on solid ground — a package that imports, has tests, is cleanly structured, and runs fast. Add them in the modern layout, each as a small focused module with its own test file: circuit.py, substrate.py, thermal_vectorized.py, electrothermal.py, environment_orbit.py, montecarlo.py, results.py. Why it matters: new features built on a stable, tested base stay correct as they grow, instead of inheriting and compounding the old fragility. Effort: large — but de-risked by everything that came before.

6.8   How to sequence it — and why not start with the fun part

The sequencing rule. Run & test → refactor → optimise → extend. Concretely: P1–P2 make it run and tested; then P3 refactors the structure under the protection of those tests; then P4 optimises the hot spots you have profiled and proven; then P5 builds new features on the now-solid base. The justification is a chain of dependencies, not a matter of taste: refactoring without tests is risky (you cannot tell whether a structural change altered the physics), and optimising without profiling is guesswork (you may speed up code that was never the bottleneck while the real one stays slow). Each tier is the safety equipment for the next.

Why is it so tempting to do this backwards? Because the bottom of the pyramid is boring and the top is exciting. Fixing a stray bracket and renaming a file feels like janitorial work; writing a vectorised electro-thermal solver feels like real engineering. But the excitement is exactly the trap: the glamorous work is the most dependent on the unglamorous foundation, and building it first means building it on code you cannot trust, cannot test, and have not measured.

Definition of done — a checklist for a healthy module. When is a PowerPy module finished? When all of these are true:
  • Single clear responsibility — one reason to change.
  • Type-hinted public functions — the contract is on the page.
  • Docstrings — what it does, parameters with units, what it returns.
  • A test file — its behaviour is pinned, not hoped about.
  • No stdlib-name clashes — nothing named string, json, random, … that shadows the standard library.
  • No bare excepts — failures are specific and logged, never swallowed.
  • Data via package resources, not hard-coded paths — it loads its data through the package, not from C:\Users\me\....
This is the bar P5's new modules must clear, and the bar P3 is dragging the legacy modules up to.

6.9   Worked example — one legacy smell, end to end EXAM-HOT

Worked example — the string.py shadowing smell: problem → fix → the test that proves it. We take one P1 item and walk it the whole way, ending — and this is the part that ties P1 to P2 — in the concrete test you would add so the bug can never silently return.
Problem. PowerPy has a top-level module string.py (the legacy String class). When any code runs from inside the package directory, Python finds this string on the import path before the standard library's string. Most libraries import stdlib string somewhere in their machinery — pandas does — so import pandas ends up loading PowerPy's module by mistake and fails. The error names pandas, so you hunt in entirely the wrong place. 
powerpy/
  string.py        # legacy String class — SHADOWS stdlib `string`
  cell.py
  ...

# from inside powerpy/:
>>> import pandas
ImportError: ... (pandas tried to use stdlib `string`, got PowerPy's instead)
Fix. Rename the module so it no longer collides with a standard-library name, and update every import that referred to it. 
# rename the file:
powerpy/string.py   ->   powerpy/cellstring.py

# update imports throughout the package:
- from powerpy.string import String
+ from powerpy.cellstring import String
Now nothing in the package is named string, so Python's stdlib string resolves normally and import pandas works from anywhere.
The test you would add (this is the P2 step). A fix is not done until a test guards it. Add a tiny test that fails on the old code and passes on the fixed code — it asserts both that the shadow is gone and that the symptom is cured: 
# tests/test_no_stdlib_shadow.py
import importlib

def test_stdlib_string_not_shadowed():
    """The stdlib `string` must resolve to the standard library, not PowerPy."""
    import string
    assert "site-packages" in string.__file__ or "lib" in string.__file__.lower()
    # i.e. it is NOT powerpy/string.py

def test_pandas_imports_cleanly():
    """The original symptom: importing pandas must not break."""
    pandas = importlib.import_module("pandas")
    assert pandas is not None
Check. Run the tests before the rename: test_stdlib_string_not_shadowed fails (stdlib string points at powerpy/string.py) and test_pandas_imports_cleanly fails (the ImportError). Run them after the rename: both pass. The bug is now regression-proofed — if anyone ever re-adds a top-level string.py, the suite goes red and names the real problem instead of blaming pandas. ✓ That is the P1→P2 motion in miniature: fix the defect, then pin it with a test so it cannot silently return.

6.10   In what order do you fix a fragile scientific codebase?

Q·a (foundation). Before you can trust any result PowerPy prints, what must be true of the code?
At minimum it must run — the package must import without a SyntaxError and without a name clash that breaks a dependency — and its core answers must be checked against something you know. A number from code that barely imports, or that you have never compared to a hand-calculation or a physical bound, is not a result; it is an output. So the first two things to establish are "it runs" (P1) and "its answers are pinned by tests" (P2). Everything else is premature until these hold.
Q·b (build up). Why refactor under tests — why not just carefully restructure the monolith classes and re-check by eye?
Because a refactor is a structural change with a promise attached: "I changed the shape but not the behaviour." Re-checking by eye cannot verify that promise across a large class with many code paths — you will miss the one branch where the split quietly changed a sign or an index. A test suite verifies the promise mechanically and instantly: make the structural change, run the tests, and if any answer moved, a test goes red and points at it. That is why P2 (tests) precedes P3 (refactor): the tests are the equipment that makes refactoring safe rather than a gamble.
Q·c (why). Why profile before optimising — surely the slow cell-by-cell loop is obviously the bottleneck?
It probably is — but "probably" is the problem. Performance intuition is wrong often enough that committing days of work to a guess is reckless. The loop might be dwarfed by, say, the repeated ngspice start-ups, or by file I/O. A profiler turns the guess into a measurement: it shows where the time actually goes, so you spend your effort where it pays. Optimising without profiling can mean speeding up code that was never the bottleneck — you do real work and the program is no faster. So P4 starts with "profile first", and only then vectorises, batches, and parallelises.
Q·d (deep — the target). So in what order should we fix a fragile scientific codebase, and why is the exciting performance/feature work last?
Put the rungs together. First make it run and be tested (Q·a) — without that, no result is trustworthy and nothing can be verified. Then refactor under those tests (Q·b) — the tests are what make restructuring safe rather than blind. Then profile and optimise (Q·c) — measure before you accelerate, so effort lands on the real bottleneck. Then, on that stable, tested, clean, fast base, build the new features (P5), each with its own tests, in the modern layout. The exciting work is last precisely because it is the most dependent: a vectorised solver you cannot test is unverifiable, an optimisation you have not profiled may be pointless, and a feature added to code that does not import cannot even run. You fix from the foundation up because each tier is the safety equipment for the one above it — and the glamorous tiers sit at the top.
Pitfall — starting with the fun part. The most common and most expensive mistake on a fragile scientific codebase is to open it and immediately reach for the interesting work: vectorise the solver, parallelise the Monte-Carlo, add the new orbit module. On code that does not yet import, or has no tests, this is building on sand. You vectorise a thermal solve with no parity test, so you cannot tell whether your fast version still gives the right answer. You parallelise before fixing the bare except that hides failures, so now failures are hidden and spread across cores. You add a feature to a package that crashes on import, so you cannot even run it. Every one of these efforts is wasted or worse, because the foundation it needed was never laid. Do P1–P2 first. Always.

Section recap. PowerPy is a two-layer codebase: a healthy modern layer (schemas/ loader/ simulation/ render/, dataclasses, a pipeline) and a fragile legacy layer (the flat cell.py string.py section.py panel.py, OCR-reconstructed with real defects). The roadmap fixes it from the foundation up, each item framed problem → fix → why → effort: P1 make it run (fix the OCR syntax defects so it imports; rename string.py so it stops shadowing the stdlib and breaking pandas; add pyproject.toml + pip install -e .), P2 reliability (a tests/ suite — thermal parity, 2×2 sanity, physical-bound asserts; replace the bare except: pass in the ngspice wrapper with specific logged exceptions), P3 structure (split the monolith classes by concern; replace the 6-tuple with a dataclass + type hints), P4 performance (profile first, then vectorise the thermal solve, batch ngspice, parallelise Monte-Carlo with joblib), P5 new features (the Chapter 2 modules in the modern layout, each with tests). The sequencing rule that ties it together: run & test → refactor → optimise → extend, because refactoring without tests is risky and optimising without profiling is guesswork — and the exciting work goes last precisely because it depends on everything below it. The pitfall to avoid above all: starting with the fun optimisation or feature work on code that does not yet run or have tests, which is building on sand.
