4   Making it fast: performance and time-efficiency

The previous sections were about writing code that is clear and correct. This one is about making correct code fast enough — and the very first thing to learn about performance is that almost everyone gets it wrong by starting in the wrong place. They guess where the program is slow, rewrite that part to be clever, and discover the program is exactly as slow as before because the real bottleneck was somewhere they never looked. Performance work is not about being clever; it is about being empirical. You measure, you find the one place that actually costs the time, you fix that, and you measure again. Everything in this section hangs off that discipline.

The one idea. Measure before you optimise. A program's runtime is almost never spread evenly — it nearly always hides in a small hot spot, a handful of lines that run millions of times or call something expensive. Your job is to find that hot spot by measurement (profiling), then attack it with the biggest lever available: usually a better algorithm (so the work grows more slowly with problem size) or vectorisation (so a whole array is processed in one compiled operation instead of a slow Python loop). Cleverness applied anywhere else is wasted effort that only makes the code harder to read. This section defines every term from scratch, works the numbers with units, and shows where PowerPy itself is fast and where it is slow.

We build the vocabulary in the order you would actually use it: first profiling (how to find the hot spot), then Big-O (why the algorithm dominates for large problems), then vectorisation (the single biggest win in scientific Python), then a cluster of "stop doing redundant work" techniques (caching, hoisting, copying cost, batching external calls), then data structures, then parallelism, and finally the discipline of when not to optimise at all. Two fully worked numeric examples and a Socratic ladder tie it together.

4.1   The golden rule: measure first, with a profiler

The temptation, when something feels slow, is to stare at the code, form a theory ("it must be the matrix multiply"), and start optimising on the strength of that theory. This is how most performance effort is wasted. Human intuition about where time goes is famously bad: the bottleneck is routinely a line you would never have suspected — a string being reparsed in a loop, an object being deep-copied on every iteration, an engine being spun up and torn down thousands of times. The cure is to stop guessing and measure.

Profiling. Measuring where a program actually spends its time (and sometimes its memory) while it runs — which functions are called, how often, and how many seconds accumulate in each. A profiler instruments your code and hands you a ranked table: "this function took 71% of the total." That table, not your intuition, tells you what to optimise. Profiling turns "I think it's slow here" into "it is slow here, by measurement."

Python gives you two tiers of tool, for two different questions.

  • Timing one snippet — "how long does this small piece take?" Use time.perf_counter(), a high-resolution clock, around the code; or the timeit module, which runs the snippet many times and reports an average (so a single noisy measurement does not fool you). Good for comparing two implementations of the same small thing.
  • Profiling a whole run — "across this entire simulation, where does the time go?" Use cProfile, Python's built-in profiler. You run the whole program under it and it produces a per-function breakdown — call counts and cumulative time — so the hot spot rises to the top of the table on its own.
# Tier 1 — time a single snippet with perf_counter
import time
start = time.perf_counter()
result = solve_thermal(cells)            # the piece under suspicion
elapsed = time.perf_counter() - start    # seconds, wall-clock
print(f"solve_thermal took {elapsed*1e3:.1f} ms")

# Tier 1 — or let timeit average away the noise (many repetitions)
import timeit
t = timeit.timeit("solve_thermal(cells)", globals=globals(), number=100)
print(f"mean {t/100*1e3:.2f} ms over 100 runs")

# Tier 2 — profile a WHOLE run; the hot function floats to the top
import cProfile
cProfile.run("run_monte_carlo(n=1000)", sort="cumulative")
"Don't guess, profile." This is the oldest slogan in performance work, and it survives because it keeps being true. Run cProfile first, read the top line of the table, and you will usually find that one function — often one you did not suspect — accounts for the bulk of the runtime. Optimise that. A 2× speed-up of a function that is 5% of the runtime buys you 2.5% overall; a 2× speed-up of the function that is 80% of the runtime buys you 40%. The profiler tells you which function is the 80% one. Without it you are optimising blind.

4.2   Algorithmic complexity: Big-O notation

Once the profiler has pointed you at the hot spot, the first and biggest question is not "can I shave a few microseconds off each iteration?" but "is this the right algorithm?" — because for a large problem, a better algorithm beats every micro-optimisation combined, by margins no amount of clever code can close. The tool for reasoning about this is Big-O notation.

Big-O notation. A way of describing how an algorithm's running time (or memory) grows as the size of its input grows. We call the input size N (number of cells, number of Monte-Carlo runs, number of voltage points). Big-O captures the shape of that growth, ignoring constant factors and small terms, because for large N the shape is what dominates. Reading "O(...)": you say "order of".

The handful of growth rates you must recognise on sight, from gentle to catastrophic:

  • O(1) — constant. Time does not depend on N at all. Looking up one value in a dict, reading array[0]. Double the input, same time.
  • O(N) — linear. Time grows in direct proportion to N. One pass over a list. Double the input, double the time. A position sweep over N cells is O(N).
  • O(N²) — quadratic. Time grows as the square of N — typically a loop nested inside another loop, each over N. Double the input, quadruple the time. Tolerable for N in the hundreds, painful in the thousands, hopeless in the millions.
  • O(2^N) — exponential. Time doubles every time you add one to N. This is the wall: at N = 60 you already need more operations than there are atoms you could ever count. Any O(2^N) approach is usable only for tiny N.
Why the algorithm dominates. Constant factors — how fast each individual step is — matter, but they are a fixed multiplier. The growth rate is a multiplier that gets worse with N. So for large enough N, an O(N) algorithm in slow Python beats an O(N²) algorithm in hand-tuned C, and an O(2^N) algorithm loses to everything. Pick the right complexity class first; tune the constant factor second. Getting this order backwards — polishing the constant on a fundamentally bad algorithm — is the most expensive mistake in the section.

PowerPy's example. Suppose you wanted to know the probability that a solar array still meets its power budget given that some cells have failed. The brute-force idea is to enumerate every possible combination of working-and-failed cells. With N cells, each either working or failed, there are 2^N combinations — for a realistic array of several hundred cells that is a number with a hundred-plus digits, utterly impossible to enumerate. This is exactly why Chapter 2's reliability analysis uses Monte-Carlo sampling instead: rather than visit all 2^N cases, it draws a few thousand random failure patterns and estimates the probability from the sample. Monte-Carlo trades an impossible O(2^N) exact computation for a tractable, tunable O(number-of-samples) estimate. By contrast, sweeping a fault through each cell position one at a time is only O(N) — N independent runs — which is why a position sweep is cheap while exhaustive enumeration is not.

Pitfall — polishing the constant on an exponential algorithm. Making each step of an O(2^N) enumeration twice as fast does not rescue it: at N = 300 it is still 2^300 steps, and halving an impossible number leaves an impossible number. The only fix for a bad complexity class is a different algorithm (here, sampling instead of enumeration). Recognise the class before you optimise, or you will tune something that can never finish.

4.3   Vectorisation: the big win in scientific Python

This is the single most important performance technique for numeric Python, and it is the one that turns PowerPy's thermal model from slow to fast. The idea is to stop looping over array elements in Python and instead express the whole computation as operations on entire arrays, which a compiled library (numpy) executes in one shot.

Vectorisation. Replacing an explicit Python for loop that processes array elements one at a time with a single array operation that processes the whole array at once in compiled C code (via numpy). Instead of for i in range(n): out[i] = a[i] + b[i], you write out = a + b — and that one line loops over all elements down in C, not in the Python interpreter.

Why is that faster, when it does the same arithmetic? Because of where the loop runs and what the loop costs per step. In a Python loop, every single iteration pays a tax the interpreter cannot avoid: it has to look up what type each object is, unwrap ("unbox") the float out of its Python object, do the add, wrap the result back into a new Python object, and advance the loop machinery. That overhead is often larger than the arithmetic itself, and you pay it N times. A vectorised numpy operation pays the per-element overhead essentially once for the whole array: it knows up front that the array is a contiguous block of, say, 8-byte floats all of the same type, so it runs a tight compiled loop with no per-element type checks or boxing. The arithmetic is identical; the bookkeeping around it nearly vanishes.

The mechanism in one sentence. A Python loop pays the interpreter's per-iteration overhead (type check + unbox + rebox + loop step) once per element; a vectorised numpy call pays that overhead once for the entire array and does the real arithmetic in compiled code — so the more elements you have, the bigger the win.
Case study — PowerPy's thermal solver (Chapter 2 §2)

Here is the win on real PowerPy code. A solar array has many cells (call it N), and the thermal model must find each cell's equilibrium temperature — the temperature where heat in equals heat out, which is the root of a nonlinear energy-balance equation.

Case study — before and after.
Legacy — one solver call per cell, in a Python loop. The old code calls scipy.fsolve (a general-purpose root finder) once for every single cell, inside a Python for loop. Each call spins up the solver afresh, iterates to a root for that one cell, and tears down — then the loop moves to the next cell and does it all again. For N cells that is N independent solver spin-ups and N trips round the slow Python loop.
New — one vectorised Newton step over ALL cells. The new solver treats every cell's temperature as one entry in a numpy array and applies Newton's method to the whole array at once: each iteration computes the residual and its derivative for all N cells as array operations, and updates all N temperatures in a single vectorised step. A handful of iterations converge every cell together. The entire solve is roughly ~6 array operations — no per-cell loop, no per-cell solver spin-up.
The shape of the change. The legacy version turns one physical problem into N solver start-ups plus N Python iterations; the new version turns it into about six array operations that run in compiled C regardless of how many cells there are — so the cost stops scaling with the number of cells. ✓
Why this is the textbook vectorisation case. The per-cell calculations are independent — cell 12's equilibrium temperature does not depend on cell 13's — so there is nothing forcing them to happen one after another. That independence is exactly the condition that lets you stack all the cells into an array and solve them in one vectorised sweep. (Hold onto this: when the per-element work is not independent, vectorisation across that dimension is not available — we return to it in §4.10 and the final pitfall.)
An order-of-magnitude timing sketch

Let us put rough numbers on it. These are illustrative order-of-magnitude estimates to show the shape of the win, not measured benchmarks.

Suppose one fsolve call costs about 1 ms. For an array of N = 600 cells:
  legacy thermal work per run = 600 cells × 1 ms/cell = 600 ms ≈ 0.6 s per run
  vectorised thermal work per run ≈ 1–2 ms total (a few array operations, independent of cell count)
where "per run" means one full thermal solve of the whole 600-cell array; 1 ms/fsolve and ~2 ms/vectorised-solve are stand-in figures chosen to illustrate the order of magnitude.

That is a speed-up in the ~100× class for a single thermal solve (≈600 ms collapsing to a couple of ms). And it compounds: the reliability study in Chapter 2 runs the whole thing inside a Monte-Carlo sweep of, say, 1000 runs. The thermal work alone then goes from 1000 × 0.6 s = 600 s (ten minutes) down to 1000 × 2 ms ≈ 2 s. Same physics, same answer — the wait drops from "go get a coffee" to "already done." (We compute the exact ratio in Worked Example (b).)

4.4   Avoid redundant work (1): caching and memoisation

The fastest computation is the one you never do. A large class of speed-ups comes not from doing work faster but from not repeating work you have already done. The headline technique is caching.

Memoisation (caching). Storing the result of a computation the first time you do it, keyed by its inputs, and returning the stored result the next time the same inputs come in — instead of recomputing. "Caching" is the general idea; "memoisation" is the specific case of caching a function's return value keyed by its arguments. The trade is memory for time: you spend a little space holding past results to avoid spending time redoing them.

Python makes function memoisation almost free with a decorator from the standard library:

from functools import lru_cache

@lru_cache(maxsize=None)
def cell_iv_curve(area, isc_ref, temperature):
    # expensive: builds and solves the cell's subcircuit
    ...
    return curve

# First call with these args: computes and stores the curve.
# Any later call with the SAME (area, isc_ref, temperature): returns
# the stored curve instantly, no recomputation.

lru_cache ("Least Recently Used cache") remembers recent calls and serves repeats from memory. It only works when the function is pure in the §3 sense — same inputs must always give the same output — because that is exactly what makes a stored result safe to reuse.

PowerPy already caches by content. PowerPy hashes a cell's parameters into an md5 digest (a short fixed-length fingerprint of the argument values) and uses that as the cell's name. The effect: two cells with identical parameters hash to the same name, so the corresponding subcircuit is recognised as already built and is not constructed a second time. That is caching — keyed by the content of the parameters rather than by call order. In an array where many cells share the same specification, this quietly avoids rebuilding the same subcircuit hundreds of times.

4.5   Avoid redundant work (2): hoisting, copying cost, and batching

Hoist invariant work out of loops

If something inside a loop produces the same value every iteration, computing it inside the loop wastes that work N times over. Hoisting means lifting that loop-invariant computation out of the loop, computing it once, and reusing it.

# SLOW — recomputes a constant factor on every one of N iterations
for cell in cells:
    factor = boltzmann * t_ref / charge      # identical every pass!
    cell.adjust(factor)

# FAST — hoist the invariant out; compute it ONCE
factor = boltzmann * t_ref / charge          # once, before the loop
for cell in cells:
    cell.adjust(factor)

The arithmetic of factor happens once instead of N times. The rule generalises: anything inside a loop that does not depend on the loop variable should usually live outside it.

Copying cost

Copying data is not free, and copying big data repeatedly can quietly dominate a runtime. A deep copy duplicates an object and everything it refers to, all the way down — so a deep copy of a large nested configuration allocates and fills a whole second structure.

PowerPy's deep-copy habit. PowerPy frequently deep-copies objects by serialising them to JSON and parsing them back — a convenient way to get a fully independent copy. For a small configuration that is perfectly fine; the cost is negligible. But the same trick becomes expensive if it is done per cell, per iteration inside a hot loop: serialising and re-parsing a structure thousands of times adds up fast. The guideline: copy only what actually changes. If a loop mutates one field of a big object, copy that field — not the entire object — each pass.
Expensive external calls — batch and amortise

The most expensive thing PowerPy does per call is talk to ngspice, an external circuit simulator. Each invocation of the wrapper ng_sim creates a fresh ngspice engine instance and destroys it again when done. Spinning an engine up and tearing it down has a fixed cost that has nothing to do with the actual circuit — and if you pay that cost per cell, per voltage point, it utterly dominates.

Batching (and amortising). Batching means grouping many small operations into one larger operation so a fixed per-operation overhead is paid once instead of once-per-item. Amortising means spreading that fixed cost across all the items it now serves, so the cost per item becomes small. Pay a setup cost once and use it for 600 things, and the amortised setup cost per thing is 1/600th of what it was.

Applied to ngspice: instead of one .op (operating-point) call per voltage per cell — thousands of engine spin-ups — solve a whole circuit or a whole IV sweep in a single ngspice run, and reuse the engine across points where you can. The engine start-up is then paid once and amortised over the entire sweep, instead of being re-paid thousands of times.

Mini worked figure — amortising engine start-up.
Say an ngspice engine takes ~50 ms to spin up and tear down, and the actual solve of one operating point is ~1 ms.
per-point approach (engine per .op): (50 + 1) ms × 100 points = 5100 ms
batched approach (one engine, 100 points in it): 50 ms + (100 × 1 ms) = 150 ms
where 50 ms is the fixed engine setup, 1 ms is one operating-point solve, 100 is the number of voltage points in the sweep.
The fixed 50 ms is paid 100 times in the first case (5000 ms of pure overhead) and once in the second. Batching cuts the sweep from ~5.1 s to ~0.15 s — roughly 34× — entirely by amortising a fixed cost. ✓

4.6   Data structures and memory

The container you choose for numeric data affects both speed and memory. The key contrast is a Python list of floats versus a numpy array of floats.

  • A Python list of floats is a list of pointers to individual float objects scattered in memory; each float carries object overhead, and iterating touches them one boxed object at a time. Flexible, but heavy and cache-unfriendly for numbers.
  • A numpy array of floats is one contiguous, typed block of raw 8-byte values — no per-element object overhead, laid out so the CPU can stream through it, and the substrate that makes vectorisation (§4.3) possible.

So for numeric work, prefer numpy arrays: they are smaller in memory and faster to compute on. Memory matters in its own right for the big result tables a Monte-Carlo sweep produces — thousands of runs × many quantities per run. Two levers keep that under control: compact dtypes (store values in the smallest type that suffices — e.g. float32 instead of float64 halves the table when full double precision is not needed) and columnar storage (store each quantity as its own contiguous column rather than as rows of mixed-type records, which is both more compact and faster to scan one quantity at a time). This links directly to Chapter 2 §7's handling of Monte-Carlo result tables.

4.7   Parallelism (briefly)

Some work can be split across several CPU cores and done at the same time. Monte-Carlo is the ideal case for it.

Embarrassingly parallel. A workload made of pieces that are completely independent — none needs any result from another — so they can run simultaneously on different cores with essentially no coordination. The 1000 runs of a Monte-Carlo sweep are embarrassingly parallel: each run draws its own random sample and computes its own result, sharing nothing, so you can hand run 1 to one core, run 2 to another, and so on. With C cores you approach a C× ("near-linear") speed-up.

In Python you reach for joblib (a thin, friendly wrapper, popular in scientific code) or the standard-library multiprocessing to spread independent runs across cores. But there is a Python-specific catch you must understand to choose the right tool.

GIL — Global Interpreter Lock. A lock inside CPython (the standard Python interpreter) that allows only one thread to execute Python bytecode at a time, even on a multi-core machine. It exists to keep the interpreter's internals safe, but its consequence for performance is blunt: multiple Python threads cannot run Python computation truly in parallel — they take turns holding the lock. So threads do not speed up CPU-bound Python work.
Processes, not threads, for CPU-bound work. Because the GIL serialises threads, the way to get real parallelism for computation is to use multiple processes — each a separate Python interpreter with its own GIL — so they genuinely run at once on separate cores. joblib and multiprocessing default to processes for exactly this reason. (Threads still help for I/O-bound work — waiting on disk or network — because a thread waiting on I/O releases the GIL. But a Monte-Carlo sweep is CPU-bound, so it wants processes.)

4.8   When NOT to optimise

Everything above is about how to make code fast. This subsection is about when not to bother — which is most of the time.

Premature optimisation. Making code more complicated in the name of speed before you have measured that the speed is actually needed there. It is the habit of rewriting clear code into clever code on a hunch — and it almost always trades real, permanent readability for imaginary or trivial performance, because the "optimised" spot was usually not the bottleneck at all.
Optimise the proven hot spot, and nothing else. Clear code has a real, ongoing value: it is easier to trust, debug, and change (§3). Speed has value only where it is measurably lacking. So the rule closes the loop with §4.1: profile first, find the one function that genuinely costs the time, optimise that one — accepting some added complexity there because it buys a measured win — and leave the other 95% of the code clear and simple. Optimising code that is not hot makes the program harder to read for no benefit you can measure. The hot spot earns its complexity; nothing else does.

4.9   Worked example (a) — Big-O and parallel wall-clock EXAM-HOT

Worked example (a) — a 600-position fault sweep. You want to sweep a single-cell fault through every position in a 600-cell array: for each position, mark that one cell failed and run the full simulation. Each full run takes 0.5 s. Question: what is the wall-clock time on 1 core, and on 8 cores?
Identify the complexity. One run per position, 600 positions, runs independent of one another → the sweep is O(N) with N = 600. (Contrast the impossible O(2^N) of enumerating all failure combinations in §4.2 — here we change one cell at a time, not every subset.)
Single-core wall-clock. The runs happen one after another:
T₁ = N × (time per run) = 600 runs × 0.5 s/run = 300 s (= 5 minutes)
where N = 600 positions, 0.5 s = duration of one full simulation run, and "× 1" because one core does them sequentially.
Eight-core wall-clock. The 600 runs are embarrassingly parallel (each position is independent), so 8 cores share the load near-linearly:
T₈ ≈ T₁ / cores = 300 s / 8 = 37.5 s ≈ 38 s
where 300 s is the single-core total and 8 is the number of cores; the ÷8 assumes ideal near-linear scaling.
Check and caveat. 600 × 0.5 s = 300 s on one core; spread over 8 independent workers, ≈ 38 s — an ~8× reduction, which is the near-linear speed-up an embarrassingly-parallel workload allows. The 38 s is an idealised floor: real scaling is slightly worse because of process start-up and result-gathering overhead, so expect a little above 38 s, not below. Because the runs are CPU-bound, this uses 8 processes (not threads — the GIL would serialise threads). ✓

4.10   Worked example (b) — quantifying the vectorisation saving EXAM-HOT

Worked example (b) — fsolve-per-cell vs one vectorised solve, across a Monte-Carlo sweep. Using the case-study figures from §4.3: an array of 600 cells; the legacy thermal solve calls fsolve once per cell at ~1 ms per call; the vectorised solve does the whole array in ~2 ms total; the reliability study runs 1000 Monte-Carlo runs. (Order-of-magnitude figures, not benchmarks.)
Legacy cost per run. One fsolve per cell, 600 cells:
legacy/run = 600 cells × 1 ms/cell = 600 ms = 0.6 s per run
where 600 = cells in the array (one fsolve each) and 1 ms = cost of a single fsolve call.
Legacy cost across the whole sweep. 1000 independent runs:
legacy total = 1000 runs × 0.6 s/run = 600 s (= 10 minutes of thermal work)
where 1000 = Monte-Carlo runs and 0.6 s = legacy thermal cost of one run.
Vectorised cost across the whole sweep. ~2 ms total per run, 1000 runs:
vectorised total = 1000 runs × 2 ms/run = 2000 ms = 2 s
where 2 ms = vectorised thermal solve of the whole 600-cell array in one run.
The saving, quantified.
speed-up = 600 s / 2 s = ≈ 300×;   time saved = 600 s − 2 s = 598 s (just under 10 minutes)
ratio of legacy total to vectorised total; equivalently 0.6 s/run ÷ 2 ms/run ≈ 300× per run.
Check and honest framing. The numbers divide to ≈300×, so the thermal work of the whole 1000-run study collapses from ~600 s to ~2 s. We describe this as a "~100× class" of speed-up — an order-of-magnitude statement — rather than a hard figure, because the inputs (1 ms/fsolve, 2 ms/vectorised) are illustrative stand-ins, not measured times; the real ratio depends on the machine and the cell count. The point that survives any reasonable choice of inputs: turning N Python-loop solver calls into ~6 array operations removes an entire factor of N from the cost, and that factor multiplies across every Monte-Carlo run. ✓

4.11   Socratic ladder — why does vectorising the loop make it ~100× faster?

Q·a (foundation). When you run a plain Python for loop for i in range(n): out[i] = a[i] + b[i], what does the interpreter actually do on every single iteration, beyond the addition itself?
On each pass it does a pile of bookkeeping around the one add: it advances the loop and bounds-checks i; it indexes a and b, which means following pointers to two Python float objects; it checks the type of each object and unboxes the raw double out of it; it performs the add; it boxes the result into a brand-new Python float object; and it stores that back into out. The arithmetic is one machine instruction; everything around it — the type checks, the boxing and unboxing, the loop machinery — is many, and it repeats every iteration.
Q·b (name it). That surrounding bookkeeping has a name — what is it, and how does its cost compare to the arithmetic?
It is interpreter (per-iteration) overhead: the fixed cost the Python interpreter pays to execute each step of a loop in a dynamically-typed language where every value is a boxed object of unknown type until checked. For simple numeric work this overhead is typically larger than the arithmetic it surrounds — often by an order of magnitude — so a Python numeric loop spends most of its time not doing the maths, but managing objects and the loop.
Q·c (the shift). When you replace the loop with one numpy operation out = a + b, what work gets pushed out of the interpreter and into compiled code, and why does that kill the overhead?
numpy knows in advance that a and b are contiguous, typed arrays — say, blocks of 8-byte doubles all of the same type. So it can run a single tight loop in compiled C that strides straight through the raw memory adding doubles, with no per-element type check and no boxing/unboxing: the values are already raw doubles, not Python objects. The per-iteration overhead that Python paid N times is paid essentially once, at the boundary, for the whole array. Same additions; the interpreter bookkeeping around each one is gone.
Q·d (why it grows with N). Why does the speed-up get bigger as the array gets larger, rather than staying a fixed percentage?
Because the Python loop's cost is roughly N × (overhead + arithmetic) while the vectorised cost is roughly (fixed setup) + N × (arithmetic). The expensive overhead term scales with N in the loop but is paid once in the vectorised version. So as N grows, the loop keeps re-paying the overhead while numpy does not, and the ratio between them widens. At N = 600 cells (the case study) the loop has paid the per-element tax 600 times; the vectorised solve, essentially zero times.
Q·e (the boundary — the target). Put it together: why does rewriting a per-cell Python loop as one numpy array operation make it ~100× faster when it does the same arithmetic? And what kind of computation would not vectorise this way?
The speed-up: the arithmetic is identical, but the Python loop wraps every element in interpreter overhead — type checks, boxing/unboxing, loop machinery (Q·a–b) — and pays it N times, whereas the numpy operation pushes the loop into compiled C over a typed contiguous block and pays that overhead essentially once (Q·c). Since the dominant cost was the overhead and it no longer scales with N, the larger the array the larger the win (Q·d) — for PowerPy's 600 independent cells, a ~100×-class collapse from ~600 ms to a couple of ms. What would NOT vectorise: a computation with a genuine per-element sequential dependency — where computing element i requires the result of element i−1. You cannot do all elements "at once" if each one needs the previous one's answer first. PowerPy shows both faces of this: the thermal solve vectorises across cells because the cells are independent, but the Newton iterations within the solve are inherently sequential (each step uses the previous step's estimate), as is the coupling fixed-point loop between the electrical and thermal models. You vectorise the independent dimension (cells) and accept that the dependent one (iteration steps) stays a loop. Vectorisation removes the interpreter tax wherever the work is independent; where the work is truly sequential, that tax cannot be removed by vectorising and you need a different approach.

4.12   Pitfalls

Pitfall 1 — optimising before profiling. If you optimise on a hunch instead of a measurement, you will almost certainly speed up the wrong thing: you pour effort into a function that turns out to be 3% of the runtime while the real 80% hot spot sits untouched, and the program is no faster — but now it is also harder to read. The fix is the golden rule (§4.1): run cProfile first, read the top of the table, and optimise only the function that the measurement proves is hot. Don't guess, profile.
Pitfall 2 — "vectorising" something with a genuine sequential dependency. Vectorisation only works when the per-element computations are independent. If element i truly depends on element i−1 — an iterative solver step that needs the previous estimate, a running cumulative total, a fixed-point loop that feeds its own output back as input — you cannot express it as one array operation that does everyone "at once," because the inputs for later elements do not exist yet. Forcing it produces wrong answers (silently using stale values) or simply cannot be written. The honest move is to recognise the dependency and either keep that part as a loop, or reach for a method designed for sequential structure (e.g. numpy.cumsum for cumulative sums, or a vectorised iteration that steps all elements forward together — as PowerPy's Newton solver does across cells while still iterating sequentially in time).

Practice D4-4. A colleague has sped up a tidy 12-line configuration-parsing function into 40 lines of bit-twiddling "for performance." The profiler shows that function is 0.4% of total runtime; the Monte-Carlo thermal solve is 86%. In two or three lines, critique this decision using the section's vocabulary, and say what should have been optimised instead.
Section recap. Measure before you optimise — profiling finds where time actually goes; use time.perf_counter/timeit for a snippet and cProfile for a whole run; don't guess, profile. Big-O describes how runtime grows with input size N — O(1) constant, O(N) linear, O(N²) quadratic, O(2^N) exponential — and for large N the right algorithm beats any micro-optimisation (enumerating 2^N failure combinations is hopeless, which is why Chapter 2 samples with Monte-Carlo; a position sweep is O(N)). Vectorisation replaces a per-element Python loop with one compiled numpy array operation, paying the interpreter's per-iteration overhead (type-check + box/unbox + loop step) once for the whole array instead of N times — PowerPy's thermal solver turns N fsolve calls into ~6 array ops, a ~100×-class win (~600 ms → ~2 ms for 600 cells) that compounds across a 1000-run sweep (~600 s → ~2 s). Avoid redundant work: memoise/cache (functools.lru_cache; PowerPy's md5-of-parameters cell naming is caching by content), hoist loop-invariants out, watch copying cost (PowerPy's JSON deep-copy is fine for small configs, costly per-cell-per-iteration — copy only what changes), and batch/amortise expensive external calls (one ngspice run for a whole sweep, reuse the engine, not one .op per voltage per cell). Prefer numpy arrays (contiguous, typed) over lists of floats, and use compact dtypes / columnar storage for big Monte-Carlo tables (Ch2 §7). Monte-Carlo is embarrassingly parallel: spread it across cores with joblib/multiprocessing for near-linear speed-up, using processes not threads for CPU-bound work because the GIL (Global Interpreter Lock) lets only one thread run Python bytecode at a time. Finally, avoid premature optimisation: optimise only the profiled hot spot, and leave the rest clear.
