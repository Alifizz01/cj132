D3-1. The bug: a mutable default argument. The default log=[] is created once, when the function is defined — not freshly on each call. Every call that omits log therefore shares the same list, so readings accumulate across unrelated calls: 
append_reading(1)   # [1]
append_reading(2)   # [1, 2]   <-- the 1 from the previous call is still there
Why it misbehaves (one line). The default list is evaluated once at definition time, so all calls that don't pass their own list mutate one shared object.
The fix — default to None, create the list inside: 
def append_reading(x, log=None):
    if log is None:
        log = []            # a fresh list on every call that omits it
    log.append(x)
    return log
Now append_reading(1) → [1] and append_reading(2) → [2]: the two calls are independent. The same pattern fixes any mutable default (list, dict, set) — never use a mutable literal as a default; default to None and build it in the body.
D3-2. Rewrite returning a dataclass instead of a 4-tuple. 
from dataclasses import dataclass

@dataclass
class IVPoints:
    isc: float
    imp: float
    vmp: float
    voc: float

def iv_points(cell, t: float) -> IVPoints:
    """Return the four IV points of `cell` at temperature `t` (°C)."""
    isc = cell.isc_ref * (1 + 0.0005 * (t - 25))
    imp = isc * 0.95
    vmp = cell.vmp_ref * (1 - 0.003 * (t - 25))
    voc = vmp * 1.18
    return IVPoints(isc=isc, imp=imp, vmp=vmp, voc=voc)
A call site reading a field by name: 
pts = iv_points(cell, 40.0)
peak_power = pts.vmp * pts.imp        # by NAME — no counting positions
One bug the dataclass version makes impossible. Mis-indexing. With the tuple, a caller writing result[3] intending vmp would silently get voc instead (both are floats, so nothing complains) — a wrong number with no error. With the dataclass you write pts.vmp; there is no position to miscount, and a typo such as pts.vmpp raises AttributeError at the access site rather than flowing on as a wrong value. (Bonus: reordering or adding a field can't shift any caller's index, and the editor autocompletes the field names.)
D3-3. The two problems.
  1. Hard error — duplicated self in the signature. def calcT(self, self, p) lists self twice, which is a SyntaxError: duplicate argument 'self'. The method is uncallable and the module won't even import.
  2. Bad practice — bare except: pass. The try/except catches everything and silently discards it. If self.k is zero (division by zero), missing, or wrong-typed, the failure vanishes and self.T is left stale or never set — a silent wrong answer. It even swallows KeyboardInterrupt.
Clean rewrite (PEP 8 name, docstring, type hints, early validation, loud failure): 
class Panel:
    def compute_temperature(self, power: float) -> float:
        """Set and return the panel temperature from dissipated `power` (W).

        Parameters
        ----------
        power : float
            Dissipated power, in watts.

        Returns
        -------
        float
            The computed temperature (°C), also stored on self.temperature.

        Raises
        ------
        ValueError
            If the thermal conductance `self.k` is not positive.
        """
        if self.k <= 0:
            raise ValueError(f"thermal conductance k must be > 0, got {self.k}")
        self.temperature = power / self.k
        return self.temperature
What changed and why. One self (fixes the syntax error); the PEP 8 name compute_temperature with a descriptive power parameter instead of calcT/p; a docstring stating units, return, and what it raises; type hints (power: float -> float); the bad input (k ≤ 0) is validated early and raises a clear ValueError instead of being swallowed — so a real problem fails loudly and locatably rather than silently corrupting self.T. The method also now returns the value (not purely a hidden mutation), so callers can use it directly.
Note: this method still has a side effect (it writes self.temperature); if you wanted it fully pure you would compute and return power / self.k without storing it, letting the caller decide whether to save it.
